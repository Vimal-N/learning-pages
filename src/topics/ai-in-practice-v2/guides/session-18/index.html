When to Build Your Own Server

Most of the time, an existing server is enough. These are the cases where building one is worth it.

Worth building a custom server
  • Your team uses an internal tool with no public MCP server (internal Jira, internal test reporting dashboard)
  • You want to expose your test runner's output in a structured, queryable way
  • You have a custom test data factory the agent needs to call
  • You want to wrap an internal API so agents can query your test environment status
  • You want to gate tool access — limit what the agent can do more precisely than config allows
Use an existing server instead
  • GitHub, Jira Cloud, Linear, Notion — public MCP servers already exist
  • Browser control — Playwright MCP covers this
  • Filesystem access — the built-in filesystem MCP server
  • Web fetch / search — community servers available
  • When the capability exists but you haven't looked for it yet

Check the MCP server registry first: The MCP community registry at modelcontextprotocol.io/servers lists hundreds of published servers. Most standard tools already have one. Building custom is for internal or bespoke tooling.

MCP Server Anatomy

The four structural components every MCP server has — regardless of what it does.

1
Server initialisation
Creates the MCP server instance and names it. The name is what appears in the agent's tool list. Uses the MCP SDK (@modelcontextprotocol/sdk).
2
Tool definitions
A list of tools the server exposes. Each tool has: a name (what the agent calls it), a description (what the LLM reads to decide whether to use it — write this carefully), and an input schema (what parameters the tool accepts).
3
Tool handler
The function that runs when the agent calls a tool. Receives the input parameters, performs the action (API call, file read, calculation), and returns a structured result. Errors should be returned as results, not thrown — the agent can handle a returned error message; an uncaught exception crashes the server.
4
Transport layer
How the server communicates with the client. Most local servers use StdioServerTransport — the client starts the server as a subprocess and communicates via stdin/stdout. Remote servers can use HTTP/SSE transport instead.

Annotated Server File — Full Example

A complete MCP server that exposes two tools for QA use: reading Playwright test results and querying test environment status.

import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "zod";
import fs from "fs";

// ─── 1. SERVER INITIALISATION ─────────────────────────────────────────────
// The name and version appear in the agent's tool discovery response.
const server = new McpServer({
  name: "qa-tooling",
  version: "1.0.0",
});

// ─── 2. TOOL DEFINITIONS ──────────────────────────────────────────────────
// Each tool() call registers one capability. The description is the most
// important field — the LLM reads it to decide whether to call this tool.
// Write it as "use this when you want to..." instructions to the model.

server.tool(
  "get_test_results",
  "Use this when you need to read the latest Playwright test run results. " +
  "Returns a summary of passed, failed, and skipped tests with failure details.",
  {
    // Input schema — defines what parameters this tool accepts.
    // Use zod for validation; the SDK converts it to JSON Schema automatically.
    resultsPath: z.string()
      .optional()
      .describe("Path to playwright-report/results.json — defaults to ./playwright-report/results.json"),
  },
  // ─── 3. TOOL HANDLER ──────────────────────────────────────────────────
  // Runs when the agent calls this tool. Must return { content: [...] }.
  // Catch errors and return them as content — do not throw.
  async ({ resultsPath }) => {
    const path = resultsPath ?? "./playwright-report/results.json";
    try {
      const raw = fs.readFileSync(path, "utf-8");
      const results = JSON.parse(raw);
      const summary = {
        passed: results.stats.expected,
        failed: results.stats.unexpected,
        skipped: results.stats.skipped,
        failures: results.suites
          .flatMap(s => s.specs)
          .filter(s => s.ok === false)
          .map(s => ({ title: s.title, error: s.tests[0]?.results[0]?.error?.message })),
      };
      return {
        content: [{ type: "text", text: JSON.stringify(summary, null, 2) }],
      };
    } catch (err) {
      // Return errors as content — don't throw
      return {
        content: [{ type: "text", text: `Error reading results: ${err.message}` }],
      };
    }
  }
);

server.tool(
  "check_test_environment",
  "Use this to check whether the test environment is reachable before running tests. " +
  "Returns status and response time for the configured BASE_URL.",
  {
    url: z.string().url().describe("URL to check — use the app BASE_URL"),
  },
  async ({ url }) => {
    try {
      const start = Date.now();
      const res = await fetch(url, { method: "HEAD", signal: AbortSignal.timeout(5000) });
      const ms = Date.now() - start;
      return {
        content: [{
          type: "text",
          text: `${url} → ${res.status} ${res.statusText} (${ms}ms)`,
        }],
      };
    } catch (err) {
      return {
        content: [{ type: "text", text: `Unreachable: ${err.message}` }],
      };
    }
  }
);

// ─── 4. TRANSPORT LAYER ───────────────────────────────────────────────────
// StdioServerTransport: the client starts this file as a subprocess and
// communicates via stdin/stdout. This is the standard for local MCP servers.
const transport = new StdioServerTransport();
await server.connect(transport);

Testing Your Server

How to verify your server works before connecting it to an agent.

Step 1 — Test the server manually

Run the server directly in your terminal. A working server starts silently and waits for input. An error on startup means a bug before connecting to any agent.

# Install dependencies
npm install @modelcontextprotocol/sdk zod

# Run directly — should start silently (waiting for stdin)
node server.js

# If it errors immediately, fix the error before adding to agent config

Step 2 — Use the MCP Inspector

The MCP Inspector is a browser-based tool for testing servers interactively. It sends tools/list and tools/call messages and shows the responses.

# Run the inspector against your server
npx @modelcontextprotocol/inspector node ./server.js

# Opens a browser UI at localhost:5173
# Use it to: list your tools, call them with test inputs, inspect responses

Step 3 — Add to agent config

Once the Inspector confirms the tools work, add to your agent config using the local server pattern:

// .claude/settings.json
{
  "mcpServers": {
    "qa-tooling": {
      "command": "node",
      "args": ["./mcp-servers/qa-tooling.js"]
    }
  }
}

Write descriptions as instructions to the model, not humans. The tool description is read by the LLM to decide whether to call it. "Reads test results" is weaker than "Use this when you need to know which tests passed or failed in the most recent run." The second form gives the model a decision trigger.
