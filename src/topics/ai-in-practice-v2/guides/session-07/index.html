The Three-Step AI Test Workflow

Don't ask AI to write code first. Get the test cases right, then generate the code.

The most common mistake is going straight from a feature description to a prompt like "write me some tests." That skips the step where you — the QA engineer — decide what to test. AI writes what you describe; if you describe nothing, it guesses.

1
Extract test cases from acceptance criteria
Use AI to turn your AC into a structured list of test scenarios — happy path, error states, boundary values, edge cases. You review and trim this list. This is still your decision.
2
Generate code from the test case list
Feed AI your approved test case list plus context about your project (framework, selectors, existing test style). It writes the Playwright code. You review it before committing anything.
3
Verify, run, and fix
Run the generated tests. Check the four most common AI mistakes (covered later in this session). Fix selectors, assertions, or timing issues before the tests go into your suite.

The key mindset shift: AI writes the code. You decide what to test. Never hand over the decision about coverage to the model — it doesn't know your product, your risk areas, or your users.

Step 1 — AC to Test Cases

Paste in your acceptance criteria. Ask AI to enumerate what needs to be tested. Then review the list before writing a single line of code.

The prompt

Act as a senior QA engineer.

Here is the acceptance criteria for a checkout feature:
[PASTE YOUR AC HERE]

Generate a complete list of test cases covering:
1. Happy path (all valid inputs, expected success)
2. Error states (invalid input, payment failure, network error)
3. Boundary values (minimum/maximum quantities, character limits)
4. Edge cases (empty cart, session expiry mid-checkout, duplicate submission)

Format each test case as: Given [state] / When [action] / Then [expected result].
Flag any ambiguities in the AC that I should clarify before testing.

What AI does well here

It finds error states and boundary values you might have missed. It asks about ambiguities in the AC — things like "what happens if the user applies a discount code after adding an item to the cart?" These are real test coverage gaps.

Your job

Review the list. Remove tests that aren't worth automating (highly manual flow, unlikely edge case). Add anything AI missed that you know from product knowledge. The list you approve is the list that becomes code.

Pro tip: Ask AI to flag AC ambiguities explicitly. These become questions for your developer or product owner before you write tests, not after the sprint ends.

Step 2 — Test Cases to Code

Now give AI your approved test list plus enough context to write code that actually fits your project.

What context to provide

Selectors
Required
List the actual element IDs, data-testid values, or CSS selectors for your feature. AI can't inspect your app — if you don't provide selectors, it invents them and the tests will fail immediately.
Existing test example
Strongly recommended
Paste one complete test from your project and say "match this style." AI will replicate your naming conventions, import structure, POM patterns, and assertion style — without being told each rule explicitly.
Page Object reference
If applicable
If your project uses Page Objects, paste the relevant PO class (or use #file: in Copilot Chat) so AI uses your existing methods rather than duplicating selector logic in the test file.
Expected URLs / messages
Required for assertions
Tell AI the exact URL a successful action redirects to, the exact text of error messages, the exact element that confirms success. These are assertions — wrong values mean tests that pass when they shouldn't.

The prompt structure

Using Playwright and TypeScript, write tests for the following test cases:
[PASTE YOUR APPROVED TEST CASE LIST]

Project context:
- Page URL: /checkout
- Selectors: #quantity-input, #promo-code, button[data-testid="place-order"], div.order-confirmation
- On success: redirects to /order-confirmation?id=xxx with "Order placed" heading
- On invalid promo code: shows span.promo-error with text "Invalid code"

Here is an example test from my project — match this exact style:
[PASTE ONE EXISTING TEST]

Write each test case as a separate test() block inside one test.describe("Checkout"). No shared beforeEach — each test must set up its own state.

The Style-Matching Technique

The single most effective thing you can do to get usable AI-generated tests: show it what you already have.

When you paste an existing test and say "match this style," AI infers dozens of rules it would otherwise have to guess: your import paths, whether you use POM methods or raw page.fill(), how you name test.describe blocks, which assertion style you prefer. This is the fastest path to code you don't have to rewrite.

Without style example
  • AI picks an import style at random
  • May use page.locator() when you use page.fill()
  • Invents a beforeEach when your project doesn't use one
  • Names describes inconsistently with your suite
  • Requires rewriting before it fits the project
With style example (one pasted test)
  • Replicates your exact import structure
  • Uses the same method style you already use
  • Follows your naming conventions automatically
  • Matches your POM usage patterns
  • Ready to drop into your test suite with minor edits

In Copilot Chat — use #file: instead of pasting

In Copilot Chat you don't have to paste the example test. Reference it directly:

#file:tests/auth/login.spec.ts Write new tests for the checkout flow. Match the style of the existing tests in this file. [paste your test case list here]

This technique works in any AI tool — ChatGPT, Claude, Copilot Chat. It works because "match this existing code" is a far more precise instruction than any set of abstract rules you could write.

Building a Full Test Suite

How to use AI to generate tests systematically across a feature — not just one file at a time.

Once you have the workflow working for one feature, applying it to a full suite is a matter of repeating the three steps for each area of coverage. Here's how to keep it organised.

One AC section → one test file

Structure your prompts the same way you structure your test files. If your suite has auth/login.spec.ts, checkout/order.spec.ts, checkout/payment.spec.ts, generate each file as a separate prompt with its own context. Don't ask AI to generate everything in one go — the output gets unwieldy and harder to review.

The context card approach

Before generating each file, list the context cards AI needs for that feature:

FILE An existing test (style reference)
Paste one test or use #file:. Provides import style, assertion patterns, naming.
Required Every prompt
DOM Selectors for the feature page
data-testid values, IDs, or CSS selectors for elements the tests interact with.
Required Every prompt
POM Relevant Page Object
If a Page Object exists for this feature, include it so AI uses your existing methods.
If applicable When PO exists
TEXT Exact error messages & URLs
Copy the exact strings the UI shows on success/error — AI will use these in assertions.
Required For assertion accuracy

Step 3 — Reviewing What AI Wrote

Four things to check on every AI-generated test before it goes into your suite.

AI makes predictable mistakes when generating tests. Knowing them in advance makes review fast — you're looking for a known set of issues, not reading every line for correctness.

1
Selectors — are they real?
Check every selector against your actual DOM. AI commonly invents plausible-sounding IDs like #submit-btn when your button is actually button[data-testid="place-order"]. If you didn't provide selectors, assume all of them are wrong.
2
Assertions — do they actually test something?
Check that each assertion would fail if the feature broke. expect(page).toHaveURL('/dashboard') is real. expect(response.status()).toBe(200) on a page navigation is not testing what you think. Weak assertions are worse than no assertions — they give false confidence.
3
Test isolation — can it run independently?
Each test should work in any order and with no prior state. If a test depends on a previous test having logged in, or assumes items are in a cart from a previous test, it will fail intermittently. AI often generates tests that share state unless you explicitly said not to.
4
Waiting patterns — no arbitrary timeouts
Flag any page.waitForTimeout(2000) calls. AI reaches for these when it doesn't know the right condition to wait for. Replace them with expect(locator).toBeVisible() or page.waitForURL() — Playwright's built-in retry makes these reliable; arbitrary timeouts make tests flaky.

Common mistake: Committing AI-generated tests without running them first. Always run the full generated spec and check that each test fails for the right reason before adding it to CI. A test that passes without doing anything isn't protecting you.

Prompt Patterns — Test Writing

Three copy-ready patterns for the most common test generation scenarios.

Pattern 01
Generate test cases from AC
Prompt
Act as a QA engineer. Given this acceptance criteria: [paste AC]. Generate a list of test cases covering happy path, error states, boundary values, and edge cases. Use Given/When/Then format. Flag any ambiguities in the AC.
Pattern 02
Generate Playwright tests (style-matched)
Prompt
Using Playwright TypeScript, write tests for these cases: [list]. Selectors: [list]. On success: [URL/message]. Match the style of this existing test exactly: [paste test]. No shared beforeEach — each test self-contained. No waitForTimeout — use expect assertions for waiting.
Pattern 03
Generate a Page Object for a new feature
Prompt
Write a Playwright Page Object class for [page name]. Selectors: [list]. Methods needed: [list actions — e.g. fillForm(), submitOrder(), getErrorMessage()]. Here is an existing PO from my project for style reference: [paste PO]. Export as default class. No test() blocks in this file.

Try This — Your First AI-Generated Suite

Apply the three-step workflow to a real feature from your current project.

Try This — 20 minutes

Pick one feature in your current project that either has no automated tests or has tests you're not satisfied with. Work through all three steps:

Step 1 (5 min): Paste the acceptance criteria (or write a plain-English description of what the feature does) into Copilot Chat or Claude. Use Pattern 01 to generate a test case list. Review it — keep what you'd actually test, remove what you wouldn't.

Step 2 (10 min): Use Pattern 02 with your approved list, your real selectors, and one existing test pasted as a style reference. Generate the Playwright code.

Step 3 (5 min): Run the generated tests locally. Check selectors, assertions, and waiting patterns using the four-point review checklist. Note what needed fixing.

The goal is not perfect tests first time. It's understanding where in the workflow AI helps most and where you need to add the most context.
