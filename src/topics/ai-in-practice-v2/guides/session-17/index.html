Where Config Files Live

Each client reads its MCP configuration from a different location. Get this wrong and nothing connects.

Claude Desktop
Mac / Windows app
Mac: ~/Library/Application Support/Claude/claude_desktop_config.json
Windows: %APPDATA%\Claude\claude_desktop_config.json
Restart Claude Desktop after editing. Changes don't take effect until restart.
Claude Code (CLI)
Terminal agent
Project-level: .claude/settings.json in your project root (checked into version control — shared with team).
User-level: ~/.claude/settings.json (applies to all projects on your machine).
VS Code (Copilot)
Editor extension
.vscode/mcp.json in your project root, or user settings via VS Code settings UI (MCP Servers). The project-level file is team-shareable; user settings are machine-local.

The most common setup mistake: Adding the config to the wrong file or the wrong location. Verify the path before debugging the server. A config in the right format at the wrong path silently does nothing.

Config File Format — Annotated Side by Side

The format is similar across clients — the key differences are the top-level key name and where environment variables go.

Claude Desktop — claude_desktop_config.json

{
  "mcpServers": {
    "playwright": {                          // ← your name for this server
      "command": "npx",                      // ← how to start the server
      "args": ["@playwright/mcp@latest"],    // ← the package / command args
      "env": {                               // ← environment variables (optional)
        "PLAYWRIGHT_BROWSER": "chromium"
      }
    },
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "ghp_your_token_here"
      }
    }
  }
}

Claude Code — .claude/settings.json

{
  "mcpServers": {                            // ← same key as Claude Desktop
    "playwright": {
      "command": "npx",
      "args": ["@playwright/mcp@latest"]
      // env vars go here too, same format
    }
  }
}

VS Code — .vscode/mcp.json

{
  "servers": {                               // ← different key: "servers" not "mcpServers"
    "playwright": {
      "type": "stdio",                       // ← VS Code requires explicit type
      "command": "npx",
      "args": ["@playwright/mcp@latest"]
    }
  }
}

Key difference to remember: Claude Desktop and Claude Code use "mcpServers". VS Code uses "servers". VS Code also requires "type": "stdio" for most servers. Mixing these up is a common copy-paste error.

Installing an MCP Server

Most servers are npm packages. The install is the config — no separate install step needed for npx-based servers.

Most published MCP servers are npm packages. When you use "command": "npx" with "args": ["@package/name"], npx downloads and runs the package automatically — no separate npm install needed. Alternatively, install globally first for faster startup.

Three server install patterns

→
npx (no install needed) — Use "command": "npx", "args": ["-y", "@package/name"]. The -y flag skips the "are you sure?" prompt. Downloads on first use; cached after that.
→
Global install (faster startup) — Run npm install -g @package/name first, then use "command": "mcp-server-name" directly in config. Avoids the npx download on every session start.
→
Local/custom server — For a server you've built yourself (Session 18), use "command": "node", "args": ["./path/to/server.js"]. Or if it's a Python server: "command": "python", "args": ["./server.py"].

Installing the Playwright MCP server — full walkthrough

# Option 1: Add to Claude Code project config (creates the file if absent)
# .claude/settings.json
{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": ["@playwright/mcp@latest"]
    }
  }
}

# Option 2: Add via CLI (Claude Code)
claude mcp add playwright -- npx @playwright/mcp@latest

# Verify the server is recognised
claude mcp list

Troubleshooting — 4 Common Failures

Most MCP connection problems fall into one of four categories. Here's how to identify and fix each.

FAIL 1 Server not found / command not recognised
Symptom: Agent starts but the MCP tool isn't available. Error may say "command not found" or the tool simply doesn't appear in the agent's tool list.
Fix
Check the config file path — is it in the right location for this client? Check the package name is spelled correctly. Try running the command manually in your terminal: npx @playwright/mcp@latest. If that fails, the problem is the package, not the config.
FAIL 2 Server starts but immediately exits
Symptom: The agent reports the server connected briefly then disconnected. Tools may appear then disappear.
Fix
Run the server command directly in a terminal and watch the output: npx @package/name. If it exits immediately, the server likely needs an environment variable (API key, config path) that's missing. Check the server's README for required env vars and add them to your config's "env" block.
FAIL 3 Auth / permissions error
Symptom: Server connects but tool calls fail with 401, 403, or "invalid token" errors. Filesystem servers may return "permission denied".
Fix
For API-based servers (GitHub, Jira): verify the token is valid and has the required scopes. For filesystem servers: check the allowed paths in the server config match what you're actually trying to access. Tokens set in "env" in the config file are not shell environment variables — they're passed to the server process specifically.
FAIL 4 Config JSON is malformed
Symptom: Client ignores the config entirely — no MCP servers appear. No error message, just silence.
Fix
Validate your JSON. Paste it into a JSON validator (or run node -e "JSON.parse(require('fs').readFileSync('./file.json','utf8'))"). Common culprits: trailing commas (invalid in JSON), mismatched braces, and unescaped backslashes in Windows paths. Use forward slashes or escaped backslashes in path strings.

MCP in a Real QA Workflow

What a session looks like when the agent has MCP servers available — combining browser, filesystem, and GitHub.

With MCP servers connected, an agent task can span multiple systems in a single session. Here's what that looks like for a QA engineer investigating a reported bug:

Agent session with Playwright + GitHub MCP servers
You
GitHub issue #88 says the promo code field accepts negative values. Verify if this is reproducible and add a regression test for it.
Agent
[GitHub MCP] Reading issue #88… The issue describes entering "-10" in the promo field which applies a negative discount, increasing the total. Let me verify this in the browser.
Agent
[Playwright MCP] Navigating to /checkout… [Playwright MCP] Filling promo-code field with "-10"… [Playwright MCP] Clicking Apply… Confirmed: the field accepted "-10" and the order total increased by £10. Screenshot attached. Now writing the regression test.
Agent
[Filesystem] Writing tests/checkout/promo-validation.spec.ts… Test written: verifies that negative values in the promo field are rejected with a validation error. The test currently fails (as expected — bug is open). Run with --grep "negative promo" to see the failure before the fix.
Try This — 20 minutes

Install one MCP server and use it on a real task. Good starting points: Playwright MCP (@playwright/mcp@latest) for live browser control, or Fetch MCP (@modelcontextprotocol/server-fetch) for fetching documentation or changelogs. Configure it for your preferred client, verify it connects (claude mcp list or check the VS Code MCP panel), and give the agent a task that uses it.
