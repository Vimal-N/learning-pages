Why Prompting Matters

The same model, the same question — very different results depending on how you ask.

The model doesn't change based on how you phrase your question. Your description of the task is the only input it has. If your description is vague, the model fills in the gaps with its best guess — and those guesses often miss what you actually needed.

Vague prompt → useless output
Prompt: "Write me some tests."
Result: generic unit tests for made-up functions, wrong framework, no relevance to your actual app. Not usable.
Specific prompt → usable output
Prompt: "Act as a Playwright TypeScript test engineer. I have a login page at /login with #email, #password, and button[type=submit]. Write three tests: successful login, wrong password, and empty fields. Use page.fill() and expect(page).toHaveURL(). Add a descriptive test.describe block."
Result: three runnable tests, correct selectors, correct assertions, correct structure. Ready to adapt.

The difference is not knowing a secret formula. It's providing: a role, context about your project, a specific task, and constraints about how you want it done.

The Four-Element Framework

Role, Context, Task, Constraints — the four things a good prompt always answers.

ELEMENT 1
Role
Tell the AI what kind of expert to behave as. This shapes the vocabulary, style, and depth of the response.
Example: "Act as a senior Playwright test engineer who follows the Page Object Model pattern."
ELEMENT 2
Context
Tell it what's specific about your situation: which framework, what your selectors are, what already exists in the project.
Example: "I'm working on a TypeScript Playwright project. The login page has #email, #password, and button[type=submit]. I already have a LoginPage PO in pages/login.page.ts."
ELEMENT 3
Task
Be specific about what you actually want. "Write tests" is weak. Naming the test scenarios, the assertions, the edge cases is strong.
Example: "Write three test cases: (1) successful login redirects to /dashboard, (2) wrong password shows 'Invalid credentials', (3) empty email shows 'Email is required'."
ELEMENT 4
Constraints
Specify what the output must or must not include: format, length, structure, what to avoid.
Example: "Use page.fill() not page.locator().fill(). Add a test.describe block. Don't use beforeEach — each test should be self-contained."

The Framework in Action

A complete prompt assembled from all four elements — and why each part earns its place.

Act as a senior Playwright TypeScript test engineer who follows the Page Object Model pattern.

I'm working on a TypeScript Playwright project. The login page is at /login.
Selectors: #email, #password, button[type=submit].
On success it redirects to /dashboard.
On wrong password it shows a div.error-message with text "Invalid credentials".
I already have LoginPage in pages/login.page.ts — don't rewrite it.

Write three new test cases in tests/auth/login.spec.ts:
1. Successful login redirects to /dashboard
2. Wrong password shows "Invalid credentials" error
3. Empty email field shows "Email is required" validation

Use page.fill() style. Wrap in test.describe("Login"). Each test must be self-contained — no shared beforeEach.
Role
"Act as a senior Playwright TS engineer who follows POM"
Context
URL, selectors, error messages, existing LoginPage PO
Task
Three named test cases with specific assertions
Constraints
page.fill() style, test.describe wrapper, self-contained tests

Prompt Library — 8 QA Templates

Copy, adapt, and use these prompts in Copilot Chat, ChatGPT, or Claude. Replace the bracketed placeholders with your specifics.

1 — Generate test cases from acceptance criteria
"Act as a QA engineer. Given this acceptance criteria: [paste AC here]. Generate a list of test cases covering happy path, error states, boundary values, and edge cases. Format as numbered list with Given/When/Then structure."
2 — Write a Playwright test from description
"Using Playwright and TypeScript, write a test that navigates to [URL], [actions: fill/click/select], and asserts [expected result]. Use page.fill(), page.click(), expect(locator).toBeVisible(). Wrap in test.describe('[describe name]')."
3 — Explain a test failure
"This Playwright test is failing with the error: [paste error]. Here is the test code: [paste code]. Explain why it might be failing and suggest a fix. Tell me if the issue is likely a selector problem, a timing problem, or something else."
4 — Generate test data
"Generate 10 variations of test data for a [field type, e.g. email address / UK phone number / postcode] field. Include: valid examples, invalid format examples, boundary length examples, and special character examples. Return as a TypeScript array."
5 — Review a test for weaknesses
"Review this test for quality issues: [paste test]. Check for: fragile selectors, race conditions, missing assertions, tests that could pass even when the feature is broken, and any Playwright anti-patterns. List issues with severity (high/medium/low)."
6 — Refactor duplicated tests
"Refactor these tests to remove duplication using a Page Object: [paste tests]. Extract repeated selectors and actions into a class. Keep each test file readable — the Page Object should not be called in beforeEach. Return the PO class and refactored tests separately."
7 — Explain unfamiliar code
"Explain what this code does, line by line, as if talking to someone who understands testing but isn't a developer: [paste code]. Focus on what the test is checking and why each step matters. Highlight anything that might be a risk or gotcha."
8 — Identify coverage gaps
"Here is a list of my existing test cases for [feature]: [paste list]. Here is the acceptance criteria: [paste AC]. What scenarios are missing? Pay particular attention to: error paths, boundary values, concurrent actions, and session/auth edge cases."

Iterating When Results Aren't Right

The first response is a starting point, not a final answer. Here's how to steer it.

Getting good AI output is rarely one shot. The effective approach is to treat the conversation as a refinement loop — get something out, then steer it towards what you actually need.

1
Be specific about what's wrong
"That's not quite right" doesn't help. Say what's wrong: "The selector you used (div.login-btn) doesn't match my app. My submit button is button[type=submit]. Update just that selector."
2
Ask for alternatives when you're unsure
If you're not sure which approach is better: "Show me two ways to handle this — one with beforeEach and one without — and explain the trade-off."
3
Add a constraint to fix a pattern issue
If it keeps using a pattern you don't want: "Never use page.waitForTimeout(). If you're waiting for something, use expect(locator).toBeVisible() instead."
4
Start a new chat if the context is muddled
Long conversations with many corrections confuse the model. If it stops making progress, paste your best result into a fresh chat with a clean prompt rather than continuing to patch a broken thread.

The "show me your existing code" technique: Paste one example of what you want the output to look like — an existing test from your project — and say "Generate a new test in this exact style." The model will match your patterns far more reliably than following abstract instructions.

Prompting in Copilot Chat

How the four-element framework adapts when you're working inside VS Code.

In Copilot Chat, the context element of your prompt is partially handled by the @ and # references you attach. This means your prompts can be shorter — because you're letting Copilot read the relevant files rather than pasting them manually.

Standalone prompt (ChatGPT / Claude)
"Act as a Playwright engineer. I have a test file [paste 60 lines of code]. My login page has selectors #email, #password. Add a test for the session expiry scenario..."
Copilot Chat version (shorter — context via references)
"#file:login.spec.ts Add a new test case for session expiry — the user should be redirected to /login with a 'Session expired' message. Match the style of the existing tests."

The #file: reference tells Copilot to read login.spec.ts and use it as the Context element — so you don't have to paste the code. The task and constraint are still explicit in your message.
