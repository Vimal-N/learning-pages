What Is MCP? — Understanding the Model Context Protocol

The protocol that lets AI agents connect to external tools — and why it matters more than the buzzword suggests.

Session 16  ·  Phase 5: MCP, Agents & AI-Powered Workflows  ·  Advanced
⚡
What you'll get from this session
By the end of Session 16 you will understand what MCP is and the problem it solves, be able to read a basic MCP message exchange, and know which MCP servers are most useful for QA engineering work.

From Sessions 10–15: You've run terminal agents that read your filesystem and run shell commands. MCP extends that capability — it lets an agent connect to external services (browsers, databases, APIs, ticketing systems) using a standardised protocol, without custom integration code.

The Problem MCP Solves

Before MCP, connecting an AI agent to an external tool required custom integration code for every combination. MCP standardises the interface.

Every AI agent — Claude Code, Copilot Agent Mode, any LLM with tools — can call functions. But each agent had its own way of defining and calling external tools. If you wanted Claude to read your Jira tickets, you needed Anthropic-specific integration code. If you wanted GPT-4 to do the same, you needed a different implementation. Two agents, two codebases, same capability.

Before MCP
  • Every AI tool required bespoke integration per external service
  • "Jira + Claude" and "Jira + Copilot" were separate engineering efforts
  • No reuse between agent frameworks
  • External tool providers had to build one connector per AI platform
With MCP
  • One MCP server for Jira — works with any MCP-compatible agent
  • Claude Code, Cursor, Claude Desktop — all read the same server
  • Tool providers build once, support all clients
  • You configure, not code — point your agent at the server

The analogy: MCP is to AI tools what USB is to hardware peripherals. Before USB, every device needed a custom port and driver. After USB, one standard plug worked everywhere. MCP is that standard plug for AI agent capabilities.

How MCP Works

Client, server, and the message exchange — without needing to know the implementation details.

MCP uses a client-server model over a simple message protocol (JSON-RPC). The AI agent is the client. The external capability (filesystem access, browser control, database read) is provided by an MCP server. The client asks the server what tools it offers, then calls those tools by sending structured messages.

1
Discovery — "what can you do?"
When the agent starts, it sends a tools/list request to each configured MCP server. Each server responds with a list of tools it provides: name, description, and the input parameters it accepts.
2
Decision — the LLM decides which tool to call
The LLM sees the tool list as part of its context. When it determines a tool would help complete the user's task, it generates a tools/call request with the tool name and input parameters.
3
Execution — the server runs it
The MCP server receives the call, executes the tool (runs a search, reads a database row, controls a browser), and returns the result as structured data.
4
Response — result goes back to the LLM
The tool result is added to the LLM's context. The LLM uses it to generate its next response or decide what to do next in the loop.

The Message Exchange — Annotated

What the actual JSON-RPC messages look like — and what each field means.

You don't need to write these messages — your MCP client generates them. But reading them demystifies what's happening when an agent "uses an MCP server."

1. Agent discovers tools

// Agent → MCP server: "what tools do you have?"
{ "jsonrpc": "2.0", "method": "tools/list", "id": 1 }

// MCP server → Agent: "here are my tools"
{
  "result": {
    "tools": [{
      "name": "search_jira_tickets",
      "description": "Search Jira for tickets matching a query string",
      "inputSchema": {
        "type": "object",
        "properties": {
          "query": { "type": "string", "description": "JQL query or keyword" },
          "project": { "type": "string", "description": "Project key (e.g. QA)" }
        },
        "required": ["query"]
      }
    }]
  }
}

2. Agent calls a tool

// Agent → MCP server: "run this tool"
{
  "jsonrpc": "2.0",
  "method": "tools/call",
  "params": {
    "name": "search_jira_tickets",
    "arguments": { "query": "failing checkout tests", "project": "QA" }
  },
  "id": 2
}

// MCP server → Agent: "here is the result"
{
  "result": {
    "content": [{ "type": "text", "text": "QA-1042: Checkout test fails on Safari\nQA-1039: Payment timeout test flaky..." }]
  }
}

MCP Servers for QA Engineering

The servers most useful for QA work — what they do, how to install them, and a concrete use case for each.

SERVER Playwright MCP
Gives the agent a live browser it can navigate, click, fill forms, and take screenshots — without writing Playwright code. The agent controls the browser directly via MCP tools.
QA use case
"Navigate to /checkout, fill in test payment details, submit the order, and screenshot the confirmation page." The agent uses the browser MCP server to do this — no test file written, no Playwright API, just a task prompt. Useful for exploratory testing and one-off verification.
SERVER Filesystem MCP
Gives the agent structured read/write access to specific directories. More explicit than the agent's native file access — you configure exactly which paths it can reach.
QA use case
Restrict the agent to tests/ only — even if it wanted to touch src/, it can't via the filesystem MCP server. Useful when you want stricter scope control than CLAUDE.md provides.
SERVER GitHub MCP
Gives the agent access to GitHub: read issues, pull requests, code, comments. Can create issues or add PR comments.
QA use case
"Read the failing CI run on PR #442, identify which test failed and why, and add a review comment explaining the failure." The agent reads GitHub, diagnoses the failure, and posts the comment — without you leaving your terminal.
SERVER Fetch / Web Search MCP
Gives the agent the ability to fetch URLs or run web searches — pulling in current documentation, Playwright release notes, or Stack Overflow answers as context.
QA use case
"Search for the Playwright 1.50 changelog and tell me if any assertion APIs changed that could affect our existing tests." Agent fetches the changelog, reads it, and gives you a targeted summary — no manual searching.
