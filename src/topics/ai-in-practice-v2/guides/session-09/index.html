How to use AI to improve the quality of existing tests — not just write new ones. Reviewing for weaknesses, extracting Page Objects, and reducing duplication.

AI as Test Reviewer

What AI looks for when you ask it to review a test — and what it misses.

When you ask AI to review a test, it checks for known patterns of weakness: fragile selectors, missing assertions, test isolation failures, anti-patterns. It doesn't know whether your tests cover the right business scenarios — that judgement is still yours. But it finds structural problems fast.

What AI catches reliably

CHECK 1
Fragile selectors
Selectors that break when the DOM structure changes: positional selectors (nth-child), text-content selectors on UI labels that change, XPath that traverses multiple levels. AI flags these and suggests data-testid alternatives.
CHECK 2
Weak or missing assertions
Tests that check the wrong thing (status code instead of UI state), tests with no meaningful assertion, tests where the assertion could pass even if the feature is broken. AI identifies "this assertion doesn't actually prove what you think."
CHECK 3
Anti-patterns
page.waitForTimeout(), hardcoded sleeps, duplicate selector definitions, page.evaluate() for things Playwright can do natively, missing await on async operations. AI knows the Playwright anti-pattern catalogue.
CHECK 4
State leakage risk
Tests that create database records, send emails, or modify shared state without cleanup. AI flags these as risks for contaminating other tests, especially in parallel runs.

The review prompt

Review this Playwright test for quality issues:
[paste test]

Check for: fragile selectors, weak or missing assertions, tests that could pass even when the feature is broken, waitForTimeout usage, state leakage, and any Playwright anti-patterns.

List each issue with: severity (high / medium / low), what the problem is, and the fix.

Extracting Page Objects

When the same selectors and actions appear in multiple tests, AI can extract them into a Page Object — faster than doing it by hand.

The Page Object Model (POM) isn't mandatory in Playwright — but once you have three or more tests hitting the same page, duplicated selectors become a maintenance problem. One DOM change breaks every test file that hardcodes that selector. A Page Object centralises it.

When to extract a Page Object

Same selector appears in 3+ test files
Extract to a Page Object now
Same sequence of actions repeated across tests (login, navigate, fill form)
Extract action into a Page Object method
Only 1–2 tests use a page
Keep inline — Page Object adds overhead, not value yet
Selector changes regularly (fast-moving UI)
Prioritise Page Object extraction — maintenance cost is high without it

The POM extraction prompt

Extract a Page Object from these test files:
[paste 2–3 test files or relevant test blocks]

Rules:
- Create one class per page (e.g. LoginPage, CheckoutPage)
- Move all selectors into the class as private readonly locators
- Create methods for each distinct user action (e.g. fillLoginForm(), submitOrder())
- Do NOT put assertions in the Page Object — keep those in test files
- Export each class as default
- Return the Page Object class and the refactored test code separately

Here is an existing Page Object from my project for style reference:
[paste existing PO, or say "no existing PO — create in standard Playwright POM style"]

Finding Coverage Gaps

AI can compare your existing tests against your acceptance criteria and tell you what you haven't tested.

Coverage gaps are scenarios you haven't thought to test — error paths, edge cases, combinations of conditions. These are exactly the failures that reach production. AI is effective at identifying them because it looks at your test list through a different lens than you do.

The coverage gap prompt

Here is a list of my existing test cases for the [feature name] feature:
[paste test names or test.describe / test() structure]

Here is the acceptance criteria for this feature:
[paste AC]

What test scenarios are missing? Focus on:
- Error paths (what happens when things go wrong)
- Boundary values (min/max, empty, maximum length)
- Concurrent actions (what if two users do this at the same time)
- Session and authentication edge cases
- Scenarios where the feature could silently fail (wrong data saved, no error shown)

Return a prioritised list: P1 (likely to find bugs), P2 (worth having), P3 (low value).

AI sees your test names, not your test logic. If you have a test named "user can submit order" but the assertion only checks that a button was clicked (not that the order was saved), AI won't flag the coverage gap — it sees the test name as coverage. Provide your full test code, not just names, for a meaningful gap analysis.

Reducing Test Duplication

Three patterns AI can refactor for you — and one it's bad at.

→
Parametrize repeated tests — If you have 10 nearly identical tests that differ only in input data, AI can convert them to a single data-driven test using for...of or Playwright's test.each equivalent. Feed it the tests and ask: "Convert these to a parametrized test that loops over a test data array."
→
Extract shared setup into fixtures — If multiple test files need the same setup (logged-in user, seeded database record), AI can write a Playwright fixture that handles it. Paste your repeated beforeEach blocks and ask for a fixture extraction.
→
Consolidate selector definitions — If the same selectors are defined in multiple test files, AI can identify the duplication and show you which should move to a shared Page Object or constants file.
AI refactors well
  • Structural refactors with clear rules (extract PO, parametrize, add fixture)
  • Renaming for consistency (matching your naming convention)
  • Adding missing assertions to existing test structure
AI refactors poorly
  • Deciding which tests to delete (requires product knowledge)
  • Reorganising test file structure (needs understanding of your CI/reporting setup)
  • Any refactor that requires knowing your app's domain logic

Try This — Review Your Own Tests

Apply the AI review and gap analysis to a real test file from your project.

Try This — 15 minutes

Pick an existing test file from your project — ideally one you haven't touched in a while or one you know has issues. Work through two exercises:

Exercise 1 — Review (5 min): Use the review prompt from the "AI as Test Reviewer" slide. Paste the full test file. Ask AI to list quality issues by severity. See what it finds that you already knew about, and what it finds that surprises you.

Exercise 2 — Gap analysis (10 min): Pair that test file with the acceptance criteria (or a plain-English description of what the feature does). Use the coverage gap prompt. Ask AI to return a prioritised list of missing test scenarios. Pick one P1 gap and write that test using the Session 07 workflow.

The goal is to leave this session with at least one improved or added test — not just knowledge of how to do it.
