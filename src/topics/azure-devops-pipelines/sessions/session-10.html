Matrix Strategy: One Job, Many Configurations

If you have opened your product team's pipeline and seen a strategy: matrix: block, you have encountered one of the most common patterns in team pipelines. Matrix strategy runs the same job multiple times, each time with a different set of variable values — across browsers, across environments, across operating systems. This session gives you a complete framework for reading any matrix configuration, understanding what it produces in ADO, and knowing when your team is using matrix instead of writing separate jobs.

What matrix strategy is — and why teams use it
the concept
Matrix runs one job definition many times, each time with different variable values

Instead of writing three identical jobs — one for Chrome, one for Firefox, one for Safari — you write the job once and let the matrix populate the variable values. ADO (Azure DevOps) creates one job instance per matrix configuration and runs them all. The result is N jobs from one definition.

  • Without matrix: three separate job definitions in the YAML, each 15 lines long, differing only in one variable value. When you need to change the Node version, you change it in three places.
  • With matrix: one job definition, a matrix block above it with three named configurations. Change the Node version in one place. The three jobs update automatically.
  • Teams use matrix for: running tests across multiple browsers (Chrome, Firefox, Safari), across multiple environments (dev, QA, staging), across multiple operating systems (Ubuntu, Windows, macOS), or across multiple dependency versions.
  • Parallel by default: the matrix configurations run in parallel — three browser configurations run simultaneously, not sequentially. The total pipeline time is the duration of the slowest configuration, not the sum of all three.
common uses
Three patterns where you will see matrix in your team's pipelines
Cross-browser testing

Running the same Playwright suite against Chrome, Firefox, and Safari. Each browser configuration is a matrix entry. Each runs simultaneously on its own agent — a dedicated virtual machine spun up and torn down for that job only.

Cross-environment testing

Running the same smoke suite against dev, QA, and staging environments after a deployment. The environment URL comes from the matrix variable — the same test steps hit a different base URL for each matrix configuration.

Cross-OS builds

Building a desktop application on Windows, macOS, and Ubuntu to verify it compiles cleanly on all supported platforms. The matrix variable controls which agent image each job runs on.

Cross-version testing

Testing against Node 18 and Node 20 to verify backward compatibility before upgrading. Each Node version is a matrix configuration — one job installs and tests with 18, the other with 20.

Reading a matrix block — annotated completely
cross-browser-matrix.yml — a real matrix configuration, every line explained
jobs:
  - job: E2ETests
    strategy:
      matrix:
        Chrome_Windows:              # matrix key — this becomes the job display name
          browser: chrome               # variable set for this configuration
          os: windows-latest             # second variable for this configuration
        Firefox_Linux:                # second configuration — creates a second job
          browser: firefox
          os: ubuntu-latest
        Safari_Mac:                   # third configuration — creates a third job
          browser: safari
          os: macos-latest
      maxParallel: 3                 # all three run at the same time
    pool:
      vmImage: $(os)                  # $(os) resolves to the matrix variable for this job
    steps:
      - checkout: self
      - script: npm ci
      - script: npx playwright test --project=$(browser)
        # $(browser) resolves to 'chrome', 'firefox', or 'safari' for each job
        continueOnError: true
      - task: PublishTestResults@2
        condition: always()
        inputs:
          testResultsFormat: 'JUnit'
          testResultsFiles: 'test-results/*.xml'
          testRunTitle: '$(browser) on $(os)'   # appears in Tests tab as the run label
decoding it
The five things to understand in any matrix block
  • The matrix keys (Chrome_Windows:, Firefox_Linux:, Safari_Mac:) are display names. They appear as job names in the ADO pipeline summary and in the Tests tab. You can name them anything descriptive. ADO creates one job for each key.
  • The variables beneath each key (browser: chrome, os: windows-latest) are pipeline variables scoped to that specific job instance. They are available as $(browser) and $(os) anywhere in the job's steps, pool, and conditions.
  • $(variableName) in a step resolves to the matrix variable value for the current job. When the Chrome_Windows job runs, $(browser) is chrome. When the Firefox_Linux job runs, $(browser) is firefox. The step definition is written once; the matrix supplies the variable.
  • maxParallel: 3 means all three configurations run simultaneously. Set it to 1 to run them sequentially. Set it to 2 to run two at a time. When agent parallelism is limited (the free tier provides 1,800 minutes per month), reducing maxParallel conserves capacity.
  • testRunTitle: '$(browser) on $(os)' names each test run in the Tests tab. Without this, all three runs appear with the same default name and are hard to distinguish. Always include a descriptive testRunTitle that incorporates a matrix variable.
What matrix produces in ADO — the pipeline summary and Tests tab
pipeline summary
The pipeline summary shows one job per matrix configuration

When a pipeline with a matrix strategy runs, the ADO pipeline summary shows multiple jobs within the same stage — one for each matrix configuration. This is normal and expected.

  • The Test stage in the summary expands to show: E2ETests (Chrome_Windows), E2ETests (Firefox_Linux), E2ETests (Safari_Mac) — one row per matrix key.
  • Each job has its own duration, its own status (pass/fail), and its own log output. Clicking into any job row opens the logs and steps for that specific configuration only.
  • If you open the Timeline tab, you see all three jobs starting at approximately the same time and running in parallel on separate agents.
  • A developer or QE engineer looking at this summary can immediately see which browser configuration (if any) failed — without needing to read logs or decode a combined output.
tests tab
The Tests tab aggregates all matrix runs — with run labels to distinguish them

Each matrix job publishes its own test results. ADO aggregates all of them into a single Tests tab for the pipeline run.

  • The Tests tab shows all tests from all matrix configurations in one view — you see the total pass/fail count for all browsers combined.
  • Each test run is labelled with the testRunTitle value — Chrome on windows-latest, Firefox on ubuntu-latest, Safari on macos-latest.
  • You can filter the Tests tab by run title to see results for a specific browser only.
  • A test that fails on Safari but passes on Chrome will appear once in the failed list, under the Safari run. It is immediately visible which configuration is the problem.
  • If all three configurations publish their results with a unique testRunTitle, you will always be able to identify which browser, OS, or environment produced a failure.
quality gate
When the stage fails — one failing matrix job fails the whole stage

The stage-level result is the union of all matrix job results. If any matrix job fails, the Test stage fails.

  • A single failing test in the Safari configuration causes the E2ETests (Safari_Mac) job to fail.
  • A failed job causes the Test stage to fail.
  • A failed Test stage blocks any downstream stage with dependsOn: Test — including a Deploy stage.
  • This means three browser configurations, running in parallel, still produce a single pass/fail verdict at the stage level. All three must pass for the stage to pass.
  • You do not need extra configuration for this. Stage-level rollup is how ADO works by default — it is not something your team had to set up.
Matrix vs parallel sharding — which pattern to use
decision
Matrix and sharding solve different problems — here is how to tell them apart

Both matrix strategy and parallel sharding (covered in Session 11) run multiple agents in parallel. But they answer different questions.

Use matrix when
  • You need to run the same tests in different configurations (browsers, environments, OS)
  • Each configuration produces a meaningfully different result — a Safari failure is different from a Chrome failure
  • You want the Tests tab to show results labelled by configuration
  • Your team has cross-browser or cross-environment requirements in the definition of quality
Use sharding when
  • You have one configuration but too many tests to run in time
  • All agents run the same browser on the same OS — they split the test files between them
  • You want to cut wall-clock time from 18 minutes to 6 minutes
  • You do not need to differentiate results by configuration — all results are equivalent regardless of which agent ran them

You can combine both. A matrix with three browser configurations, each sharded across two agents, produces six parallel jobs total. Session 11 covers sharding in detail.

Practical guidance — reading and diagnosing matrix pipelines
reading an unfamiliar matrix
Four questions to ask when you see a matrix block you did not write

When you encounter a matrix in a pipeline your team already has, these four questions give you a complete picture.

  • What are the matrix keys? List them: Chrome_Windows, Firefox_Linux, Safari_Mac. Each key is one job instance. Count the keys to know how many parallel jobs this produces.
  • What variables does each key define? Look at the indented properties under each key (browser: chrome, os: windows-latest). These are the values that change between job instances.
  • Where are those variables used? Search for $(variableName) in the pool, steps, and inputs. Every occurrence is a place where the value changes between job instances.
  • What is maxParallel set to? If it is not set, all configurations run simultaneously. If it is set to a number lower than the configuration count, some will queue behind others.
when a matrix job fails
Diagnosing a matrix failure — which configuration, which test, what error

When a matrix pipeline fails, the first question is always: which configuration? The ADO summary makes this visible immediately.

  • In the pipeline summary, find the Test stage and expand it. The failing job will be labelled with its matrix key — for example, E2ETests (Safari_Mac).
  • Go to the Tests tab and filter by the run title for that configuration — for example, Safari on macos-latest.
  • The failing test name, error message, and stack trace are there. If the test passes on Chrome and Firefox but fails on Safari, that is a browser-specific compatibility issue — route it to the developer with: "The [test name] test is failing on Safari only, error: [paste message]."
  • If the same test fails on all three configurations, it is not a browser-specific issue. It is likely a change to the application or a test environment problem — investigate the most recent deployment or environment change.
in conversation
Words to use in your next standup

You now have language for these situations:

  • "This pipeline uses a matrix strategy — it runs the same job three times, once for each browser, in parallel." — when explaining what the strategy: matrix: block does to someone who has not seen it before
  • "The matrix key Chrome_Windows is the display name for that job instance — it appears in the summary and the Tests tab." — when someone asks why the job is named that way in ADO
  • "The $(browser) in the test command resolves to whatever value the current matrix configuration sets for browser — chrome for the Chrome job, firefox for the Firefox job." — when explaining how one step definition runs different commands for each matrix job
  • "The Safari matrix job failed — I can see it in the summary. The other two configurations passed. This looks like a Safari-specific issue." — when reporting a cross-browser test failure in a standup or incident channel
  • "We have maxParallel: 3 so all three browser configurations run at the same time — the total pipeline time is the duration of the slowest one, not the sum." — when explaining why a three-browser matrix does not triple the pipeline duration

Key takeaways
  • Matrix strategy runs one job definition multiple times, each time with a different set of variable values — one job instance per matrix key
  • Matrix keys are display names that appear in the ADO pipeline summary and Tests tab — name them descriptively so failures are immediately identifiable
  • $(variableName) inside the job resolves to the matrix variable value for the current job instance — the step definition is written once and the matrix supplies the variation
  • maxParallel controls how many matrix jobs run simultaneously — default is all of them at once; reduce it to conserve agent minutes on limited plans
  • The Test stage fails if any matrix job fails — one failing browser configuration blocks the whole stage and any downstream Deploy stage
  • Use matrix for different configurations (browsers, environments, OS). Use sharding for the same configuration with too many tests. Combine both for maximum parallelism.
