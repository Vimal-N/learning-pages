Phase 3 · Pipeline Internals

Reading a Build: Summary Page + Full Playwright Pipeline

Knowing how to read a pipeline run is as important as knowing how to write one. This session walks through the ADO build summary page — its five tabs, what each one tells you — then shows a complete, production-quality Playwright pipeline from trigger to published results.

The build summary page
navigation
Five tabs, five different answers

When a pipeline run completes (or fails), ADO's build summary page is where you go to understand what happened. Each tab answers a different question.

Summary
High-level view: stages, pass/fail, duration, and triggered by whom. First stop.
Tests
All test results published by this run. Filter by pass/fail, search by name, see stack traces.
Artifacts
Files published by the pipeline — reports, screenshots, build output, packages.
Logs
Full stdout/stderr for every step in every job. Click the step in the timeline to jump to its log.
Timeline
Visual Gantt chart of all stages and jobs, with per-step timing. Identifies where time was spent.
tests tab
What the Tests tab shows — and how results get there

Test results don't appear automatically. They have to be explicitly published using the PublishTestResults@2 task. Without this task, the Tests tab is empty even if your tests ran.

  • Playwright (with a JUnit reporter) writes results to test-results/*.xml
  • PublishTestResults@2 reads the XML and uploads it to ADO
  • The Tests tab shows: total tests, pass %, failed count, and individual test names
  • Failed tests show the error message and stack trace inline — no digging through logs
  • Results from multiple jobs in the same run are merged into a single Tests tab
artifacts tab
Pipeline artefacts — publishing files for later use

The Artifacts tab holds anything your pipeline publishes using PublishPipelineArtifact@1. This includes Playwright HTML reports, screenshots, trace files, build outputs, and coverage reports.

  • Artefacts persist after the VM is destroyed — they're uploaded to ADO's storage
  • Can be downloaded manually from the Artifacts tab, or downloaded by a later job using DownloadPipelineArtifact@2
  • Playwright's HTML report is the most valuable artefact for QE: full traces, screenshots, and video
  • Always publish artefacts with condition: always() — you want them even when the run fails
When a pipeline fails — what to do
scenario
A developer messages you: "The pipeline failed — can you take a look?"

This will happen. Here is the exact sequence of steps to go from "pipeline failed" to "here is what went wrong and who should fix it" in under five minutes.

  • Step 1 — Open the pipeline run. In ADO, go to Pipelines → the pipeline name → the failed run (it has a red X). Click it.
  • Step 2 — Identify which stage failed. The Summary tab shows all stages as a row. The red stage is where the failure is. If the Build stage is red, the failure is in compilation or linting — nothing to do with tests. If the Test stage is red, tests ran but something failed.
  • Step 3 — Go to the right tab based on the failing stage. Test stage failure → Tests tab. Other failure → Logs tab. Artifacts missing → Artifacts tab.
  • Step 4 — Find the specific failure. In the Tests tab: filter by Failed, click a failing test, read the error message and stack trace. In the Logs tab: click the red step in the timeline, scroll to the first error line — that is usually the root cause, not the lines after it.
  • Step 5 — Route it. Test failure with a clear assertion error → route to the developer who changed that feature. Test failure with a connection error → environment issue, route to DevOps or infrastructure. Build failure (lint/compile) → route to the developer who pushed the change.
the html report
Downloading the Playwright report for deeper investigation

The Tests tab in ADO shows pass/fail at the test name level. The Playwright HTML report — published as a pipeline artifact — shows screenshots, traces, and video for every failing test. When the error message alone is not enough, the HTML report is where you find out exactly what the browser saw.

  • Go to the Artifacts tab of the failed pipeline run
  • Find the artifact named playwright-html-report (or similar)
  • Download it and open index.html in your browser
  • Click any failing test to see: the exact error, a screenshot at the point of failure, a trace timeline showing every network request and DOM action, and optionally a video of the full test run
  • This is the most detailed debugging tool available — and it is always there if the pipeline published it with condition: always()
Full Playwright pipeline
playwright-pipeline.yml — production-quality CI pipeline
# Fires on PRs and pushes to main
trigger:
  - main
pr:
  - '*'

pool:
  vmImage: 'ubuntu-latest'

variables:
  NODE_VERSION: '20'

stages:

  - stage: Build
    displayName: 'Build'
    jobs:
      - job: InstallAndLint
        steps:
          - checkout: self
          - task: NodeTool@0
            inputs:
              versionSpec: '$(NODE_VERSION)'
          - script: npm ci
            displayName: 'Install dependencies'
          - script: npm run lint
            displayName: 'Lint'

  - stage: Test
    dependsOn: Build
    displayName: 'Test'
    jobs:
      - job: PlaywrightTests
        steps:
          - checkout: self
          - task: NodeTool@0
            inputs:
              versionSpec: '$(NODE_VERSION)'
          - script: npm ci
            displayName: 'Install dependencies'
          - script: npx playwright install --with-deps chromium
            displayName: 'Install Playwright browsers'
          - script: npx playwright test --reporter=junit,html
            displayName: 'Run Playwright tests'
            continueOnError: true
          - task: PublishTestResults@2
            condition: always()
            inputs:
              testResultsFormat: 'JUnit'
              testResultsFiles: 'test-results/junit*.xml'
              mergeTestResults: true
              testRunTitle: 'Playwright E2E'
          - task: PublishPipelineArtifact@1
            condition: always()
            inputs:
              targetPath: 'playwright-report'
              artifact: 'playwright-html-report'
walkthrough
What the pipeline above does — step by step

Seven things happen in this pipeline on every push. Understanding each one helps you diagnose failures and extend the pipeline later.

  • Triggers: Fires on every push to main and every open PR — CI and a merge trigger in one file
  • Build stage: Installs exact dependency versions from the lock file (npm ci), then runs the linter to catch syntax and style issues before tests
  • Why a separate Build stage? Each stage runs on a fresh agent — Build's agent and Test's agent are two different machines. Separating Build from Test means: if linting fails, we never spin up a second agent for tests. We catch the cheapest failure (lint) before paying for the expensive one (test execution). It also means the pipeline summary shows you exactly which phase failed at a glance.
  • Test stage: Only starts if Build passed. Gets a fresh agent — separate from Build's agent
  • Browser install: npx playwright install --with-deps chromium downloads Chromium and its OS-level dependencies. This is not in npm ci — it must be an explicit step
  • Test run: Playwright runs all tests and writes JUnit XML + HTML report. continueOnError: true ensures the pipeline continues even if tests fail
  • PublishTestResults: condition: always() ensures results are uploaded even if tests failed. The Tests tab in ADO is populated from this
  • PublishPipelineArtifact: The HTML report (with screenshots and traces) is published as an artefact. Download it from the Artifacts tab to debug failures

Key takeaways
  • The build summary page has five tabs: Summary, Tests, Artifacts, Logs, and Timeline — each answers a different question
  • Test results only appear in the Tests tab if you include a PublishTestResults@2 task
  • The Playwright HTML report goes into Artifacts via PublishPipelineArtifact@1 — always with condition: always()
  • Always pair continueOnError: true on the test step with condition: always() on the publish steps
  • Each job gets a fresh agent — the Test stage reinstalls dependencies even though Build already did
