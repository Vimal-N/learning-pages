Phase 3 · Pipeline Internals

Flow Control: Making Your Pipeline Smart

A pipeline that runs every step unconditionally — even when earlier steps fail — produces misleading results. This session covers the mechanisms that give your pipeline decision-making ability: stage dependencies, built-in conditions, and the patterns used to build reliable quality gates.

Stage dependencies
dependsOn
Controlling the order stages run in

By default, stages in a YAML pipeline run sequentially in the order they're declared. You can make this explicit — and conditional — using dependsOn.

  • dependsOn: Build means "this stage only starts after the Build stage completes successfully"
  • If you omit dependsOn, ADO infers sequential order from position in the file
  • Multiple dependencies: dependsOn: [Build, SecurityScan] — both must succeed
  • To run a stage in parallel with another: set dependsOn: [] (empty array)
  • A downstream stage will not run if its upstream dependency fails — unless you add a condition
dependsOn — explicit stage ordering
stages:
  - stage: Build
    # runs first — no dependsOn needed

  - stage: Test
    dependsOn: Build
    # only runs if Build succeeded

  - stage: Deploy
    dependsOn: Test
    # only runs if Test succeeded
Conditions
built-in conditions
Four expressions that control when a step or stage runs

Every step and stage has an implicit condition: succeeded(). You can override this with any of the four built-in condition functions. Understanding these four is enough to handle almost every flow-control scenario.

succeeded()
Run only if all previous steps/stages succeeded. This is the default.
failed()
Run only if at least one previous step/stage failed. Useful for cleanup or notification on failure.
always()
Run regardless of success or failure. Used for teardown and reporting steps that must never be skipped.
canceled()
Run only if the pipeline was manually cancelled. Rarely used — mostly for cleanup of long-running operations.
always() — publish results even when tests fail
steps:
  - checkout: self
  - script: npm ci && npx playwright test
    displayName: 'Run Playwright tests'
    continueOnError: true

  - task: PublishTestResults@2
    condition: always()
    inputs:
      testResultsFormat: 'JUnit'
      testResultsFiles: 'test-results/*.xml'
pattern
Why continueOnError + always() is the standard test pattern

If your test step fails and doesn't have continueOnError: true, the pipeline stops and never reaches the publish step. You get a failed pipeline with no test results — the worst of both worlds.

  • continueOnError: true on the test step: the pipeline continues even if tests fail
  • condition: always() on the publish step: results are always uploaded
  • The pipeline still ends as "failed" — but the Tests tab shows which tests failed
  • Without this pattern: failed tests produce a failed pipeline with no actionable detail
  • With this pattern: the failure tells you exactly which tests failed and why

the QE identity moment
This is what you own

The quality gate pattern is the single most important thing a QE engineer can build into a pipeline. When configured correctly, no code can reach production with failing tests — not because a developer remembered to check, not because a manager approved a release, but because the pipeline you configured will not allow the deploy stage to run unless the test stage passed. That enforcement is automatic. That standard is yours.

  • The deploy stage has dependsOn: Test. If Test fails, Deploy is skipped — automatically, every time, without exception
  • No approval workflow, no ticket, no Slack message — the pipeline decides based on test results
  • QE teams define what "pass" means: which tests run, which failures are blocking, what threshold triggers a stop
  • When someone asks "how do we ensure quality before deployment?" — this is the answer you can point to and say "I built that"
  • This is not a DevOps feature. This is quality engineering with teeth.
in conversation
Words to use in your next standup

You now have language for these situations:

  • "The deploy stage is blocked because the test stage failed — dependsOn means it will not start unless Test succeeded." — when explaining why a deployment did not happen after a pipeline ran
  • "We use continueOnError: true on the test step so that even when tests fail, the pipeline continues to publish the results. Otherwise we get a failed pipeline with no test detail." — when someone asks why the pipeline is configured the way it is
  • "The condition: always() on the publish step means it runs regardless of whether the tests passed or failed — we always want the results uploaded." — when walking someone through the test reporting setup
  • "The pipeline is the quality gate — it does not let code deploy if tests fail. That is by design." — when someone questions why a deployment was blocked

Key takeaways
  • dependsOn controls which stages run and in what order — with implicit "must succeed" semantics
  • The four conditions: succeeded() (default), failed(), always(), canceled()
  • Always pair continueOnError: true with condition: always() on your PublishTestResults step
  • Blocking deployment on test failure is a pipeline-enforced quality gate — the most important pattern in QE-owned pipelines
  • The quality gate pattern — Test stage required before Deploy stage — is the pipeline enforcing your team's quality standard automatically
