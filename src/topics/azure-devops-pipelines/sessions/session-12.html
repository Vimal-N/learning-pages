BrowserStack: Running Tests on Real Browsers Through Your Pipeline

Running Playwright on a CI agent tests on one browser, on one operating system. BrowserStack gives you a real device and browser matrix — Chrome on Windows 11, Safari on macOS 14, Edge on Windows 10 — without managing any of those machines yourself. This session shows how to connect BrowserStack to Azure Pipelines, how to store credentials securely, and how to read a BrowserStack-enabled pipeline when you see one in your team's codebase.

What BrowserStack is and what it provides
what it is
A cloud of real browsers on real operating systems

BrowserStack Automate is a cloud platform that hosts real browsers on real operating systems, accessible over an API. You connect to it through Playwright's configuration and your tests run there — on BrowserStack's infrastructure, not on the CI agent.

  • Over 3,000 real browser and OS combinations: Chrome, Firefox, Edge, and Safari across Windows, macOS, Android, and iOS
  • No virtual machines to provision — BrowserStack manages the underlying infrastructure entirely
  • Test results, video recordings, screenshots, and network logs appear in the BrowserStack dashboard after each run
  • The ADO agent becomes a thin coordinator — it sends test commands to BrowserStack and collects results, but does not run the browser itself
architecture
Two-tier execution: ADO agent → BrowserStack

Your pipeline runs on an ADO-hosted agent as usual — but instead of launching a browser locally, Playwright connects to BrowserStack's remote endpoint. The tests run there, on real hardware.

What CDP means: CDP stands for Chrome DevTools Protocol — it is the communication standard that Playwright uses to control browsers. When you run Playwright locally, it uses CDP to control a browser installed on your machine. When you run Playwright against BrowserStack, it uses CDP over a secure WebSocket connection to control a browser running on BrowserStack's servers. The protocol is the same; the browser is in a different location.

Storing credentials securely — variable groups
secrets
BrowserStack credentials must never appear in your pipeline YAML

BrowserStack authentication requires a username and an access key. These are sensitive values — if they appear in your YAML file, anyone who can read the repository can use your BrowserStack account and exhaust your test quota. Store them in an ADO variable group and reference the group by name.

  • In ADO: go to Pipelines → Library → Variable Groups → New variable group
  • Add two variables: BROWSERSTACK_USERNAME and BROWSERSTACK_ACCESS_KEY
  • Mark both as secret using the padlock icon — the values will be masked in all pipeline logs
  • Optionally link the group to Azure Key Vault to store the secrets outside ADO entirely, with rotation managed centrally
  • Reference the group in your pipeline YAML with group: BrowserStack-Credentials under the variables: block
Configuration — connecting Playwright to BrowserStack
playwright.config.ts
What the Playwright configuration file is and how it controls where tests run

Playwright reads its runtime configuration from a file called playwright.config.ts (or playwright.config.js) in your repository root. This is where you tell Playwright whether to run tests locally using an installed browser or remotely using BrowserStack's cloud. When you see this file in your team's repository, the use: section at the top is what determines the target.

You do not need to write TypeScript to read this file. The key parts are the use: object and the condition that switches between local and remote execution.

playwright.config.ts — switching between local and BrowserStack
import { defineConfig } from '@playwright/test';

// Check whether BrowserStack credentials are present in the environment
const isCI = process.env.BROWSERSTACK_USERNAME;

export default defineConfig({
  use: isCI ? {
    // BrowserStack remote endpoint — used when credentials are in the environment
    connectOptions: {
      wsEndpoint: `wss://cdp.browserstack.com/playwright?caps=
        {"browser":"chrome","os":"Windows","osVersion":"11",
        "userName":"${process.env.BROWSERSTACK_USERNAME}",
        "accessKey":"${process.env.BROWSERSTACK_ACCESS_KEY}"}`
    }
  } : {
    // Local run — use the installed Chromium browser
    channel: 'chromium'
  },
  reporter: [['junit', { outputFile: 'test-results/junit.xml' }]]
});
reading it
What that configuration file does — without needing to know TypeScript
  • const isCI = process.env.BROWSERSTACK_USERNAME — this checks whether the BROWSERSTACK_USERNAME environment variable exists in the current process. If the pipeline injected it from the variable group, isCI holds that value and is treated as true. If you are running locally with no BrowserStack credentials, it is undefined and treated as false.
  • use: isCI ? {...} : {...} — this is a conditional expression (the ternary operator). It means: "if BrowserStack credentials are available, use the remote endpoint; otherwise, use local Chromium." The ? separates the condition from the true branch; the : introduces the false branch.
  • wsEndpoint — the WebSocket URL that Playwright connects to for remote browser control. It encodes the browser, operating system, and credentials as part of the URL. The credentials come from environment variables injected by the pipeline — they never appear hardcoded in the file.
  • Local vs CI behaviour: when a developer runs tests locally with no BROWSERSTACK_USERNAME in their environment, Playwright uses local Chromium. When the pipeline runs with the variable group injected, it connects to BrowserStack automatically. The same configuration file handles both cases.
azure-pipelines.yml — BrowserStack-enabled pipeline using stages
trigger:
  - main

pool:
  vmImage: 'ubuntu-latest'

variables:
  - group: BrowserStack-Credentials  # injects BROWSERSTACK_USERNAME and BROWSERSTACK_ACCESS_KEY

stages:
  - stage: Test
    displayName: 'Cross-browser tests on BrowserStack'
    jobs:
      - job: BrowserStackTests
        steps:
          - checkout: self
          - task: NodeTool@0
            inputs:
              versionSpec: '20'
          - script: npm ci
            displayName: 'Install dependencies'
          - script: npx playwright test
            displayName: 'Run tests on BrowserStack'
            continueOnError: true
            env:
              # These inject the variable group values as environment variables
              # playwright.config.ts picks them up via process.env
              BROWSERSTACK_USERNAME: $(BROWSERSTACK_USERNAME)
              BROWSERSTACK_ACCESS_KEY: $(BROWSERSTACK_ACCESS_KEY)
          - task: PublishTestResults@2
            condition: always()
            inputs:
              testResultsFormat: 'JUnit'
              testResultsFiles: 'test-results/junit.xml'
              testRunTitle: 'BrowserStack — Chrome on Windows 11'
Two result surfaces — ADO and BrowserStack
two dashboards
Results appear in both ADO and the BrowserStack dashboard — here is what each shows

When tests run via BrowserStack, results appear in two places. You do not need to choose — both are available after every run.

  • ADO Tests tab: shows pass and fail counts by test name, populated from the JUnit XML file published by PublishTestResults@2. Filter by test name, view the stack trace inline, and track trends across runs. This is the primary view for QE reporting within your team's workflow.
  • BrowserStack dashboard: shows a richer view — a video recording of the test session, the network waterfall showing every HTTP request, browser console logs, and a screenshot captured at every step. This is the primary view for debugging browser-specific failures where the error message alone is not enough.
  • Recommended workflow: check the ADO Tests tab first to identify which tests failed. Go to the BrowserStack dashboard for the visual evidence of what the browser actually did during those tests.
  • Finding a session in BrowserStack: log in to your BrowserStack account, go to Automate, and look for the session that matches the timestamp of your pipeline run. The testRunTitle value you set in the pipeline helps identify the correct session in the list.
Decision framework — when BrowserStack makes sense
when to use
BrowserStack in your pipeline — when it makes sense and when it does not
Good fit for BrowserStack
  • You need to verify behaviour on Safari or Internet Explorer — browsers not available on Linux CI agents
  • You have cross-browser compatibility requirements as part of your quality definition
  • Your team already has a BrowserStack Automate licence with sufficient parallel session capacity
  • Cross-browser runs are part of your release acceptance criteria and must pass before deployment
When local agents are enough
  • Tests only need to run on Chromium — which covers the rendering behaviour of the vast majority of web applications
  • You are testing APIs, JSON payloads, or backend logic — nothing that involves rendering
  • Budget constraints — BrowserStack licences add monthly cost per parallel session
  • Speed is the priority — local agent browser launches are faster than establishing a remote connection

Key takeaways
  • BrowserStack Automate provides real browsers on real OS combinations — no virtual machines to provision or manage
  • Playwright connects to BrowserStack via a CDP WebSocket URL configured in playwright.config.ts. The config switches between local and remote based on whether BrowserStack credentials are present in the environment.
  • Store BROWSERSTACK_USERNAME and BROWSERSTACK_ACCESS_KEY in an ADO variable group, not in YAML. Inject them into the test step via the env: block.
  • Results appear in both ADO (Tests tab — pass/fail counts and stack traces) and BrowserStack (dashboard — video, screenshots, and network logs). Use both: ADO for identification, BrowserStack for investigation.
  • Choose BrowserStack when cross-browser coverage is a release requirement. Skip it when Chromium-only testing is sufficient and speed or cost are constraints.
