Parallel Sharding: Splitting a Large Suite

Running 300 Playwright tests sequentially takes 18 minutes. Running them across 3 parallel agents takes 6. This session shows how to split tests across multiple jobs using Playwright's built-in shard flag, how ADO automatically merges the results into a single Tests tab, and when this pattern is worth the added complexity. If you have seen multiple jobs in a Test stage that look nearly identical, your team is probably already using sharding.

The time problem — long suites break the feedback loop
problem
Long test suites break the CI feedback loop

The whole point of CI is fast feedback. If your test suite takes 20 minutes, developers have stopped thinking about the change that triggered it. Parallelism brings the loop back under 10 minutes.

  • A 300-test suite at approximately 3.6 seconds per test runs for 18 minutes on a single agent
  • Split across 3 agents: each agent runs 100 tests, finishing in approximately 6 minutes total
  • Same infrastructure cost: three agents run concurrently, not sequentially — you are not paying for three times the time
  • The developer gets a result before they have had time to start on the next task

What sharding buys you

Playwright sharding — the --shard flag
playwright flag
The --shard flag splits tests by file across agents

Playwright has built-in support for sharding — you tell it which shard this agent is and how many total shards there are. Playwright handles the division automatically.

  • npx playwright test --shard=1/3 runs the first third of all test files on this agent
  • npx playwright test --shard=2/3 runs the middle third on a second agent
  • npx playwright test --shard=3/3 runs the last third on a third agent
  • Playwright distributes by file, not by individual test — each file goes to exactly one shard
  • No code changes needed — the flag is the only configuration required
parallel-shards.yml — three jobs running simultaneously
stages:
  - stage: Test
    displayName: 'Run tests across 3 parallel shards'
    jobs:

      # Shard 1 of 3 — runs the first third of all test files
      - job: Shard1
        displayName: 'Playwright — Shard 1/3'
        pool:
          vmImage: 'ubuntu-latest'
        steps:
          - checkout: self
          - script: npm ci && npx playwright install --with-deps chromium
            displayName: 'Install dependencies and browsers'
          - script: npx playwright test --shard=1/3 --reporter=junit
            displayName: 'Run tests — shard 1/3'
            continueOnError: true
          - task: PublishTestResults@2
            condition: always()
            inputs:
              testResultsFormat: 'JUnit'
              testResultsFiles: 'test-results/junit.xml'
              testRunTitle: 'Shard 1'

      # Shard2 and Shard3 are identical jobs — only the --shard=N/3 flag changes
      - job: Shard2
        displayName: 'Playwright — Shard 2/3'
        # same steps as Shard1 — only --shard=2/3 and testRunTitle: 'Shard 2' differ

      - job: Shard3
        displayName: 'Playwright — Shard 3/3'
        # same steps as Shard1 — only --shard=3/3 and testRunTitle: 'Shard 3' differ
ADO result aggregation — one Tests tab from three jobs
aggregation
ADO merges all shard results into one Tests tab

Each shard job independently publishes its results using PublishTestResults@2. ADO automatically aggregates results from all jobs in the same pipeline run into a single Tests tab.

  • Each job's testRunTitle ('Shard 1', 'Shard 2', 'Shard 3') labels the results separately so you can tell which shard a test ran on
  • The Tests tab shows the total pass and fail count across all three shards combined
  • A single failing test in Shard 2 is visible and searchable in the same view as passing tests from Shard 1 and Shard 3
  • The stage-level result is the union: if any shard job has a failing test, the Test stage is marked as failed
quality gate
The Test stage as a combined quality gate

With all three shards inside the same stage, the stage itself becomes the pass/fail decision point for everything downstream.

  • A downstream Deploy stage with dependsOn: Test will not start unless all three shard jobs pass
  • Any shard job failing causes the Test stage to fail
  • A failed Test stage blocks any downstream stage that depends on it
  • No additional configuration needed — stage-level rollup is how ADO works by default
Sharding vs matrix strategy — knowing the difference
comparison
Sharding and matrix are both parallel patterns — they solve different problems
Sharding — for speed
  • Same tests, same configuration — too many to run in time on one agent
  • Each shard runs a subset of the test files
  • All shards produce equivalent results
  • Use when the suite takes 10 or more minutes on a single agent
  • Results in the Tests tab are labelled by shard number
Matrix — for coverage
  • Same tests, different configurations — browser, environment, or OS
  • Each matrix job runs the full suite (or a filtered subset) against a different target
  • Results differ meaningfully between configurations
  • Use when you have cross-browser or cross-environment requirements
  • Results in the Tests tab are labelled by configuration name

Session 10 covers matrix strategy in full. You can combine both patterns: use matrix for browsers and sharding within each browser job for speed.

When sharding is worth the added complexity
when to use
When sharding is worth the complexity — and when it is not
Good reasons to shard
  • Suite takes longer than 10 minutes on a single agent
  • Feedback delay is causing developers to context-switch away from the change
  • You have parallel agent capacity available in your ADO organisation tier
  • Tests are independent and do not share state or external resources
When NOT to shard
  • Suite takes under 5 minutes — the added complexity is not worth it
  • Tests share a database or external state — shard conflicts will cause intermittent failures
  • Parallel agent capacity is not available in your organisation's tier
  • Tests are not yet deterministic — fix flakiness before adding parallelism
in conversation
Words to use in your next standup

You now have language for these situations:

  • "The Test stage has three shard jobs — they run in parallel, each handling a third of the test files. Total time is the duration of the slowest shard, not the sum." — when explaining why the pipeline has three nearly-identical jobs in the Test stage
  • "Each shard publishes its own results to the Tests tab — ADO merges them automatically. You see all 300 tests in one view." — when someone asks how the results get combined
  • "If any shard fails, the Test stage fails and deployment is blocked. All three have to pass." — when explaining the quality gate behaviour
  • "The difference between sharding and matrix: sharding splits the same tests for speed, matrix runs the same tests in different configurations for coverage." — when clarifying which pattern is being used and why

Key takeaways
  • Playwright's --shard=N/M flag splits test files across M agents — no code changes required, just a flag
  • Each shard is a separate job in the same stage; jobs run in parallel on separate agents
  • Each shard publishes its own PublishTestResults@2; ADO aggregates them into one Tests tab automatically
  • The Test stage fails if any shard job fails — this makes the stage the combined quality gate for all downstream stages
  • Sharding is for speed with one configuration. Matrix (Session 10) is for coverage across multiple configurations. Combine both for maximum parallelism.
  • Shard when the suite takes 10 or more minutes. Skip it when tests share state or the suite is already fast.
