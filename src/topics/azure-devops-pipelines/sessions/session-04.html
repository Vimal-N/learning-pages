Azure DevOps: The Whole Suite

Azure Pipelines is one product inside Azure DevOps — a larger suite of five integrated services. Understanding what surrounds Pipelines explains why ADO workflows feel cohesive and why integrations that would require plugins elsewhere just work out of the box.

The five services
overview
Azure DevOps is a suite of five services under one roof

Each service covers a distinct phase of the software delivery lifecycle. They share the same organisation, the same IAM, and the same project structure — no stitching required.

Azure Boards
Work items, epics, user stories, sprints, and backlogs. Kanban and Scrum boards built in.
Azure Repos
Git repositories hosted inside ADO. Branch policies, PR reviews, and required builds live here.
Azure Pipelines
CI/CD engine. Build, test, and deploy using YAML pipelines on hosted or self-hosted agents.
Azure Artifacts
A private store for shared code libraries your team builds and reuses across projects.
Azure Test Plans
Manual and exploratory test management. Test cases, test runs, and results link directly to pipeline results.
what is a package?
A package is a bundle of reusable code someone else already wrote

Every time you run npm install playwright or npm ci, you're downloading packages — zip files containing code your project depends on, published to a central registry. You don't write that code; you just declare you need it and the tool fetches it.

  • A package has a name, a version number, and a list of its own dependencies
  • npm is the public registry for JavaScript packages — millions of open-source libraries live there
  • NuGet is the equivalent for .NET — Microsoft and the community publish packages there
  • Most projects depend on dozens or hundreds of packages fetched automatically on build
why Azure Artifacts?
Some packages are too sensitive or too specific to publish publicly

Your team might build a shared login library, a test utilities module, or an internal API client that shouldn't be published to npm for the world to see. Azure Artifacts is your private package registry — your team publishes to it, and only your organisation can consume from it.

  • Think of npm as the public library — anyone can borrow any book
  • Azure Artifacts is your team's private bookshelf — your custom books, internal use only
  • Your pipeline publishes a new version to Artifacts when the build passes; other pipelines consume it
  • For most QE engineers: you consume packages (via npm ci) but rarely publish them — knowing Artifacts exists explains where internal libraries come from
context
Pipelines connects to everything else

When a pipeline runs tests, the results flow automatically into Azure Test Plans. When it publishes a package, that package goes to Azure Artifacts. Branch policies in Azure Repos can require a pipeline to pass before a PR can merge. These connections come for free — they don't need configuration.

  • A branch policy in Repos can block merges until a linked pipeline passes
  • Test results published with PublishTestResults@2 appear in the Tests tab and in Test Plans
  • Packages published with NuGetCommand@2 or Npm@1 land in the project's Artifacts feed
  • Work items in Boards can be linked to specific pipeline runs
Reading YAML — the format pipelines are written in
foundation
YAML is a structured way to write configuration — like a very strict outline

YAML stands for "YAML Ain't Markup Language." You do not need to remember that. What matters is this: YAML is how Azure Pipelines definitions are written. It is a text format where indentation controls structure — the number of spaces at the start of a line determines what belongs to what. There are three things to know to read any YAML file.

The three YAML building blocks
  • Key-value pair: vmImage: 'ubuntu-latest' — a property name, a colon, a space, then the value. This is the most common pattern.
  • List item: - main — a dash followed by a space starts a list item. A list of branches, a list of steps, a list of stages.
  • Nested structure: Indentation (always spaces, never tabs) creates hierarchy. A property indented under another belongs to it.
The one rule that catches everyone
  • Indentation in YAML is not decorative — it is structural. Two spaces vs four spaces means different things.
  • A single wrong indent can make a valid pipeline invalid. If a pipeline fails to start with an error about "mapping" or "unexpected key," indentation is almost always the cause.
  • Every code example in this course is correctly indented. When in doubt, copy the indentation exactly.
  • Use a code editor (VS Code) that shows you indentation guides — it makes misalignment immediately visible.

  • Step 1 — Top-level keys: trigger, pool, variables, stages — these are the outer shape of every pipeline. Find them first. They tell you what triggers the pipeline, where it runs, what variables it uses, and what phases it has.
  • Step 2 — Stages in order: Each stage is a named phase (Build, Test, Deploy). Read the displayName of each — it tells you what that phase does without reading the implementation.
  • Step 3 — Template references: When you see - template: path/to/file.yml@alias, the actual steps are in another file. The parameters: block below it shows what is being customised. Session 08 and 09 cover this in full.
  • Step 4 — Variable references: $(variableName) is a runtime value — find where it is defined (in the variables: block, a variable group, or a matrix). That is what it resolves to when the pipeline runs.
  • Step 5 — Unknown tasks: A task like PublishTestResults@2 or NodeTool@0 — the name before @ is the task name, the number after is the version. Search ADO's task documentation or your team's wiki for details.
Classic vs YAML pipelines
formats
Classic (GUI) pipelines vs YAML pipelines

ADO supports two pipeline formats. Classic pipelines are built in the portal using a visual editor. YAML pipelines are defined in a file in your repository. Both produce the same outcome — but YAML is the modern approach and should be used for all new work.

Classic (GUI) pipelines
  • Configured in the ADO portal — no file in the repo
  • Drag-and-drop task editor
  • Cannot be code-reviewed or version-controlled
  • Changes aren't auditable without portal logs
  • Still used in some legacy projects
YAML pipelines (modern)
  • Defined in azure-pipelines.yml in your repo
  • Versioned with your code — PRs, history, blame
  • Can be reviewed before it runs
  • Supports templates, reuse, and parameters
  • Use this for all new pipelines
minimal-pipeline.yml — the smallest valid pipeline
# Triggers on every push to main
trigger:
  - main

# Use a Microsoft-hosted Ubuntu agent
pool:
  vmImage: 'ubuntu-latest'

steps:
  - checkout: self
  - script: npm ci
    displayName: 'Install dependencies'
  - script: npm test
    displayName: 'Run tests'
line by line
What every line in that minimal pipeline does

Every line above does a specific, necessary thing. Nothing is boilerplate you can ignore.

  • trigger: - main — "run this pipeline automatically whenever code is pushed to the main branch." Without a trigger, the pipeline only runs when someone manually starts it.
  • pool: vmImage: 'ubuntu-latest' — "run this pipeline on a Microsoft-hosted Ubuntu virtual machine." This machine is created fresh for this run and destroyed when it finishes. Session 05 explains exactly what this means.
  • steps: — the list of things to do on that machine, in order.
  • - checkout: self — "download the code from this repository onto the machine." Without this, the machine has no code to work with. self means "this repository" — the one containing this pipeline file.
  • script: npm ci — run the shell command npm ci, which installs the exact package versions from the lock file. npm is Node's package manager. ci stands for "clean install" — it guarantees a reproducible installation.
  • script: npm test — run the shell command npm test, which executes whatever test script is defined in the project's package.json.
  • displayName — the human-readable label shown in the ADO portal for this step. It does not affect what the step does.
Organisation structure
hierarchy
Organisation → Project → Pipeline → Run

ADO is structured in a four-level hierarchy. Understanding this helps you navigate the portal and reason about permissions, billing, and shared resources.

1
Organisation
→
2
Project
→
3
Pipeline
→
4
Run
  • Organisation — your company's ADO account (e.g. dev.azure.com/mycompany). One billing account.
  • Project — a logical grouping for a team or product. Has its own Repos, Boards, Pipelines, and Artifacts.
  • Pipeline — a YAML file linked to a specific repo. Defines all the stages, jobs, and steps.
  • Run — one execution of the pipeline, triggered by a push, a PR, or manually.
in conversation
Words to use in your next standup

You now have language for these situations:

  • "YAML is the format our pipelines are written in — it uses indentation to show structure, and every level of indentation must use spaces, not tabs." — when someone asks what format the pipeline file uses, or when explaining why a pipeline edit broke something
  • "The trigger: - main line means the pipeline fires automatically on every push to main — nobody has to remember to run it." — when explaining how pipelines get triggered, or when investigating why a pipeline did or did not run after a merge
  • "The checkout: self step is what puts our code onto the agent machine — without it, the machine has nothing to work with." — when reading through a pipeline together and someone asks what that line does
  • "A $(variableName) in a pipeline is a placeholder — the actual value comes from the variables section, a variable group, or the matrix configuration." — when reviewing a pipeline and someone asks what a $() expression resolves to

Key takeaways
  • Azure DevOps is five services: Boards, Repos, Pipelines, Artifacts, and Test Plans — Pipelines is one piece
  • Classic GUI pipelines are legacy. Use YAML pipelines for all new work.
  • YAML pipelines live in your repo — they're versioned, reviewable, and auditable
  • YAML uses indentation as structure — read it top to bottom, outermost keys first, then drill into each section
  • The hierarchy is: Organisation → Project → Pipeline → Run
