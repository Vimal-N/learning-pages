Phase 2 · CI/CD Market Survey

The CI/CD Tool Market in 20 Minutes

Before going deep on Azure Pipelines, it helps to know what else is out there — because you will hear these names in team conversations, job descriptions, and migration discussions. This session gives you enough context to follow those conversations and ask good questions. The deeper insight: every CI/CD tool implements the same underlying structure. Once you understand Azure Pipelines, you can read a GitHub Actions or GitLab CI file and immediately recognise the equivalent concepts.

why this matters
You will hear these names — here is what they mean

CI/CD tools come up in conversations all the time: "we migrated from Jenkins to GitHub Actions," "this repo uses GitLab CI," "we are looking at CircleCI for the frontend team." You do not need to be an expert in every tool. But knowing what category each tool belongs to — and what the equivalent concepts are in Azure Pipelines — means you can follow those conversations and contribute to them.

  • You are not selecting a CI/CD tool in this session — your team has already chosen Azure Pipelines
  • You are building a mental map so that no tool name catches you off guard
  • The most important panel in this session is the portability table at the end — everything maps to a concept you will learn in ADO
Three categories of tools
landscape
CI/CD tools fall into three categories

Despite dozens of products, the market clusters around three delivery models. Understanding the category tells you most of what you need to know about the trade-offs before reading any documentation.

Cloud SaaS
Hosted, no infrastructure to manage. You bring the code; the platform brings the runners. Examples: GitHub Actions, CircleCI, Travis CI.
Self-hosted
You own the server, the agents, and the maintenance. Full control over the environment. Example: Jenkins.
Platform-embedded
CI/CD built into a broader DevOps suite. Pipelines share an organisation, repo, boards, and artifact store. Examples: Azure DevOps, GitLab CI/CD.
Major tools at a glance
github actions
GitHub Actions — the ecosystem play

Tightly integrated with GitHub. Workflows live in .github/workflows/, trigger on GitHub events, and can call thousands of community-built "Actions" from the GitHub Marketplace.

  • YAML-based, event-driven triggers (push, PR, issue, release, schedule)
  • Free tier generous; scales with GitHub billing
  • Best fit: open-source projects or teams already on GitHub
  • ADO comparison: very similar YAML syntax — jobs, steps, uses vs ADO's task
jenkins
Jenkins — the original, still everywhere

Open-source, self-hosted, and been around since 2004. Every enterprise CI/CD concept was either invented here or implemented here first. It's verbose and high-maintenance, but it goes anywhere.

  • Pipelines defined in Groovy (Jenkinsfile) — older than YAML pipelines
  • Massive plugin ecosystem (1,800+ plugins)
  • You own the controller, the agents, the upgrades, and the crashes
  • Best fit: teams with custom infrastructure requirements or legacy migration paths
  • ADO comparison: Jenkins "stages" ≈ ADO "jobs"; Jenkins "steps" ≈ ADO "steps"
gitlab ci/cd
GitLab CI/CD — the integrated alternative

GitLab bakes CI/CD directly into its platform — same idea as ADO. Pipelines are defined in .gitlab-ci.yml. If your organisation self-hosts GitLab, you get all of this on-premise.

  • YAML-based, pipeline runs triggered on push, MR (merge request), schedule
  • stages:, jobs:, artifacts: — nearly identical mental model to ADO
  • Strong compliance features, good for regulated industries
  • ADO comparison: GitLab MR pipelines ≈ ADO PR validation; GitLab runners ≈ ADO agents
circleci
CircleCI — fast loops for smaller teams

A cloud-native CI/CD service known for speed optimisations: test parallelism, intelligent caching, and fast spin-up times. Popular in start-ups and product teams that need quick CI cycles.

  • YAML-based (.circleci/config.yml), Docker-first execution
  • Built-in test splitting and parallel job execution
  • Usage-based pricing — can get expensive at scale
  • ADO comparison: CircleCI parallel test splitting ≈ ADO parallel jobs — same concept, different syntax.

why Azure DevOps
ADO is the right tool for enterprise teams on Microsoft's stack

If your organisation uses Azure cloud resources, Microsoft-hosted build agents, Azure Repos, and Azure Test Plans — ADO Pipelines is the natural choice. Everything is in the same portal, under the same IAM, billed on the same account.

  • Native integration with Azure Repos, Azure Boards, Azure Artifacts, and Azure Test Plans
  • Microsoft-hosted agents for Windows, Linux, and macOS — no infrastructure to provision
  • Service connections to Azure subscriptions, Kubernetes clusters, and App Services
  • Variable groups link directly to Azure Key Vault secrets
  • For QE teams: the Tests tab in the pipeline summary integrates directly with Azure Test Plans

session 03
Key takeaways
  • CI/CD tools fall into three categories: cloud SaaS, self-hosted, and platform-embedded
  • Every tool implements the same concepts: trigger → agent → steps → artefact
  • GitHub Actions and GitLab CI are the closest analogues to ADO — YAML-based, stage/job/step structure
  • Jenkins is the most powerful but most expensive to operate; it's where enterprise CI/CD conventions were established
  • ADO is the right fit when you're already inside the Microsoft ecosystem — Azure, Visual Studio, Windows, and regulated enterprise environments
