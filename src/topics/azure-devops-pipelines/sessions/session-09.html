Reading and Reusing Your Organisation's Shared Template Library

Your product team's pipeline almost certainly calls templates from a shared repository your organisation maintains. You have probably seen lines like - template: templates/test/playwright-e2e.yml@templates without knowing where that file lives or what it does. This session shows you how to find that repository, read its templates, understand the parameters they accept, and write your own QE pipeline that uses them — for a nightly regression run or a daily smoke run — without reinventing anything your organisation has already built.

What the shared template repository is
the library
Your organisation's shared template repository is the pipeline standard

Most organisations that use Azure Pipelines at scale maintain one dedicated repository containing standardised pipeline definitions. Every product team's pipeline references this repository rather than writing the same patterns from scratch. QE engineers see references to it in every team pipeline but rarely know it exists as a named, browsable repository in ADO.

  • The repository has a name — commonly ado-pipeline-library, pipeline-templates, devops-templates, or similar. Ask your platform team or DevOps lead if you are not sure what yours is called.
  • It is a normal ADO (Azure DevOps) repository — you can browse it, read its files, and see its history in ADO Repos, exactly like any other codebase.
  • It is owned and maintained by a platform or DevOps team. QE engineers consume it — you are not expected to modify it.
  • Templates inside it are organised by function: templates/build/, templates/test/, templates/deploy/. The test templates are the ones most relevant to QE work.
  • Understanding what is in this repository is the difference between feeling like pipeline YAML is a black box and knowing exactly what your team's pipeline is doing.
folder structure
What a shared template repository typically looks like inside

When you browse the repository in ADO Repos, you will find a folder structure organised by pipeline phase. The exact names vary by organisation, but the pattern is consistent across most teams that use shared templates.

ado-pipeline-library/
├── templates/
│   ├── build/
│   │   ├── install-and-lint.yml       ← compiles, lints, sets up Node
│   │   └── dotnet-build.yml           ← builds .NET solutions
│   ├── test/
│   │   ├── playwright-e2e.yml         ← runs Playwright tests (the one you want)
│   │   ├── jest-unit.yml              ← runs Jest unit tests
│   │   └── api-contract-tests.yml     ← runs API contract tests
│   └── deploy/
│       ├── azure-webapp.yml           ← deploys to Azure App Service
│       └── kubernetes.yml             ← deploys to AKS (Azure Kubernetes Service)
└── tasks/
    ├── publish-results.yml           ← publishes test results to ADO
    └── notify-teams.yml             ← sends Teams notification on failure
  • The templates/test/ folder is where QE-relevant templates live. Start there when exploring the shared library for the first time.
  • Each template file is self-contained — it defines a stage, a job, or a set of steps that accomplishes a specific task from start to finish.
  • The tasks/ folder contains smaller reusable step definitions — often called from within the larger templates to avoid duplicating common actions like publishing results.
Decoding a template reference — reading what your team's pipeline already uses
anatomy
Every template reference has three parts — here is what each one means

When you see a template call in a product team's pipeline, you can read it completely by understanding three separate pieces. Here is a fully annotated example:

resources:
  repositories:
    - repository: templates          # alias — used as @templates below
      type: git                        # it is a Git repository
      name: 'MyOrg/ado-pipeline-library'  # Organisation/RepositoryName in ADO
      ref: 'refs/tags/v4.1.0'            # pinned version — safe to copy from team's pipeline

stages:
  - template: templates/test/playwright-e2e.yml@templates
    #           └─ file path inside the repo ──────────────┘└─ alias ─┘
    parameters:
      nodeVersion: '20'             # values passed to customise the template
      testFilter: '@regression'        # these parameter names come from the template file
      environment: 'qa'
  • The alias (@templates): This is the short name assigned to the repository in the resources.repositories block. It could be @shared, @central, or anything else your organisation chose. Look up at the resources block to see which actual repository that alias refers to.
  • The path (templates/test/playwright-e2e.yml): This is the file path inside the external repository. To read what this template actually does, navigate to that repository in ADO Repos and open this file. The next section shows exactly how.
  • The parameters block: These are the values being passed to customise the template for this specific pipeline. To know what parameters are available — which ones are optional, which are required — open the template file and read its parameters: block at the top.
ref versions
Understanding the ref — which version of the template library you are using

The ref field tells ADO exactly which snapshot of the shared template repository to use when it runs your pipeline. It is one of the most important lines in any pipeline that references an external template repo.

  • refs/tags/v4.1.0 means the pipeline is using the version of the shared repository tagged as v4.1.0. This is a snapshot — even if the templates are updated tomorrow, this pipeline keeps using v4.1.0 until someone deliberately changes the ref.
  • Teams pin to a specific tag so that template updates from the platform team do not unexpectedly break their pipeline. This is the correct approach — it gives you stability and control over when you take on changes.
  • refs/heads/main means "always use the latest version of the main branch" — this is less stable and should only be used if the platform team explicitly recommends it, because any breaking change they push immediately affects your pipeline.
  • What to use when writing a new pipeline: Copy the ref value from your product team's existing pipeline. That version is known to work. You can update it later when the platform team releases a new version.
  • If you are not sure what tag to use, ask the platform or DevOps team what the current stable version is. They maintain a changelog — usually a CHANGELOG.md in the shared repo root or a dedicated Teams channel.

  • Step 1 — Open ADO: Go to your organisation's top-level URL, for example dev.azure.com/YourOrg. This is the starting point for finding any repository in your organisation.
  • Step 2 — Find the project: Look for the ADO project that houses the shared templates. It is often named Platform, DevOps, Infrastructure, or the same name as the repository itself. Ask your DevOps lead if you are not sure which project to look in — this is a perfectly reasonable question.
  • Step 3 — Go to Repos: Inside that project, click Repos in the left navigation and select the template repository (e.g. ado-pipeline-library) from the repository dropdown.
  • Step 4 — Switch to the tag: In the branch dropdown near the top of the file browser, switch to the tag that matches the ref in your pipeline (e.g. v4.1.0). Look in the Tags section of the dropdown, not the Branches section — they are separate lists.
  • Step 5 — Navigate to the file: Click through the folders to reach the file path from the template call: for example, templates/test/playwright-e2e.yml. Click the filename to open and read the template.

You are now reading the exact template that your team's pipeline calls. Bookmark this file — you will come back to it when you need to understand what the template does or what parameters you can use.

Reading a template file — the parameters block is the contract
parameters block
The parameters: block at the top of every template is the complete list of what you can control

When you open a template file, go straight to the top. The parameters: block tells you everything a caller can customise. You do not need to read the rest of the file to know how to use the template.

parameters:
  - name: nodeVersion          # the parameter name — use this in your pipeline's parameters block
    type: string
    default: '20'              # optional — if you omit this, '20' is used

  - name: testFilter           # no default — this is required, you must pass a value
    type: string

  - name: environment          # optional with a default
    type: string
    default: 'qa'

  - name: publishResults       # boolean parameter
    type: boolean
    default: true
  • Has a default → optional. You can omit this parameter in your pipeline and the template uses the default value. Most parameters have sensible defaults that work for the common case.
  • No default → required. You must pass a value or the pipeline will fail to start with a validation error before any step runs.
  • The parameter names here (nodeVersion, testFilter, environment) are exactly what you write in the parameters: block of your template call — no guessing required.
  • You do not need to read the rest of the template file to use it correctly. The parameters: block is the complete user interface for the template.
when you need more
Reading the template body — only when you need to understand what a step actually does

Occasionally you need to understand not just what parameters exist but what the template does with them — for debugging a failure, for understanding why a step takes a certain amount of time, or for verifying that a required step (like publishing test results) is already handled inside the template so you do not add it twice.

  • Scan for recognisable task names: PublishTestResults@2, PublishPipelineArtifact@1, NodeTool@0. If you see PublishTestResults@2 in the template body, you do not need to add it to your own pipeline — the template already handles it.
  • Look for condition: always() on the publish steps — if it is there, results will be published even when tests fail, which is the behaviour you need for a useful failure report.
  • If you see continueOnError: true on the test step, the template is already handling the "publish even on failure" pattern correctly and you do not need to add anything.
  • If you cannot work out what the template does from reading it, ask the platform team. The question "does the playwright-e2e template publish the HTML report as an artifact?" is a clear, well-formed question that any platform engineer can answer in 30 seconds.
Writing your own pipeline that uses shared templates
the goal
A QE-owned nightly regression pipeline — using your organisation's templates

You do not need to write the install steps, the browser installation, the test runner command, or the publish steps. Your organisation's template already does all of that correctly, and the platform team maintains it. Your job is to call the template with the right parameters and add a schedule. Here is what that looks like.

nightly-regression.yml — a complete QE-owned pipeline using shared templates
# QE-owned nightly regression pipeline
# Runs every weeknight at midnight against the QA environment

schedules:
  - cron: '0 0 * * 1-5'               # midnight, Monday–Friday
    displayName: 'Nightly regression — QA'
    branches:
      include:
        - main
    always: true                     # run even if no new commits since last run

resources:
  repositories:
    - repository: templates
      type: git
      name: 'MyOrg/ado-pipeline-library'
      ref: 'refs/tags/v4.1.0'          # copy this from your product team's pipeline

variables:
  - group: QA-Environment-Credentials  # variable group with env URLs and credentials

stages:
  - template: templates/test/playwright-e2e.yml@templates
    parameters:
      nodeVersion: '20'
      testFilter: '@regression'        # runs the full regression suite
      environment: 'qa'              # targets the QA environment
      publishResults: true          # publishes results to ADO Tests tab
line by line
What every section of that pipeline does
  • schedules: — defines when the pipeline runs automatically. cron: '0 0 * * 1-5' means midnight every Monday–Friday. always: true means it runs even if no new code was pushed since the last run — important for a nightly regression suite that should always verify the application's state.
  • resources.repositories: — declares the shared template repository so the pipeline can reference files from it. Copy this block from your product team's pipeline, changing only the ref if you need a different version.
  • variables: - group: — pulls in a variable group from ADO Library. A variable group is a named collection of secrets and configuration values stored securely in ADO — things like environment base URLs and service account credentials. Ask your DevOps team what variable group to use for QA environment access.
  • stages: - template: — calls one template from the shared library, passing four parameters. The template does all the actual work: installing Node, installing browsers, running Playwright, publishing results, and publishing the HTML report as a pipeline artifact.
  • This entire file is 22 lines. A pipeline written from scratch to do the same thing would be 60–80 lines. The shared template does the rest — and the platform team is responsible for keeping it correct and current.
which ref to use
Choosing the right version of the shared template library
The safe starting point
Copy the ref value from your product team's existing pipeline. That version is proven — your team's pipeline runs every day on it. Using the same ref guarantees compatibility between the template version and the environment configuration your team already uses.
Staying current
When the platform team releases a new version, they will communicate it. Check the shared repo's Tags page and any release notes (often in a CHANGELOG.md file in the repo root or a dedicated Teams channel). Update the ref in your pipeline after verifying the new version does not change behaviour you depend on.
What not to do
Do not use ref: refs/heads/main. Pinning to the main branch means any breaking change the platform team pushes will silently break your nightly pipeline at the next scheduled run, with no warning and no way to roll back without changing the ref anyway.
when to ask for help
You are not expected to understand every line — here is who to ask and what to ask

The platform team maintains the shared template library and expects questions from consuming teams. Asking a specific question about a template is not a sign of not knowing enough — it is correct use of the system. Here are examples of well-formed questions you can ask right now:

  • "Does playwright-e2e.yml publish the HTML report as an artifact, or do I need to add that step myself?"
  • "The nightly run is failing with a testFilter validation error — what format does that parameter expect? Can I pass multiple tags like @regression,@smoke?"
  • "We are on v4.1.0. The changelog says v4.2.0 adds browser caching. How do I update my pipeline's ref safely?"
  • "Our QA environment requires VPN access. Does the template support self-hosted agents, or is it Microsoft-hosted only?"
  • These questions are specific, clear, and answerable in a short conversation. They demonstrate that you understand the system well enough to know exactly what you do not know — which is exactly the right place to be.
in conversation
Words to use in your next standup

You now have language for these situations:

  • "This pipeline calls a template from our shared library — the actual install and test steps are defined in ado-pipeline-library, not in this file." — when explaining why a pipeline file looks short or why someone cannot find the test runner command in it
  • "The ref: refs/tags/v4.1.0 means we are pinned to version 4.1.0 of the shared template repo — we will not pick up any changes until we update that ref." — when the team asks why a template update from the platform team has not taken effect yet
  • "I checked the template's parameters: block — testFilter is required with no default, so we have to pass a value. publishResults is optional and defaults to true." — when discussing what needs to be configured before running a new pipeline
  • "I set the schedule to midnight Monday–Friday with always: true — it runs every night regardless of whether new code was pushed." — when explaining the nightly pipeline behaviour to the team
session 09
Key takeaways
  • Your organisation's shared template repository is a browsable ADO repo — find it in the platform or DevOps team's ADO project, under Repos. It is not hidden; you just need to know which project to look in.
  • A template reference has three parts: the alias (which repo), the path (which file), and the parameters (what to customise). The parameters: block in the template file is the complete list of what you can control — required parameters have no default, optional ones do.
  • A ref value pins the pipeline to a specific version of the shared library — copy it from your product team's pipeline and update it deliberately, not automatically. Never pin to refs/heads/main.
  • A QE-owned nightly pipeline can be as short as 22 lines when it calls shared templates — you configure the schedule, the environment, and the test filter; the template does the rest.
  • When you do not understand something about a shared template, ask the platform team a specific question. "Does this template publish the HTML report?" is always the right question — it is clear, answerable, and shows you know how the system is structured.
