Phase 3 · Pipeline Internals

What Happens When a Pipeline Runs

Push a commit and a pipeline fires — but what actually happens between that push and the test results appearing in your PR? This session walks through every step: agent provisioning, checkout, execution, result publishing, and teardown. Then we look at how steps, jobs, and stages nest together into a coherent structure.

The agent lifecycle
analogy
The borrowed laptop

Every pipeline job is handed a brand-new laptop that has never been used before. Your code appears on it. Your tasks run on it. When the job finishes, the laptop is destroyed. No state persists. No one else's build history is on it. This is the core of reproducibility.

  • Each job gets a clean virtual machine — no shared state between runs
  • Software is pre-installed (Node, .NET, Python, Git, Docker) but your packages are not
  • After the job ends, the VM is deallocated — nothing you wrote persists
  • This is why npm ci (not npm install) matters — you need a fresh, exact install every time

  • Provision: ADO allocates a hosted VM from the pool. Typically takes 20–60 seconds.
  • Checkout: The agent runs git clone and checks out your commit. This is the checkout: self step.
  • Execute: Each step in your YAML runs in order. If a step fails, subsequent steps are skipped by default.
  • Publish: Test results, logs, and artefacts are uploaded to ADO before the VM shuts down.
  • Destroyed: The VM is gone. Nothing you installed or wrote persists to the next run.
agent types
Microsoft-hosted vs self-hosted agents

You choose which type of agent runs each job. Microsoft-hosted agents are the default for most teams — zero setup, fresh every time. Self-hosted agents are for teams that need custom environments or access to private networks.

Microsoft-hosted
  • Pre-configured VMs managed by Microsoft
  • Always fresh — no state between runs
  • Choose: ubuntu-latest, windows-latest, macos-latest
  • Free tier: 1,800 min/month
  • Can't access private subnets or on-premise systems
Self-hosted
  • Your machine, registered to your ADO organisation
  • Can run inside a private network
  • Access to internal databases, APIs, artefact stores
  • State persists between runs (good and bad)
  • You own maintenance, capacity, and uptime
Steps, jobs, and stages
hierarchy
Three levels of structure: steps, jobs, and stages

A pipeline isn't just a flat list of commands. It has three nesting levels that control parallelism, dependencies, and which machine things run on.

  • Step — the smallest unit. A single task or script command. Steps run sequentially within a job, on the same agent.
  • Job — a collection of steps. Each job gets its own agent. Jobs within a stage run in parallel by default (unless you add dependsOn).
  • Stage — a collection of jobs. Stages run sequentially by default. Used to model phases: Build → Test → Deploy. Each stage appears as a column in the pipeline summary.
stages-jobs-steps.yml — three levels of structure
stages:
  - stage: Build
    jobs:
      - job: CompileAndLint
        pool:
          vmImage: 'ubuntu-latest'
        steps:
          - checkout: self
          - script: npm ci
          - script: npm run lint

  - stage: Test
    dependsOn: Build
    jobs:
      - job: RunTests
        pool:
          vmImage: 'ubuntu-latest'
        steps:
          - checkout: self
          - script: npm ci && npm test

What you can do right now

Before moving into flow control and more complex patterns, take a moment to recognise what you have just built. At this point in the course you can:

  • Describe what happens between a push and a test result — agent provisions, code checks out, steps run, results publish, machine is destroyed. You can explain this to a developer or a manager.
  • Read a pipeline YAML and understand its shape — you can identify the stages, jobs, and steps, and know which runs on which agent and in what order.
  • Explain why CI results are more trustworthy than local results — fresh environment, exact lock-file versions, no ambient state, reproducible every time.
  • Distinguish between Microsoft-hosted and self-hosted agents — and know when you would need each type.
  • These are not small things. Most people in your organisation who work adjacent to pipelines cannot articulate all four of these clearly.
Common pitfalls
watch out
The four mistakes that trip up new pipeline authors

Most early pipeline problems come from forgetting that each job is isolated — nothing is shared between jobs unless you explicitly pass it.

  • Forgetting checkout: self — the agent starts with an empty disk. checkout: self tells the agent to download this repository's code. The word self means "this repository — the one that contains this pipeline file." You can also check out other repositories by name, but self is by far the most common. Without checkout: self, every subsequent step fails silently because there are no files to work with.
  • Using npm install instead of npm ci — npm install can upgrade packages and update the lock file. npm ci installs exactly what's in package-lock.json. Always use npm ci in CI.
  • Assuming files carry over between jobs — each job gets a fresh agent. If job A builds a binary, job B cannot see it unless job A publishes it as a pipeline artefact first.
  • Ignoring agent queuing time — pipeline duration includes the time waiting for an available agent. If your pipeline seems slow, check the provisioning time vs execution time in the job logs.
in conversation
Words to use in your next standup

You now have language for these situations:

  • "The pipeline is queued — the agent hasn't provisioned yet. We are waiting for a VM to become available." — when a pipeline run is taking longer than expected to start
  • "Each job gets a fresh agent — nothing from the Build job carries over to the Test job unless we explicitly publish it as an artifact." — when explaining why a file from a previous step isn't available
  • "We use npm ci in the pipeline, not npm install — ci does a clean install from the lock file every time, which is what makes the results reproducible." — when someone asks why the pipeline uses a different install command
  • "The checkout: self step is what puts our code on the agent — it is the first step in every job that needs to work with our files." — when explaining pipeline setup or debugging a missing-file error

Key takeaways
  • Every job runs on a fresh, ephemeral VM — no state persists between runs or between jobs
  • The agent lifecycle: Provision → Checkout → Execute → Publish → Destroy
  • Steps run sequentially on one agent. Jobs run in parallel by default (on separate agents). Stages run sequentially.
  • Microsoft-hosted agents are zero-maintenance and always fresh. Self-hosted agents have network access but require you to manage them.
  • Always use npm ci (not npm install) and always include checkout: self
