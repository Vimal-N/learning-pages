Phase 4 · Patterns You'll See in Your Team's Pipelines

Templates: Write Once, Run Everywhere

Your product team's pipeline almost certainly uses templates — you have probably seen a line like - template: templates/test/playwright-e2e.yml@templates without knowing what it does. This session explains what templates are, how to recognise a template call in a pipeline you did not write, and how reusing your organisation's shared templates is different from writing your own. The next session covers the shared template library your organisation maintains — the one your team already uses.

The problem templates solve
before
The copy-paste pipeline problem

Most teams start by copying their working pipeline into a second project when they need the same setup. Then a third. Soon there are eight slightly-different versions of the same Node install + test pattern across eight repositories.

  • Upgrading Node 18 → 20: find and edit 8 files across 8 repos
  • Adding a security scan step: do it 8 times, hope they're all consistent
  • Finding a bug in the lint configuration: patch 8 files, open 8 PRs
  • Auditing whether all pipelines comply with the team's standards: check 8 files manually
  • Templates collapse this to: edit the template once, all consumers are updated
types
Three kinds of templates

ADO templates can operate at three levels of granularity. Choose based on how much you want to reuse: individual steps, whole jobs, or entire stages.

Step template
Reuse a sequence of steps. Insert into any job's steps: list. Most granular.
Job template
Reuse an entire job definition — pool, variables, and steps. Insert into any stage's jobs: list.
Stage template
Reuse a complete stage with all its jobs. Insert into a pipeline's stages: list. Most powerful.
Reading a template call — when you see it in someone else's YAML
consumer perspective
What to do when you see - template: in a pipeline file

When you open a pipeline file and see a template call, your job is not to understand its internals — it is to understand what it does and what parameters control it. Here is a template reference annotated completely:

# This block declares an external repository of templates
resources:
  repositories:
    - repository: templates            # alias — used below as @templates
      type: git                        # it is a Git repository
      name: 'MyOrg/ado-pipeline-library'   # Organisation/RepositoryName in ADO
      ref: 'refs/tags/v4.1.0'            # pinned version — safe to copy from team's pipeline

stages:
  # The template file path inside the repo @alias from resources.repositories
  - template: templates/test/playwright-e2e.yml@templates
    parameters:
      nodeVersion: '20'               # customise the template behaviour here
      testFilter: '@regression'          # these come from the template's parameters block
three things to know
Decoding any template reference in three steps
  • The alias (@templates): This is the short name given to the repository in the resources.repositories block. It could be @shared, @central, or anything. Look up at the resources block to find which actual repository it points to.
  • The path (templates/test/playwright-e2e.yml): This is the file path inside the external repository. To read what the template actually does, navigate to that repository in ADO and open this file at this path. Session 09 shows exactly how to do this for your organisation's shared library.
  • The parameters block: These are the values being passed to customise the template. To know what parameters are available — what you can change and what is optional — open the template file and read its parameters: block at the top. That block is the full list of what you can control.
In-repo templates
in-repo
Reusing steps within the same repository

The simplest template use-case: extract a repeated step sequence into a YAML file within the same repo, then reference it from your main pipeline. Parameters let callers customise it.

templates/install-node.yml — reusable step template
{% raw %}parameters:
  - name: nodeVersion
    type: string
    default: '20'

steps:
  - task: NodeTool@0
    inputs:
      versionSpec: '${{ parameters.nodeVersion }}'
    displayName: 'Set Node ${{ parameters.nodeVersion }}'
  - script: npm ci
    displayName: 'npm ci (clean install)'{% endraw %}
azure-pipelines.yml — calling the template
{% raw %}stages:
  - stage: Test
    jobs:
      - job: RunTests
        steps:
          - checkout: self
          # Call the template — pass nodeVersion as a parameter
          - template: templates/install-node.yml
            parameters:
              nodeVersion: '20'
          - script: npx playwright test{% endraw %}
External template repositories
cross-repo
Sharing templates across repositories

In-repo templates are only usable within one repository. To share a template across multiple projects or teams, store the templates in a dedicated repository and reference it from consumer pipelines. This is how platform or DevOps teams govern pipelines at scale.

  • Create a dedicated repo: e.g. pipeline-templates in your ADO organisation
  • Store templates there: templates/run-playwright.yml, templates/deploy-webapp.yml
  • Consumer pipelines reference the template repo using a resources.repositories block
  • ADO checks out the template repo at a specific ref (branch or tag) before parsing the caller's YAML
  • Template versioning: pin consumers to a tag like v2.1.0 to avoid unplanned updates
azure-pipelines.yml — consuming a template from another repo
{% raw %}resources:
  repositories:
    - repository: templates
      type: git
      name: 'MyOrg/pipeline-templates'
      ref: 'refs/tags/v2.1.0'

stages:
  - template: templates/run-playwright.yml@templates
    parameters:
      nodeVersion: '20'
      testFilter: '@smoke'{% endraw %}
governance
Templates as a governance mechanism

For QE and DevOps teams responsible for pipeline standards, external templates are a governance tool — not just a convenience. If every project calls your template, you control what every pipeline does, from one place.

  • Update Node version for all teams: change one line in the template repo
  • Add a mandatory security scan: add one step to the template — every caller gets it
  • Enforce consistent test reporting: the PublishTestResults step lives in the template, not in each project
  • Audit compliance: one file to review instead of dozens

Words to use in your next standup

You now have language for these situations:

  • "This pipeline uses a template from our shared template library — the actual steps are defined in the ado-pipeline-library repo, not in this file." — when explaining why a pipeline file looks short or incomplete
  • "The @templates at the end of the template path is the alias for the shared repository — you can find the real repo name in the resources.repositories block at the top." — when someone asks what @templates means
  • "The parameters block below the template call is what customises it for this specific project — change testFilter to change which tests run." — when your team wants to run a different test subset without touching the template itself

Key takeaways
  • Templates solve the copy-paste pipeline problem: one definition, called from many pipelines
  • Three template types: step (reuse steps), job (reuse a full job), stage (reuse a full stage)
  • Parameters make templates flexible — callers pass values, the template adapts
  • In-repo templates are local to one repo. External template repos share definitions across the entire organisation.
  • For QE and platform teams, templates are a governance mechanism — centrally controlled, consistently applied
