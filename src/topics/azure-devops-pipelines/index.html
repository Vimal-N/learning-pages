Azure DevOps Pipelines

From Why to How: CI/CD Pipelines for QA Teams

A 12-session series for QA engineers who want to understand what CI/CD pipelines really are, why they matter for test automation, and how to build production-quality Azure Pipelines that run, report, and scale.

12
Sessions
4
Phases
None
Prerequisites
Phase 1 Why Pipelines Exist Beginner Sessions 01 – 02
01
The Build That Worked on My Machine Beginner
A story-first session. Follow Jordan — a developer whose local tests were green, but a concurrent dependency bump caused a release failure the next morning. Understand why the conflict only surfaces in a clean shared environment, and what CI would have changed.
02
Every Merge Should Be a Safe Bet Beginner
What CI/CD actually promises — consistent, automated, trustworthy feedback on every change. Understand the feedback loop and why QE teams are specifically invested in running tests in pipelines.
Phase 2 The Landscape and ADO Introduction Beginner Intermediate Sessions 03 – 04
03
The CI/CD Tool Market in 20 Minutes Beginner
A rapid-fire survey of the major CI/CD tools — GitHub Actions, Jenkins, GitLab CI, and CircleCI. Understand the three tool categories, what mental model transfers across all of them, and why Azure DevOps is the right fit for teams on the Microsoft stack.
04
Azure DevOps: The Whole Suite, Then the One Thing We're Learning Intermediate
Azure DevOps is more than pipelines — it's a full delivery platform. Map the entire ADO product (Boards, Repos, Pipelines, Artifacts, Test Plans), then zoom into Pipelines specifically and understand the YAML-first approach.
Phase 3 Inside a Pipeline Intermediate Sessions 05 – 07
05
What Happens When a Pipeline Runs Intermediate
Open the hood on a pipeline execution. What is a build agent? What does it do? Understand steps, jobs, and stages — and the YAML hierarchy that ties them together.
06
Flow Control: Making Your Pipeline Smart Intermediate
Give your pipeline decision-making ability. Learn dependsOn for stage ordering, the four built-in condition functions, the continueOnError and always() pattern for reliable test publishing, and how to build a Test stage that blocks deployment on failure.
07
Reading a Build: Summary Page, Artifacts, and the Full Playwright Pipeline Intermediate
A guided tour of the ADO build summary page — Timeline, Tests, Artifacts, Logs tabs — plus a complete, annotated Playwright pipeline you can read and adapt line by line.
Phase 4 Patterns You'll See in Your Team's Pipelines Intermediate Advanced Sessions 08 – 12
08
Templates: Write Once, Run Everywhere Intermediate
Stop copy-pasting pipeline YAML between projects. Learn step, job, and stage templates — first in-repo, then from a shared template repository that any pipeline in your org can reference.
09
Reading and Reusing Your Organisation's Shared Template Library Intermediate
Your product team's pipeline already calls templates from a shared repository your organisation maintains. Learn how to find that repository in ADO, read its parameters block, and write your own QE-owned nightly pipeline that uses those templates — in under 22 lines of YAML.
10
Matrix Strategy: One Job, Many Configurations Intermediate
If you have opened your team's pipeline and seen a strategy: matrix: block, this session explains it completely. Understand how matrix runs one job across multiple browsers, environments, or OS configurations in parallel — and how to read the Tests tab when matrix is in play.
11
Stages as a Report: Aggregating Results Across Jobs Advanced
Run Playwright tests in parallel shards across multiple agents, then aggregate results into a single Tests tab. Learn how this pattern differs from matrix strategy, and when sharding is the right tool for cutting a long suite down to a fast PR check.
12
BrowserStack: Running Tests on Real Browsers in Your Pipeline Advanced
Connect Playwright to BrowserStack Automate and run tests on real browsers through Azure Pipelines. Covers the two-tier architecture, storing credentials in variable groups, reading the playwright.config.ts connection pattern, and how results appear in both ADO and the BrowserStack dashboard.
