Concept 1

What AI actually means

What people imagine
  • Robots with human-like consciousness
  • A digital brain that "thinks" like us
  • Skynet from the movies
  • Magic that just "knows" things
What it actually is
  • Software trained to recognise patterns
  • Very good at tasks humans find tedious
  • A tool — powerful but with real limits
  • Learned from human-generated data

Simple definition: A computer system designed to do tasks that would normally require human intelligence — like reading, writing, recognising images, or understanding language.

Concept 2

AI vs ML vs LLM

These three terms get used interchangeably. They shouldn't be.

AI
Artificial Intelligence
The broad umbrella. Any system doing tasks that normally need human intelligence. Includes AI from the 1950s onwards. Your spam filter is AI.
ML
Machine Learning
A type of AI where the system learns from data rather than following hand-written rules. Netflix recommendations are ML.
LLM
Large Language Model
A specific type of ML model trained on massive amounts of text. ChatGPT, Claude, Gemini — these are all LLMs. This is what we focus on.
Concept 3

How we got here — the four eras

Era 1
Rule-Based Chatbots
1960s – 2010s
Hand-written rules and pattern matching. ELIZA, IVR bots, FAQ bots. Type the wrong thing and the system breaks instantly.
Era 2
Voice Assistants & NLP
2011 – 2022
Siri, Alexa, Google Assistant. Natural phrasing worked better — but still domain-locked. Anything outside a known intent: "I can't help with that."
Era 3
Generative AI
2020 – present
GPT-3 → GitHub Copilot → ChatGPT (Nov 2022). Responses generated fresh. Handles novel questions. 100M users in 2 months.
Era 4
Agents
2024 – now
AI that doesn't just answer — it acts. Reads files, runs commands, writes code, checks its own work. Claude Code, Copilot Agent Mode.
Concept 4

November 2022

Why ChatGPT changed everything — and why it felt different

What existed before
  • AI tools required technical expertise
  • APIs, code, and configuration needed
  • Output was narrow and specialised
What ChatGPT did
  • You just... type in plain English
  • No setup, no code, no expertise needed
  • 100 million users in 2 months

The technology wasn't brand new. The accessibility was. That's what changed everything.

Concept 5

What is an LLM — and how does it generate a response?

1
Read almost everything humans have ever written
Books, websites, code, articles, forums — billions of documents. This is called training.
2
Learn patterns in language
Not memorising text — learning which words and ideas tend to appear together. Like learning grammar by reading, not by memorising rules.
3
Predict what should come next — one token at a time
When you send a message, the LLM uses those patterns to predict the most likely next word. Then the next. Then the next. That's a response.
Concept 6

What "training" means

→
Training data
The text an LLM learned from. Books, websites, code, Wikipedia, GitHub repos, forums. Hundreds of billions of words.
→
Knowledge cutoff
Training stops at a point in time. The model doesn't know about events after that date, unless you tell it.
→
Not a search engine
By default, LLMs don't browse the internet. They work from what they already learned. (Some tools add search on top — more on that later.)
→
Not your data
The model was not trained on your company's private files. It only learned from public text.
Era 4  ·  2024 – present

Agents — AI that takes action

→
Chatbot / Assistant
You ask → it answers. One exchange at a time. No persistent action.
→
Coding Assistant (Copilot)
You type → it suggests. Real-time help inside your editor. Still one suggestion at a time.
→
Agent (Claude Code, Copilot Agent Mode)
You describe a goal → it plans, uses tools, reads files, writes code, runs tests, iterates. Autonomous multi-step execution.

The agentic loop

Step 1 — Receive goal

User describes what they want done

Step 2 — Plan

LLM decides which steps to take

Step 3 — Act & observe

Runs a tool, reads output, adjusts

Step 4 — Repeat or finish

Loops until goal is achieved

Clearing the Air

Four myths — and what's actually true

Myth

"AI is sentient — it thinks and feels."

LLMs predict text. There are no thoughts or feelings. "I think..." is a trained writing pattern, not a view.

Myth

"If it sounds confident, it must be right."

LLMs hallucinate — inventing methods and logic errors with full confidence. Always verify output.

Myth

"It knows my project and conventions."

It only sees what you type in the current conversation. No codebase access unless you paste it in.

Myth

"AI will replace QA engineers."

AI automates tasks — the tedious, repetitive ones. Judgment and strategy remain yours.

Your role doesn't disappear — it shifts

Still your job
  • Deciding what to test and why
  • Identifying edge cases and risk areas
  • Verifying AI-generated code is correct
  • Understanding the system under test
  • Interpreting results with business context
AI handles more of
  • Writing the boilerplate test code
  • First-pass debugging of failures
  • Explaining unfamiliar code
  • Generating repetitive variations
  • Documentation and comments
Prompting Skills

What is a prompt?

A prompt is everything you send to an LLM — the question, the context you provide, examples, constraints, and the format you want back. The LLM reasons only from what's in the prompt. Nothing else.

The LLM knows
  • Everything you wrote in this prompt
  • Everything it learned during training
  • The conversation history (same session)
The LLM does not know
  • Your codebase or project
  • Your team's conventions or standards
  • Previous conversations (new session)

The four elements of a strong prompt

1
Context
What situation are you in? What tool, language, framework, or codebase is involved?
2
Task
What specifically do you want the AI to do? "Write a test" is not a task. "Write a Playwright test that verifies X" is.
3
Format
How do you want the response? Code only? With explanation? A numbered list? Short or detailed?
4
Constraints
What should it NOT do? "Don't use third-party libraries." "Keep it under 20 lines." "No async/await."

1 · Context — give it your situation

Missing context
"My test is failing. How do I fix it?"

The AI has to guess the framework, language, and error. Answer will be generic.

Rich context
"I'm using Playwright 1.40 with TypeScript. My test clicks #submit-btn on the login page but times out in CI. Passes locally on macOS. Here's the test and error..."

Framework, environment, symptom, code — targeted answer possible.

2 · Task — be specific about what you want

Too vague

"Write a test"

Specific

"Write a Playwright TypeScript test that navigates to /checkout, fills the address form, clicks Submit, and asserts the confirmation URL contains /order-confirmed"

3 · Format — tell it how to respond

Useful format instructions
  • "Give me only the code, no explanation"
  • "Explain in simple terms, no jargon"
  • "List the options as bullet points"
  • "Give me a one-sentence answer"
Choose based on your goal
  • Just need the code? Ask for code only
  • Learning something? Ask for explanation + code
  • Need options? Ask for a comparison list

4 · Constraints — tell it what NOT to do

Without: "Write a test for the API response" → AI picks whatever library it prefers, possibly one you don't use

With: "Write a test using Jest and supertest only — no other libraries. Keep it under 25 lines."

A complete prompt — all four elements

I'm writing Playwright TypeScript tests for an e-commerce checkout flow. Write a test that adds a product to the cart, goes through checkout, and asserts the order confirmation number is visible on the success page. Return only the test code with inline comments explaining each step. Do not use any helpers or utilities not already in Playwright — no third-party packages.
Context

Playwright TypeScript, e-commerce checkout

Task

Cart → checkout → assert confirmation

Format

Code only, inline comments per step

Constraints

Playwright built-ins only

Three QA Patterns to Use Today

Prompting templates for your daily work

Pattern 01
Writing Tests
I'm writing Playwright TypeScript tests.
Page: [URL or component name] | User action: [what the user does]
Expected result: [what should happen]
Write a test. Use getByRole/getByLabel for selectors. Return code only.
Pattern 02
Debugging a Failing Test
My Playwright test is failing with this error:
[paste full error message]
Here is the failing test: [paste test code]
The test passes locally but fails in CI (GitHub Actions).
What are the most likely causes? List them ranked by probability.
Pattern 03
Reviewing Your Own Tests
Review this Playwright test and flag any issues.
Look for: brittle selectors, missing assertions, missing edge cases,
anti-patterns, and anything that would make this test flaky in CI.
[paste your test code]
Return a numbered list of issues with severity. Also suggest any missing test cases.
Key Takeaways

What to leave with today

1
AI is a broad category. LLMs (ChatGPT, Claude, Gemini) are the specific type we care about — trained on text, generating text, predicting one token at a time.
2
We went from rule-based chatbots → voice assistants → generative AI → agents. We're now in the agent era — AI that takes action, not just answers.
3
LLMs don't think or understand — they predict patterns. They can be confidently wrong. Always verify output before using it in real work.
4
Your QA expertise is more valuable, not less. AI handles boilerplate — you still decide what to test, identify edge cases, and verify results.
5
A good prompt has four elements: Context, Task, Format, Constraints. Specificity is the single biggest lever — vague in, vague out.
6
Three QA patterns to try today: writing tests, debugging failures, reviewing your own test code before submitting a PR.
