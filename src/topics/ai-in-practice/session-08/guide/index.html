Debugging Tests with AI — Stop Guessing, Start Asking

How to use AI to diagnose test failures faster — by feeding it the right information and asking the right questions.

Why AI Debugging Works

AI has seen thousands of Playwright failures. It pattern-matches faster than you can Google.

When a test fails, the error message points at a symptom — not the cause. Finding the cause requires knowing which common failure patterns to check. AI has been trained on enormous amounts of test code, error messages, and debugging discussions. It recognises patterns you'd have to spend 20 minutes searching for.

Old approach (guessing)
  • Read error. Guess it's a timing issue. Add waitForTimeout. Still fails.
  • Guess it's a selector problem. Change selector. Still fails.
  • Google the error message. Find a 2-year-old Stack Overflow thread.
  • Try the top answer. Slightly different problem. 20 minutes lost.
AI-assisted debugging
  • Paste error + test code + context.
  • Ask AI to categorise the failure type and suggest the most likely fix.
  • AI explains the likely root cause and gives a targeted fix.
  • Try it. If it doesn't work, feed AI the new error — it iterates.

Limitation to know: AI can't run your app, inspect your DOM, or watch your test execute. It works from what you paste. The better your context, the better the diagnosis. A vague "my test is failing" gets a vague answer.

The Debugging Prompt Structure

Four pieces of information. All four are required. Missing any one of them makes the diagnosis less reliable.

1
The error message — verbatim
Paste the full error output from your terminal or CI log. Don't paraphrase it. The exact wording, stack trace lines, and error type are all signals AI uses. "It says something about a timeout" is much less useful than the actual message.
2
The test code
Paste the complete failing test. Not just the line it fails on — the whole test() block. AI needs to see how the test is structured, what actions precede the failure, and what assertion is being made.
3
What the test is supposed to do
One sentence: "This test should fill the login form, submit it, and verify the user lands on /dashboard." AI needs to know the intended behaviour to tell you whether the test logic is correct or just the implementation.
4
What you already tried
If you've already ruled something out, say so. "I checked and the selector is correct" or "this test was passing yesterday and nothing in the test changed" — these rule out whole categories of failure and point AI toward less obvious causes.

Full debugging prompt template

This Playwright test is failing. Please diagnose it.

ERROR:
[paste full error output here]

TEST CODE:
[paste full test() block here]

WHAT IT SHOULD DO:
[one sentence describing expected behaviour]

WHAT I'VE ALREADY TRIED:
[list anything you've ruled out, or "nothing yet"]

Tell me: (1) what type of failure this is, (2) the most likely root cause, (3) the fix to try first.

Four Failure Categories AI Spots Fast

Most Playwright failures fall into one of four categories. Knowing them helps you read AI's diagnosis and verify it's correct.

TYPE 1
Selector Failure
Most common
Symptom: TimeoutError: Locator … not found or strict mode violation. Cause: selector is wrong, changed, or matches multiple elements. Fix: inspect the live DOM, update the selector, use data-testid attributes where possible.
TYPE 2
Timing Failure
Very common
Symptom: Test passes locally, fails in CI — or fails intermittently. Cause: test acts before the element is ready or a navigation hasn't resolved. Fix: replace waitForTimeout with expect(locator).toBeVisible() or page.waitForURL().
TYPE 3
State Contamination
Common in suites
Symptom: Test passes alone, fails when run with others. Cause: test depends on state left by a previous test (logged-in session, cart contents, database record). Fix: make the test self-contained — set up and tear down its own state.
TYPE 4
Wrong Assertion
Trickiest to spot
Symptom: Test fails with an assertion error but the feature looks correct in the browser. Cause: assertion checks the wrong thing (wrong text, wrong URL format, wrong element). Fix: clarify what "success" actually looks like and update the assertion to match reality.

Ask AI to classify before it fixes: Add "Tell me which of these four categories this is before suggesting a fix" to your debugging prompt. If AI's classification matches your intuition, the fix will likely be correct. If it doesn't match, investigate further before applying it.

Diagnosing Flaky Tests

Flaky tests are the hardest category. AI can help you find timing assumptions you didn't know you made.

A flaky test is a test that sometimes passes and sometimes fails with no code change. The root cause is almost always an assumption about timing or state that holds in fast environments and breaks in slow ones (CI, parallel runs, slower machines).

What to give AI for flaky test diagnosis

This test passes locally about 80% of the time and fails in CI about 30% of the time.
It always fails at the same assertion.

TEST CODE:
[paste test]

ERROR (from a failing run):
[paste error output]

It was passing reliably until [describe any recent change — new feature, dependency update, parallel test addition].

Diagnose what timing assumption this test is making and how to make it deterministic.

Common AI-identified timing fixes

Test acts on element before it's visible
Add expect(locator).toBeVisible() before the action
Test asserts URL before navigation completes
Use await page.waitForURL('/target') then assert
API call hasn't resolved before assertion
Wait for a visible UI response to the API (loading indicator gone, data rendered)
Intermittent on CI only — different base URL or env
Check playwright.config.ts baseURL and environment variable handling

When Not to Use AI for Debugging

AI is fast at pattern-matching known failure types. It's slow and unreliable for failures that require live investigation.

AI debugs well
  • Error message is clear and specific
  • Failure is consistent and reproducible
  • Test code is the likely problem
  • You can describe what the feature is supposed to do
  • Failure matches a known category (selector, timing, assertion, state)
Investigate manually instead
  • Error is in a 3rd-party library with no clear message
  • Failure only happens with specific data in your real database
  • Test is correct but the feature itself is broken
  • Failure requires watching the browser to understand what's happening
  • Environment issue (network, proxy, auth token) unrelated to test logic

The 10-minute rule

Spend no more than 10 minutes on AI debugging before switching to manual investigation. If three AI-suggested fixes don't work, the failure is probably in a category AI can't diagnose from text alone. Open Playwright's --debug mode or use the trace viewer and watch what actually happens.

Playwright Trace Viewer: Run npx playwright test --trace on and then npx playwright show-trace trace.zip. You get a timeline of every action, screenshot at every step, and network request log. Once you can see what happened, feed the key observation to AI: "The test clicked submit but the form didn't send a network request — here's the error…"

Iterating — When the First Fix Doesn't Work

AI debugging is rarely one shot. Feed it the new error and it continues the investigation.

When an AI-suggested fix doesn't resolve the failure, don't start a new chat. Continue the conversation — AI has context about what it already tried. Give it the new error and it will rule out the previous theory and suggest the next most likely cause.

Debugging Conversation — Playwright Selector Failure
You
[Paste failing test + error: TimeoutError: Locator '#submit-btn' not found]
AI
This is a Type 1 selector failure. The selector #submit-btn isn't matching any element. Most likely cause: the button uses a different ID or has a data-testid instead. Try updating the selector to button[type="submit"] as a starting point.
You
Tried that — still failing. New error: TimeoutError: Locator 'button[type="submit"]' not found after 30000ms.
AI
Ruling out selector mismatch — the button exists but Playwright isn't finding it in the expected state. This suggests a timing issue: the page may not have fully rendered before the locator is checked. Add await page.waitForLoadState('networkidle') before the click, or check if the form is inside an iframe. Also verify your baseURL in config — if the test is navigating to the wrong page, no selectors will match.

Try This — Debug a Real Failure

Apply the debugging prompt to a test that's currently failing or flaky in your project.

Try This — 15 minutes

Find a failing or flaky test in your project — or deliberately break a passing test by changing a selector to something wrong. Then:

Part 1 (5 min): Assemble the four pieces of the debugging prompt: the full error output, the complete test code, one sentence describing what the test should do, and anything you've already tried. Don't try to diagnose it yourself first — let AI go first.

Part 2 (5 min): Send the prompt to Copilot Chat or Claude. Check: does AI correctly classify which of the four failure types this is? Does its root cause theory match your intuition?

Part 3 (5 min): Apply the fix. If it works, note what type of failure it was. If it doesn't, continue the conversation with the new error. Track how many iterations it took to resolve.

After 3 sessions of this, you'll have a clear picture of which failure types AI is reliable for and which require manual investigation in your specific project.
