What Makes Agent Mode Different from Chat

Chat is reactive. Agent Mode is goal-directed. Here's what that means in practice.

Copilot Chat (standard mode) is reactive: you send a message, it responds, you copy code and apply it manually. You orchestrate every step.

Agent Mode changes the operational model. You give it a goal — a description of the desired end state — and it works through the steps to get there. It can:

  • Read your project files to understand existing patterns before generating anything
  • Create new files and edit existing ones
  • Run terminal commands (like npx playwright test) and read the output
  • See if a test fails, diagnose the failure, and correct it — without you prompting each step

The key distinction: in Chat, you steer every turn. In Agent Mode, you describe the destination and supervise the journey.

Copilot Chat (normal mode)
  • You ask a question or give a task
  • Copilot generates a response
  • You copy, paste, apply manually
  • You ask the next step
  • Complete manual control — complete manual effort
Agent Mode
  • You give a goal, not a sequence of steps
  • Agent reads your project to understand patterns
  • Edits multiple files, runs tests, reads errors
  • Self-corrects without you prompting each step
  • Shows diffs for approval — you stay in control

How to Enable Agent Mode

Four steps to switch from Chat to Agent — and what to check if you can't see the option.

  1. Open Copilot Chat: click the chat icon in the VS Code sidebar or press Ctrl+Alt+I / Cmd+Shift+I
  2. Look for the mode dropdown at the top of the chat panel (it may say "Chat" or "Ask")
  3. Select Agent from the dropdown
  4. Type your goal (not a question — a desired outcome)

Requires a paid plan. Agent Mode is not available on the free Copilot tier as of 2025. If you don't see "Agent" in the dropdown, check your plan at github.com → Settings → Copilot. Business and Enterprise plans both include it.

Writing Goals That Work

Every effective Agent Mode goal has four components: Goal, Context, Constraints, Verification.

The most common reason Agent Mode produces disappointing results is an unclear goal. The agent needs to understand four things:

GOAL
What you want the end result to be — in terms of the project state, not the steps to take. "Add E2E tests for the payment flow" not "create a file called payment.spec.ts with three tests."
CONTEXT
What the agent needs to know — relevant files, frameworks, patterns. "Use the pattern from tests/checkout/cart.spec.ts. Stripe test cards are in tests/fixtures/stripe.ts."
CONSTRAINTS
What the agent must not do — files to leave alone, scope limits, patterns to follow. Without this, the agent may helpfully "fix" files you didn't ask it to touch.
VERIFICATION
How to confirm the task is done. "Run npx playwright test tests/checkout/payment.spec.ts and confirm all pass." This tells the agent how to close the loop.

Poor goal: "Add some tests for the checkout flow"

Why it fails: No scope (which part of checkout?), no pattern (which file to follow?), no constraints (can it touch existing files?), no verification (what counts as done?). The agent will make all these decisions itself — and they may not match what you wanted.

Constraint Vocabulary Guide

Five phrases to add to every goal prompt — learn these and use them every time.

These five constraint phrases are the most useful in Agent Mode. Learn them. Use them in every goal prompt.

"Only modify files in tests/checkout/"
Prevents the agent from touching files outside the specified directory. Without this, a helpful agent may "improve" files that were working fine. Be as specific as the task requires — a single directory, or even a single file.
"Do not create new directories"
Agents sometimes decide your project needs a new folder structure. This constraint keeps everything in existing directories, preventing a rearrangement you didn't ask for.
"Do not run commands that modify the database"
If your test setup involves database operations, this prevents the agent from running migrations, seeding scripts, or delete commands. Especially important on shared development environments.
"Stop after creating files — I will run tests manually"
Prevents the agent from running any terminal commands. Use this when you want to review the generated files before any test execution — especially useful when tests touch a staging environment.
"Match the existing Page Object pattern in tests/pages/"
Tells the agent to learn from and replicate your existing Page Object structure rather than inventing its own. Combine with #file:tests/pages/loginPage.ts to make the reference explicit.

The Approval Workflow — Why It's Not a Formality

Four phases of approval — plan, diff, command, verification. Here's what to look at in each.

Agent Mode shows you every file change as a diff before applying it. This is not a formality — it's the control mechanism that makes agent use safe. The sequence:

  1. Plan phase: Agent describes what it intends to do. Read this. If the plan is reading the wrong files or proposes touching files you didn't expect, stop here and add constraints.
  2. Diff phase: Each file change appears as a coloured diff (green = added, red = removed). This is where you catch selector problems, wrong assertion patterns, and scope creep.
  3. Command phase: If the agent runs a terminal command, it shows you the command before executing. You can reject it.
  4. Verification phase: After changes, the agent may run tests and show you the output. A passing test is not proof the test is correct — check the assertions too.

What to actually look at in each diff:

  • Scope: Is it only touching the files you expected? If you see changes to files outside your constraint, reject and redirect.
  • Selectors: Are they from your Page Object, or did the agent invent raw selectors? Invented selectors look plausible but often don't exist in your DOM.
  • Assertions: Does the test actually verify the behaviour, or just that an element is visible?
  • Imports: Are all imports valid for your project's installed dependencies?

Recovery Playbook — When Agent Mode Goes Wrong

Five scenarios and exactly what to do in each one.

Things will go wrong. Here's what to do for each situation.

The agent is going in the wrong direction mid-task

You don't have to reject everything and start over. You can redirect in the approval step by typing a correction instead of accepting:

Don't use page.goto() here — use the navigateToPayment() method
from the checkoutPage Page Object instead. It handles the cart step first.

The agent adjusts its approach and regenerates. This is faster than starting over with a new goal prompt.

You accepted a diff you shouldn't have

Run git diff to see exactly what changed. Use git restore <filename> to revert a specific file, or git restore . to revert everything since your last commit. Always work on a feature branch when using Agent Mode for this reason — reverting to a known-good commit is straightforward.

The agent ran a command you didn't want

If the command modified files, git restore . covers the file changes. If it modified a database or sent a request to an API, you'll need to undo that manually. This is why the constraint "Do not run commands that modify the database" exists — use it proactively, not reactively.

The agent stopped partway through

Click Stop in the Copilot Chat panel to halt the agent. Any diffs you already accepted remain applied — check git diff to see the state. You can continue with a new goal from where it stopped, or revert and restart with a tighter scope.

The agent keeps looping on the same failure

This happens when a test fails in a way the agent can't diagnose without more context (a selector that only exists in a running app, a race condition, an environment variable that's missing in CI). Redirect: "Stop running tests. The test is failing because of [reason]. Update the assertion to [specific change] and do not run tests again."

Decision Framework + Common Mistakes

Which tool for which task — and five mistakes to avoid from day one.

Use this reference any time you're wondering whether to use Agent Mode or a simpler approach. The goal is to match the tool's power to the task's complexity.

When you want to...Use...
Complete one test or finish a half-written function in the file you're editing
Completions (Tab)
Ask a question, explain code, generate a single test, debug with context you provide
Copilot Chat
Make an edit at the cursor without switching to the chat panel
Inline Chat (Ctrl+I)
Build a complete test file, refactor a test suite, create multiple related files from a goal
Agent Mode
Work across multiple directories, run git commands, long multi-session tasks, work that needs shell access

Common Mistakes Catalogue

Vague goal → unpredictable scope
"Improve our test coverage" has no boundary. The agent may start touching files across your entire project. Always specify the feature, directory, or files you mean.
Clicking through diffs without reading them
The agent is fast and confident. It's easy to click Accept on each diff assuming they're correct. A 30-second diff review is the difference between a high-quality session and a debugging session tomorrow.
Trusting "tests pass" as proof of correctness
Agent Mode can run tests and confirm they pass. But a test that always passes regardless of application state is a false positive. After any agent session, manually review the assertions — "does this test fail if the feature breaks in the way I care about?"
Using Agent Mode for small tasks
Agent Mode has overhead: it reads files, plans steps, asks for approvals at each stage. For a one-line fix, it's slower than Chat. Reserve it for tasks that genuinely span multiple files or require multiple sequential steps.
No constraints → scope creep
Without explicit constraints, the agent may helpfully "fix" files you didn't ask it to touch. It sees your whole project. Add at minimum: which directory to work in and which existing files not to modify.
