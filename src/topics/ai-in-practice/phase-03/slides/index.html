Session 07

The Three-Step AI Test Workflow

The most common mistake: going straight from a feature description to "write me some tests." That skips the decision about what to test — which is your job, not AI's.

1
AC → Test Cases
Use AI to turn your acceptance criteria into a structured list of scenarios — happy path, errors, boundaries, edge cases. You review and trim this list. This decision is still yours.
2
Test Cases → Code
Feed AI your approved list plus context about your project — selectors, existing test style, Page Object. It writes the Playwright code. You review before committing anything.
3
Verify, Run, and Fix
Run the generated tests. Check the four most common AI mistakes — selectors, assertions, test isolation, waiting patterns. Fix before the tests go into your suite.

Mindset shift: AI writes the code. You decide what to test. Never hand over coverage decisions to the model — it doesn't know your product, your risk areas, or your users.

Session 07 · Step 1

AC → Test Cases Prompt

Paste your acceptance criteria. Ask AI to enumerate what to test. Review the list — then write a single line of code.

Try this live → Open Copilot Chat. Take an AC from your current sprint. Paste the prompt below with your AC in the placeholder.

Act as a senior QA engineer.

Here is the acceptance criteria for a checkout feature:
[PASTE YOUR AC HERE]

Generate a complete list of test cases covering:
1. Happy path (all valid inputs, expected success)
2. Error states (invalid input, payment failure, network error)
3. Boundary values (minimum/maximum quantities, character limits)
4. Edge cases (empty cart, session expiry mid-checkout, duplicate submission)

Format each test case as: Given [state] / When [action] / Then [expected result].
Flag any ambiguities in the AC that I should clarify before testing.
What AI does well here
  • Finds error states and boundary values you missed
  • Flags AC ambiguities — questions for your dev or PO
  • Produces a reviewable list before any code is written
Your job after
  • Remove tests that aren't worth automating
  • Add anything AI missed from your product knowledge
  • The approved list becomes the brief for code generation
Session 07 · Step 2

Test Cases → Code Context Cards

Give AI your approved test list plus these four pieces of context. Missing any one of them produces code you'll have to rewrite.

Selectors
Required
List the actual element IDs, data-testid values, or CSS selectors for your feature. AI can't inspect your app — without these, it invents selectors and every test fails immediately.
Existing test example
Strongly recommended
Paste one complete test from your project and say "match this style." AI replicates your naming conventions, import structure, POM patterns, and assertion style — without being told each rule explicitly.
Page Object reference
If applicable
If your project uses Page Objects, paste the relevant PO class (or use #file: in Copilot Chat) so AI uses your existing methods rather than duplicating selector logic.
Expected URLs & error messages
Required for assertions
Tell AI the exact URL a success redirects to, the exact text of error messages, the exact element that confirms success. Wrong values create assertions that pass when they shouldn't.
Session 07 · Technique

The Style-Matching Technique

The single most effective thing you can do to get usable AI-generated tests: show it what you already have. One pasted example replaces ten abstract rules.

Without a style example
  • AI picks an import style at random
  • May invent a beforeEach your project doesn't use
  • Names describes inconsistently with your suite
  • Uses page.locator() when you prefer page.getByLabel()
  • Requires rewriting before it fits the project
With one pasted test
  • Replicates your exact import structure
  • Uses the same method style you already use
  • Follows your naming conventions automatically
  • Matches your POM usage patterns
  • Ready to drop in with minor edits

Try this live → In Copilot Chat, type the prompt below — don't paste the test file, reference it directly with #file:.

#file:tests/auth/login.spec.ts
Write new tests for the checkout flow. Match the exact style of the tests in this file.

[paste your approved test case list here]
Session 07 · Step 3

Reviewing What AI Wrote — 4-Point Checklist

AI makes predictable mistakes. Knowing them in advance makes review fast — you're looking for a known set of issues, not reading every line cold.

1
Selectors — are they real?
Check every selector against your actual DOM. AI commonly invents plausible-sounding IDs like #submit-btn when your button is actually button[data-testid="place-order"]. If you didn't provide selectors, assume all of them need checking.
2
Assertions — do they actually test something?
Check that each assertion would fail if the feature broke. expect(page).toHaveURL('/dashboard') is real. A status-code check on a page navigation is not. Weak assertions give false confidence — worse than no assertion.
3
Test isolation — can it run independently?
Each test should work in any order with no prior state. If a test assumes items are already in a cart from a previous test, it will fail intermittently. AI often generates tests that share state unless you explicitly said not to.
4
Waiting patterns — no arbitrary timeouts
Flag any page.waitForTimeout(2000) calls. AI reaches for these when it doesn't know the right condition to wait for. Replace with expect(locator).toBeVisible() or page.waitForURL().

Always run the tests before committing. A test that passes without doing anything isn't protecting you. Run the full generated spec, check each test fails for the right reason first.

Session 08

Why AI Debugging Works

AI has seen thousands of Playwright failures. It pattern-matches faster than you can Google. But it only works from what you paste — vague input gets a vague answer.

Old approach (guessing)
  • Read error. Guess it's timing. Add waitForTimeout. Still fails.
  • Guess it's a selector. Change it. Still fails.
  • Google the error message. Find a 2-year-old Stack Overflow thread.
  • Try the top answer. Slightly different problem. 20 minutes gone.
AI-assisted debugging
  • Paste error + test code + context.
  • Ask AI to categorise the failure type and suggest the first fix.
  • AI explains the likely root cause with a targeted fix.
  • If it doesn't work — feed AI the new error and continue.

Limitation to know: AI can't run your app, inspect your DOM, or watch your test execute. It works from what you paste. The better your context, the better the diagnosis.

Session 08 · Prompt

The Debugging Prompt Template

Four pieces of information. All four are required. Missing any one makes the diagnosis less reliable.

1
Full error output
Paste verbatim from your terminal or CI log. Don't paraphrase — the exact wording, stack trace, and error type are all signals AI uses.
2
Complete test code
Paste the full test() block — not just the failing line. AI needs to see how the test is structured and what actions precede the failure.
3
Intended behaviour
One sentence: "This test should fill the login form and verify the user lands on /dashboard." AI needs the intended behaviour to separate test logic errors from implementation errors.
4
What you've already tried
Rules out whole failure categories. "The selector is correct" or "this was passing yesterday with no test changes" point AI toward less obvious causes.

Try this live → Take a failing test from your project. Assemble the four pieces. Paste the template below into Copilot Chat.

This Playwright test is failing. Please diagnose it.

ERROR:
[paste full error output here]

TEST CODE:
[paste full test() block here]

WHAT IT SHOULD DO:
[one sentence describing expected behaviour]

WHAT I'VE ALREADY TRIED:
[list anything you've ruled out, or "nothing yet"]

Tell me: (1) what type of failure this is, (2) the most likely root cause, (3) the fix to try first.
Session 08 · Failure Types

Four Failure Categories AI Spots Fast

Most Playwright failures fall into one of four categories. Know them so you can verify AI's diagnosis — and catch it when it's wrong.

TYPE 1
Selector Failure
Most common
Symptom: TimeoutError: Locator … not found or strict mode violation. Cause: selector is wrong, changed, or matches multiple elements. Fix: inspect the live DOM, update the selector, prefer data-testid attributes.
TYPE 2
Timing Failure
Very common
Symptom: Passes locally, fails in CI — or fails intermittently. Cause: test acts before element is ready or navigation hasn't resolved. Fix: replace waitForTimeout with expect(locator).toBeVisible() or page.waitForURL().
TYPE 3
State Contamination
Common in suites
Symptom: Passes alone, fails when run with other tests. Cause: test depends on state left by a previous test (logged-in session, cart contents, database record). Fix: make the test self-contained — set up and tear down its own state.
TYPE 4
Wrong Assertion
Trickiest to spot
Symptom: Assertion fails but the feature looks correct in the browser. Cause: assertion checks the wrong thing — wrong text, wrong URL format, wrong element. Fix: clarify what "success" actually looks like and update the assertion to match reality.

Ask AI to classify first: Add "Tell me which of these four categories this is before suggesting a fix" to your debugging prompt. If AI's classification matches your intuition, the fix will likely be correct.

Session 08 · Flaky Tests

Diagnosing Flaky Tests

A flaky test sometimes passes and sometimes fails with no code change. The root cause is almost always a timing assumption that holds locally and breaks in CI or parallel runs.

Try this live → Find a test that's been failing intermittently. Use the flaky test prompt below in Copilot Chat.

This test passes locally about 80% of the time and fails in CI about 30% of the time.
It always fails at the same assertion.

TEST CODE:
[paste test]

ERROR (from a failing run):
[paste error output]

It was passing reliably until [describe any recent change — new feature, dependency update, parallel test addition].

Diagnose what timing assumption this test is making and how to make it deterministic.

Common AI-identified timing fixes

Test acts on element before it's visible
Add expect(locator).toBeVisible() before the action
Test asserts URL before navigation completes
Use await page.waitForURL('/target') then assert
API call hasn't resolved before assertion
Wait for a visible UI response to the API (loading indicator gone, data rendered)
Intermittent on CI only — different base URL or env
Check playwright.config.ts baseURL and environment variable handling
Session 08 · Limits

When Not to Use AI for Debugging

AI is fast at pattern-matching known failure types. It's unreliable for failures that require live investigation — watching the browser, inspecting real data, or reading 3rd-party library internals.

AI debugs well
  • Error message is clear and specific
  • Failure is consistent and reproducible
  • Test code is the likely problem
  • Failure matches a known category (selector, timing, assertion, state)
  • You can describe what the feature is supposed to do
Investigate manually instead
  • Error is in a 3rd-party library with no clear message
  • Failure only happens with specific data in your real database
  • Test is correct but the feature is broken
  • Failure requires watching the browser to understand what's happening
  • Environment issue (network, proxy, auth token) unrelated to test logic

The 10-minute rule: If three AI-suggested fixes don't work, switch to manual investigation. Open Playwright's debug mode or use the trace viewer — npx playwright test --trace on then npx playwright show-trace trace.zip. Once you can see what happened, feed that specific observation back to AI.

Session 09

AI as Test Reviewer

When you ask AI to review a test, it checks for known patterns of weakness. It finds structural problems fast — but it doesn't know whether your tests cover the right business scenarios. That judgement is still yours.

CHECK 1
Fragile selectors
Positional selectors (nth-child), text-content selectors on labels that change, XPath that traverses multiple levels. AI flags these and suggests data-testid alternatives.
CHECK 2
Weak or missing assertions
Tests that check the wrong thing, tests with no meaningful assertion, tests where the assertion could pass even if the feature is broken. AI identifies: "this assertion doesn't actually prove what you think."
CHECK 3
Anti-patterns
page.waitForTimeout(), hardcoded sleeps, duplicate selector definitions, missing await on async operations, page.evaluate() for things Playwright can do natively.
CHECK 4
State leakage risk
Tests that create database records, send emails, or modify shared state without cleanup. AI flags these as risks for contaminating other tests, especially in parallel runs.
Review this Playwright test for quality issues:
[paste test]

Check for: fragile selectors, weak or missing assertions, tests that could pass even when the feature is broken, waitForTimeout usage, state leakage, and any Playwright anti-patterns.

List each issue with: severity (high / medium / low), what the problem is, and the fix.
Session 09 · POM

Extracting Page Objects

When the same selectors appear in multiple tests, AI can extract them into a Page Object faster than doing it by hand. One DOM change then only needs to be fixed in one place.

When to extract

Same selector appears in 3+ test files
Extract to a Page Object now
Same sequence of actions repeated across tests
Extract action into a Page Object method
Only 1–2 tests use a page
Keep inline — Page Object adds overhead, not value yet
Selector changes regularly (fast-moving UI)
Prioritise extraction — maintenance cost is high without it

Try this live → Paste 2–3 test files that hit the same page. Use the prompt below.

Extract a Page Object from these test files:
[paste 2–3 test files or relevant test blocks]

Rules:
- Create one class per page (e.g. LoginPage, CheckoutPage)
- Move all selectors into the class as private readonly locators
- Create methods for each distinct user action
- Do NOT put assertions in the Page Object — keep those in test files
- Export each class as default
- Return the Page Object class and the refactored test code separately

Here is an existing Page Object from my project for style reference:
[paste existing PO, or say "no existing PO — create in standard Playwright POM style"]
Session 09 · Coverage

Finding Coverage Gaps

AI can compare your existing tests against your acceptance criteria and tell you what you haven't tested. These are exactly the failure modes that reach production.

Try this live → Take a feature you think is well-covered. Run the prompt below against it. You'll almost always find at least one P1 gap you didn't know about.

Here is a list of my existing test cases for the [feature name] feature:
[paste test names or test.describe / test() structure]

Here is the acceptance criteria for this feature:
[paste AC]

What test scenarios are missing? Focus on:
- Error paths (what happens when things go wrong)
- Boundary values (min/max, empty, maximum length)
- Concurrent actions (what if two users do this at the same time)
- Session and authentication edge cases
- Scenarios where the feature could silently fail (wrong data saved, no error shown)

Return a prioritised list: P1 (likely to find bugs), P2 (worth having), P3 (low value).

Provide full test code, not just test names. If a test is named "user can submit order" but the assertion only checks that a button was clicked, AI won't flag the gap — it sees the test name as coverage. Full code gives full picture.

Session 09 · Refactoring

Reducing Test Duplication

Three patterns AI can refactor for you — and one category it's bad at.

→
Parametrize repeated tests
10 nearly-identical tests that differ only in input data → one data-driven test. Prompt: "Convert these to a parametrized test that loops over a test data array."
→
Extract shared setup into fixtures
Multiple test files needing the same setup (logged-in user, seeded database record) → Playwright fixture. Paste your repeated beforeEach blocks and ask for a fixture extraction.
→
Consolidate selector definitions
Same selectors defined across multiple test files → AI identifies the duplication and shows which should move to a shared Page Object or constants file.
AI refactors well
  • Structural refactors with clear rules (PO extraction, parametrize, fixtures)
  • Renaming for consistency (matching your naming convention)
  • Adding missing assertions to existing test structure
AI refactors poorly
  • Deciding which tests to delete (requires product knowledge)
  • Reorganising test file structure (needs knowledge of your CI/reporting setup)
  • Any refactor that requires knowing your app's domain logic
Watch Out

Common Mistakes — And How to Avoid Them

Skipping step 1
Going straight from feature description to "write me some tests." This hands coverage decisions to AI. Instead: always generate test cases first and review the list before generating code.
Omitting selectors in generation prompts
If you don't provide real selectors, AI invents plausible-sounding ones. Every test will fail immediately. Selectors are required context — always paste them or reference your POM.
Committing AI tests without running them
A test that passes without doing anything isn't protecting you. Always run the full generated spec and confirm each test fails for the right reason before adding to CI.
Spending more than 10 minutes on AI debugging
Three fixes that haven't worked means you're in a failure category AI can't diagnose from text. Switch to Playwright's trace viewer — watch what actually happens, then feed that specific observation back to AI.
Asking AI what to test
AI doesn't know your product, your risk areas, or your users. You decide what to test. AI writes the code and generates the case list. Keep those roles separate.
Key Takeaways

Phase 3 Takeaways

1
Generate test cases before code
Always run step 1 — AC to test cases — and review the list. You decide what to test. AI writes the code.
2
Context is everything — for both generation and debugging
Paste your real selectors. Paste one existing test for style. Paste the full error. Paste the full test code. More context = better output every time.
3
Know the four failure categories
Selector · Timing · State Contamination · Wrong Assertion. They cover the vast majority of Playwright failures. Recognising them makes AI's diagnosis verifiable.
4
Use AI to review your existing tests
AI is a fast, available-any-time reviewer for structural quality: fragile selectors, weak assertions, anti-patterns, state leakage. Run it on any test file you haven't looked at in a while.
5
10 minutes on AI, then switch to manual
AI is fast for known failure types. For anything requiring live investigation — use the trace viewer, watch the browser, then feed your specific observation back to AI.
