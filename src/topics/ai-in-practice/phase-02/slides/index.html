Concept 1

What is GitHub Copilot?

An AI coding assistant built into your editor. It watches what you type and suggests what comes next — and it can answer questions about your code. It was trained on billions of lines of public GitHub code, which makes it far more useful for coding tasks than a general-purpose chatbot.

Built by
  • GitHub (owned by Microsoft)
  • Powered by OpenAI models
  • Available in VS Code, JetBrains, Neovim
Trained on
  • Billions of lines of public GitHub code
  • Knows Playwright, Jest, Cypress patterns
  • Understands TypeScript, Python, Java and more
Concept 2

Two modes — know both

Inline Completions
  • Passive — Copilot watches and suggests
  • Ghost text appears as you type
  • Tab to accept  ·  Escape to dismiss
  • Best for: repetitive patterns, boilerplate
// Test: user submits the login form
test('user submits the login form', async ({ page }) => {
  await page.goto('/login');
  await page.getByLabel('Email').fill('user@example.com');
});
Copilot Chat
  • Active — you ask, Copilot responds
  • Opens in a sidebar panel
  • Has access to your open file and selection
  • Best for: debugging, explaining, refactoring

Slash commands: /explain   /fix   /tests   /doc

Setup

Get Copilot running in 3 steps

1
Install the extensions
VS Code Extensions panel → search GitHub Copilot. Install both: GitHub Copilot (completions) and GitHub Copilot Chat. JetBrains users: same plugin available in JetBrains Marketplace.
2
Sign in to GitHub
Click the Copilot icon in the status bar → sign in via browser OAuth → click Authorize. Sign into the account that has your Copilot licence (Individual, Business, or Enterprise).
3
Your first completion
Open a .ts test file. Write a comment: // Test: user logs in successfully. Pause — watch the grey suggestion. Press Tab to accept. Copilot is active.

Plans: Free tier (limited), Individual ~$10/month, Business ~$19/seat. Check with your manager before signing up individually.

Concept 3

How Copilot reads your code

Reads surrounding context
Lines above and around your cursor — your imports, variable names, existing functions, and comments. Not your whole project.
Matches your patterns
If you use getByRole for selectors in your existing tests, Copilot will too. It replicates what it sees — good patterns and bad patterns.
Better context = better suggestions
Descriptive comments, meaningful variable names, and well-structured existing code all improve what Copilot generates.

Copilot Chat has access to your open file and any selected code — you don't need to paste it in. Use #file to reference other files explicitly.

Copilot Chat

Controlling what Copilot sees

Use @ to choose a chat participant. Use # to attach specific context. Both give you precision — instead of hoping Copilot finds the right thing.

@ Chat Participants — choose a specialised mode

@workspace
Searches your whole project
Indexes and searches across all files in the workspace. Use for: "Find all tests covering the checkout flow", "Are there any tests for the 404 error state?", "Which page objects exist in this project?"
@vscode
VS Code features & settings
Answers questions about the editor itself. Use for: "How do I run only the tests in this file?", "What setting controls test retries?", "How do I configure Playwright in launch.json?"
@terminal
Terminal commands & shell
Explains and generates shell commands in context. Use for: "What command runs only my smoke tests?", "How do I filter Playwright test results by tag?"

# Context Variables — attach exactly what Copilot needs

File & code references
  • #file — attach a specific file by name
  • #selection — attach your currently selected code
  • #editor — attach the full content of the active file
  • #codebase — broad search across all files (like @workspace)
Terminal references
  • #terminalLastCommand — the last command you ran
  • #terminalSelection — selected text in the terminal
QA examples
  • "Fix this failing assertion #selection"
  • "Does #file:login.spec.ts cover the lockout case?"
  • "Why did this fail? #terminalLastCommand"
How To

Using # references in practice

Type # in the Chat input — a picker appears

#selection — fastest way
1
Highlight a failing assertion in your test file
2
Open Copilot Chat
3
Type: Why would this fail in CI? #selection

No copy-paste. Copilot sees your question and your highlighted code together.

#file — three ways to attach
  • Type and pick: type #file → file picker opens
  • Drag and drop: drag any file from Explorer into the Chat input — it attaches instantly
  • Inline name: #file:login.spec.ts directly in your message

QA examples — copy these patterns

# Diagnose a failure from the test runner
Fix this  #testFailure

# Find coverage gaps in your PR
What test coverage is missing for  #changes

# Explain a terminal error without copy-pasting
What went wrong?  #terminalLastCommand

# Reference another test file without opening it
Does  #file:checkout.spec.ts  cover the empty cart edge case?

# Fix everything in the Problems panel at once
Write tests that would catch these errors  #problems

Key mental model: Without # — Copilot only sees your open file and your message text, and fills in the rest with educated guesses. With # — you hand it exactly the right material. Like the difference between asking a colleague "my test is broken" versus handing them the error, the test code, and the file it lives in.

Concept 4

Honest expectations

Copilot is great at
  • Writing test boilerplate from a comment
  • Adding tests that match your existing patterns
  • Explaining unfamiliar code in plain English
  • First-pass debugging of failing tests
  • Generating test data and edge case variations
  • Refactoring repetitive test code
Copilot cannot
  • Know your actual app's DOM or selectors
  • Run or execute code to verify it works
  • Know your team's naming conventions
  • Be certain about current library API versions

Copilot can write a test that passes but asserts the wrong thing. A test that always passes regardless of app behaviour is worse than no test at all.

Concept 5

Privacy and code sharing

Individual
Code may be used for training
Code snippets sent to Copilot may be used to improve the model. Can be opted out in GitHub settings under Copilot preferences.
Business / Enterprise
Code is NOT used for training
By default, your code is not used to train Copilot models. Additional security and policy controls available at Enterprise tier.

Three rules that apply everywhere: Never paste API keys or passwords. Never paste customer data or PII. Always check your organisation's AI usage policy first.

Mindset Shift

How to think about Copilot

Wrong mental model
  • Fancy autocomplete that guesses next words
  • Produces code you can use without reading
  • Knows your app and conventions
  • A replacement for thinking through the test
Right mental model
  • A pair programmer who types fast and knows every API
  • Produces a first draft you review and verify
  • Knows patterns — not your specific app
  • A tool that amplifies your QA expertise

Your job is to steer. Copilot's job is to execute. You are still the engineer.

Workflow Patterns

8 patterns for your daily work

Writing & Generating

Write a Test from Scratch
P1
Start with a comment describing what the test should do. Copilot generates the skeleton — you fix selectors, add assertions, verify before keeping.
Add to an Existing Suite
P2
Open a test file that already has tests. Copilot reads your patterns and matches your style — selectors, assertion approach, imports — automatically.
Generate Test Data
P3
Ask Chat for test users, boundary values, invalid inputs, or edge case strings. What takes 20 minutes manually takes 10 seconds.

Understanding & Fixing

Understand Unfamiliar Code
P4
Select any function → right-click → Copilot: Explain. Plain-English breakdown in 30 seconds. Invaluable when onboarding or reviewing a PR.
Debug a Failing Test
P5
Paste the full error message and the failing test. Ask for ranked hypotheses. Copilot narrows the cause — you verify by running the test.
Refactor Repetitive Patterns
P6
Select copy-pasted setup code across tests. Ask Copilot to extract it into a shared helper or page object. Review the result before applying.

Getting the Most from Copilot

Completions Best Practices
P7
Write a descriptive comment above a function — that's your brief to Copilot. Use Alt+] to cycle alternatives. Accept word-by-word with Ctrl+→.
Chat vs Completions
P8
Quick and repetitive → Completions. Complex, multi-step, or needs reasoning → Chat. After a few weeks this becomes instinct.

The Rule Across All 8

Run the test first
Always
Never commit Copilot-generated test code without running it. A test that fails to run is obvious. A test that passes but asserts the wrong thing is dangerous.
Verify selectors and assertions
Always
Check selectors against the real UI. Confirm assertions test actual app behaviour — not just that an element exists on the page.

P1 · Write a test from scratch

Without Copilot

Open blank file, look up API, write boilerplate, check docs for selectors, repeat.

With Copilot

Write a comment describing the test. Pause. Tab to accept the skeleton. Fix selectors and verify assertions. 

// Test: user fills the checkout form and sees order confirmation
test('user fills the checkout form and sees order confirmation', async ({ page }) => {
  await page.goto('/checkout');
  await page.getByLabel('Full name').fill('Jane Smith');
  await page.getByLabel('Card number').fill('4111111111111111');
  await page.getByRole('button', { name: 'Place order' }).click();
  await expect(page.getByText('Order confirmed')).toBeVisible();
});

P2 · Add to an existing suite

Copilot reads your existing tests and replicates your style — selector strategy, assertion patterns, import structure. The better your existing tests, the better the new ones.

P3 · Generate test data

Chat prompt
Generate 10 test users as a TypeScript array.
Each needs: name, email, role (admin/viewer),
and an invalid password that fails our
validation (min 8 chars, 1 number required).
Also useful for
  • Boundary values for numeric inputs
  • Invalid email formats for negative tests
  • Edge case strings (empty, very long, special chars)
  • Date ranges and timezone edge cases

P4 · Understand unfamiliar code

Select any code → right-click → Copilot: Explain. Or open Chat and ask "What does this function do and when would it fail?" Instant explanation in plain English.

Pattern 5

Debugging a failing test

1
Copy the full error message
Don't describe it — paste it. The exact wording is the signal Copilot uses to reason about the cause.
2
Paste the failing test code
Select the test → open Chat → paste both the error and the test. Add context: "passes locally, fails in CI on GitHub Actions."
3
Ask for ranked hypotheses
"What are the most likely causes? List them ranked by probability." You get a shortlist to investigate, not a single guess.
4
Verify each fix by running the test
Copilot gives hypotheses — not guaranteed fixes. It didn't run the test. Always verify before moving on.

P6 · Refactor repetitive patterns

Select repeated code across tests → open Chat → "Extract this into a reusable helper. Match the style of the existing test file." Review the result before applying.

P7 · Inline completions tips

Habits that help
  • Write a comment above what you're building
  • Alt+] / Option+] cycles alternatives
  • Ctrl+→ accepts one word at a time
P8 · Chat vs Completions
  • Completions — quick, repetitive, inline
  • Chat — complex, multi-file, needs reasoning
  • After a few weeks this becomes instinct

The rule across all 7 patterns presented: Run the test before committing. Check selectors against the real UI. Verify assertions are testing the right behaviour.

Agent Mode

P8 - What changed — and why it matters

Copilot Chat (normal mode)
  • You ask → it answers one response
  • Generates text and code suggestions
  • You apply changes manually
  • No access to terminal or file system
  • One round trip per message
Copilot Agent Mode
  • You give a goal → it figures out the steps
  • Reads project files, plans, edits, verifies
  • Applies changes itself (with your approval)
  • Can run terminal commands and check output
  • Multi-step autonomous execution

Enable via VS Code Copilot Chat → mode dropdown → Agent. Requires Individual, Business, or Enterprise plan — not available on the free tier.

Agent Mode

Capabilities and limits

What Agent Mode can do
  • Read your existing codebase for context
  • Edit multiple files in one run
  • Run terminal commands and check results
  • Self-correct when a command fails
  • Show its plan before making changes
  • Build a full test file from a one-sentence goal
Agent Mode cannot
  • Guarantee tests are logically correct
  • Know your business rules or domain logic
  • Stay within scope without clear constraints
  • Understand what your app actually does at runtime

A vague prompt can cause Agent Mode to edit files you didn't intend. Always scope your prompt clearly — tell it exactly what files or folders it should touch.

Agent Mode

You always stay in control

1
Agent shows its plan
Before doing anything, it lists the steps it intends to take. You can redirect or stop here.
2
Diffs before every file change
Every edit shows as a diff. You approve or reject each change individually — nothing is applied without your say-so.
3
Runs verification commands
After applying changes, it runs commands (like your test runner) to verify its work. If something fails, it tries to fix it.
4
Stop any time
Hit the stop button at any point in the run. No partial changes are applied unless you approved them step by step.

When to use Agent Mode

Good candidates
  • Build a test suite for a new feature from scratch
  • Refactor repetitive tests across multiple files
  • Add page objects to an existing suite
  • Generate data fixtures and test helpers
Stick with Chat or Completions
  • Single-file changes or small fixes
  • Quick explanations or one-off questions
  • Adding one test to an existing file

Writing a good Agent prompt

Too vague

"Write tests for the checkout flow."

Structured and clear

"In /tests/checkout/, write Playwright TypeScript tests for the 3-step checkout flow. Match the patterns in /tests/auth/login.spec.ts. Don't touch any files outside /tests/checkout/. Run the tests after creating them."

Watch Out For

Common mistakes to avoid

Mistake 1
Accepting without reviewing
Tab on every suggestion without checking. Copilot replicates patterns — it will also replicate your bad habits and may generate selectors that don't exist in your UI.
Mistake 2
Trusting a passing test
A Copilot-generated test can pass but assert the wrong thing. Always verify that the test is checking real app behaviour — not just that an element is present.
Mistake 3
Vague Agent Mode prompts
A vague goal can cause Agent Mode to edit files across your project that you didn't intend to change. Always define scope with folder paths and explicit constraints.
Mistake 4
Pasting secrets into prompts
Never paste API keys, passwords, or customer data into Copilot Chat or any AI tool — regardless of your plan's privacy policy.
Key Takeaways

What to leave with today

1
Copilot has two modes: Completions (passive, watches and suggests) and Chat (active, you ask). Both are useful — learn when to reach for each.
2
Better context = better suggestions. Descriptive comments, meaningful names, and well-structured existing tests all improve what Copilot generates.
3
Copilot doesn't know your app. It knows patterns. Always verify selectors against the real UI and run every generated test before committing it.
4
Agent Mode is Copilot with tools: it can read files, edit multiple files, run commands, and self-correct. Use it for multi-step tasks. Always define scope clearly in your prompt.
5
You stay in control. Every Agent Mode change shows as a diff before being applied. Review it like a PR. Reject anything you didn't intend.
6
A passing test is not a correct test. Copilot can generate tests that always pass but assert the wrong thing. Verify that each test actually checks real app behaviour.
