Copilot in Your Daily Workflow

Eight practical patterns — each with a complete prompt, a review checklist, and the one mistake people make.

Last verified: May 2026  ·  GitHub Copilot Chat  ·  Playwright 1.44+  ·  TypeScript 5.x
⚡
Quick Start
Open Copilot Chat (Ctrl+Alt+I). Type: "I need a Playwright TypeScript test for the login page. The page is at /login. Email field: #email. Password field: #password. Submit button: #submit. On success, redirect to /dashboard." Read what comes back, then verify the selectors exist in your actual app before using it.

Prerequisite: Both extensions installed and Copilot active. If you haven't done the setup from Session 04 →, do that first — especially installing GitHub Copilot Chat.

The Right Mindset First

Think of Copilot as a pair programmer who types fast but has never seen your app.

Before the patterns, one mental model change. The biggest mistake with Copilot: treating it as smarter autocomplete — only using completions, accepting the first suggestion without looking, and giving vague context expecting magic.

The better model: Copilot is a pair programmer who types faster than you, knows every API, but has never seen your app before. Your job is to steer. Their job is to implement. The quality of the output is directly proportional to the quality of your direction.

The 8 patterns that follow are drawn from real QA workflows. The difference between a mediocre result and a useful one is usually in the prompt structure.

Pattern 1 — Writing a Test from Scratch

From acceptance criteria to a reviewable test in under 2 minutes.

Pattern 1
Writing a test from scratch

You have acceptance criteria and a blank file. The goal is to go from ACs to a reviewable test in under 2 minutes.

Complete Prompt
I need Playwright TypeScript tests for user registration.

Acceptance criteria:
- Email must be unique — duplicate email shows error: "An account with this email already exists"
- Password must be at least 8 characters with one uppercase and one number
- On success, redirect to /dashboard and show "Welcome" banner
- Submit button is disabled during the API call

Use the Page Object pattern. Reference existing style: #file:tests/auth/login.spec.ts
Output to: tests/registration/register.spec.ts
Do not create additional files or modify existing ones.
What Good Output Looks Like

Copilot should produce a describe block with 4 test cases matching your ACs, using page.locator() (not page.waitForSelector()), assertions that check specific text rather than just visibility, and imports that match your existing pattern.

Review Checklist Before Using
  • Do the selectors (#email, #password, etc.) actually exist in your DOM? Verify in browser DevTools.
  • Are assertions checking specific text ("An account with this email already exists") or just toBeVisible()?
  • Does it use page.locator() not deprecated page.waitForSelector()?
  • Are the imports consistent with the file you referenced?
  • Run it. Does it pass on the real app or fail immediately?
Common Mistake

Not referencing an existing file with #file:. Without a style reference, Copilot generates valid Playwright code but using patterns that may not match your project's conventions — different fixture setup, different assertion style, different naming. Adding #file: to a similar spec takes 10 seconds and saves 20 minutes of cleanup.

Pattern 2 — Adding to an Existing Test Suite

Add new scenarios without breaking what's already there.

Pattern 2
Adding to an existing test suite

You have an existing spec file and need to add new scenarios without breaking what's there.

Complete Prompt
Add 3 new test cases to #file:tests/checkout/checkout.spec.ts

New scenarios:
1. Guest checkout without creating an account
2. Checkout with an expired promo code — should show "This code has expired" error
3. Checkout when one item goes out of stock mid-session — cart should show stock warning

Output only the new test() blocks. I will insert them manually.
Match the existing fixture pattern and assertion style exactly.
Review Checklist Before Using
  • Did it output only the new test() blocks, or regenerate the whole file? If the whole file, ask: "Output only the new test() blocks, not the full file."
  • Do the new tests use the same fixture setup as existing tests? Compare the async ({ page, ... }) signature.
  • Are the error message strings exact? "This code has expired" vs "Code has expired" will cause the test to always fail — Copilot often invents error text.
Common Mistake

Copilot regenerates the entire file when you wanted only the new tests. It does this to "be helpful" — it may fix imports, reformat existing tests, or reorganise describe blocks. Always say "output only the new test() blocks" to prevent this.

Pattern 3 — Generating Test Data & Fixtures

Fixtures, boundary values, and realistic user data — quickly and consistently.

Pattern 3
Generating test data and fixtures

Test data is tedious to write but important for meaningful tests. Copilot handles this quickly and consistently.

Complete Prompt — User Fixtures 
Generate a TypeScript file at tests/fixtures/users.ts with 10 test users.

Cover these scenarios:
- Standard active user
- Admin user with all permissions
- Read-only user
- User with MFA enabled
- User with expired password (must change on next login)
- Locked account (failed too many login attempts)
- New user, no profile completed
- User with no profile photo

Use realistic UK first names, surnames, email formats.
Each user: { id, email, password, role, status, mfaEnabled, profileComplete }
Export as: export const testUsers = [...] and individual named exports for each.
Plain text passwords are fine — this is test data only.
Complete Prompt — Boundary Value Data 
Create a TypeScript array for boundary-value testing on a date-of-birth field.

Cover:
- Valid adult (clearly 18+)
- Exactly 18 years old today (lower boundary)
- One day under 18 (fails validation)
- Over 100 years old (upper boundary)
- Future date
- Invalid format: "13/32/1990", "1990-13-01", "not-a-date"
- Empty string

Type: Array<{ input: string; expectedError: string | null }>
null = no error expected.
Common Mistake

Using hardcoded "today's date" in the output. For boundary cases like "exactly 18 today", Copilot may generate a hardcoded date string that will be wrong next year. Ask for it to compute the date dynamically: "Use JavaScript date arithmetic rather than a hardcoded date string."

Pattern 4 — Understanding Unfamiliar Code

Ask Copilot to explain before you trace through it manually.

Pattern 4
Understanding unfamiliar code

You're handed a test file, a helper, or a Page Object you've never seen. Ask Copilot before you start trying to trace through it manually.

Complete Prompt
Explain #selection in plain English. Focus on:
1. What inputs it expects and what it returns or does
2. The main logic paths (normal flow + what happens when something's wrong)
3. What could make it behave unexpectedly or produce a false positive in tests
Do not include code in your response. Use plain sentences only.
Why This Works

Asking for "logic paths" and "what could make it behave unexpectedly" directly maps to test scenarios. You get an explanation and a mental map of what to test — in one request.

Common Mistake

Using /explain without selecting anything first, then getting a generic explanation of the file rather than the specific function you're confused about. Select the function, then ask.

Pattern 5 — Debugging a Failing Test

The Debugging Brief — the single most useful structure you'll use with Copilot.

Pattern 5
Debugging a failing test

Context is everything for debugging. The debugging brief below is the single most useful structure you'll use with Copilot.

The Debugging Brief (copy and fill in) 
Framework & version: Playwright v1.44, TypeScript, headless Chromium
Test file: tests/checkout/payment.spec.ts
Test name: "should complete purchase with saved card"

Error message:
[paste the full error and stack trace here — all of it]

Test code:
[paste the relevant test or beforeEach here]

When it fails: [local only / CI only / intermittent / always fails]
What I've already tried: [be specific — "added waitForSelector" not just "tried waiting"]
Why "When it fails" is the most important field

Local-only failures usually mean an environment issue (credentials, local config). CI-only failures usually mean timing, headless rendering, or missing environment variables. Intermittent failures suggest a race condition or flakiness. That one word ("CI only") points Copilot's diagnosis in completely the right direction.

Common Mistake

Pasting only the final error line and not the full stack trace. The stack trace shows exactly where in the execution the failure occurred. Without it, Copilot is guessing from the error message alone — and so are you.

Pattern 6 — Refactoring Duplicated Test Code

Get the extraction plan approved before you get the code.

Pattern 6
Refactoring duplicated test code

Test suites accumulate duplication: selectors repeated across 15 files, the same login setup in every spec, fixture data hardcoded everywhere. Get the plan before you get the code.

Complete Prompt — Identify First 
Here are three test files: #file:tests/auth/login.spec.ts,
#file:tests/auth/reset.spec.ts, #file:tests/profile/update.spec.ts

Identify the duplicated patterns (selectors, setup logic, assertions).
Suggest what should be extracted into a shared Page Object or helper.
Do not generate the refactored code yet — just describe the extraction plan.
Then, once you've validated the plan 
The extraction plan looks correct. Now:
1. Create tests/pages/authPage.ts with the Page Object
2. Show me the refactored login.spec.ts only — I'll apply the pattern to the other two files
Do not modify reset.spec.ts or update.spec.ts.
Common Mistake

Asking for the refactored code directly without validating the plan first. Copilot will produce a complete refactor — across all three files, all at once — that you then have to unpick if its extraction decisions don't match your project's architecture. Get the plan approved first.

Patterns 7 & 8 — Completions & Knowing Which Mode to Use

Improve completion quality and know when to switch between Chat and Completions.

Pattern 7
Inline completions best practices

Completions are passive, but you can significantly improve their quality with small habits.

What makes completions better
  • Write a descriptive test name before the body. loginWithExpiredPassword signals the intent far better than test2.
  • Write a comment above the function. // Verify rate limiting blocks after 5 failed attempts gives Copilot the scenario to generate.
  • Leave the cursor inside an incomplete structure. Positioning inside test('...', async ({ page }) => { after the opening brace generates the entire body.
  • Accept word-by-word (Ctrl+→ / Cmd+→) when a suggestion is mostly right but needs minor adjustments — you don't have to take it all or nothing.
  • Open the file you want to match in another tab. Copilot uses open files as context — if your Page Object file is open, completions will use its method names.
Common Mistake

Accepting completions without checking the selectors. The most common Copilot failure mode: it generates page.click('#login-btn') when your actual button is page.getByRole('button', { name: 'Sign in' }). The test will fail. Always verify against your real DOM.

Pattern 8
When to use Chat vs Completions

Using the wrong mode doesn't break anything, but it wastes time. Use this as a quick reference.

Use Completions when Use Chat when
You're mid-flow and want a suggestion You want to describe a task and get a full result
The code pattern is repetitive and predictable You need to explain context before asking
You want short snippets or boilerplate You need explanation, not just code
The surrounding file is enough context You need to reference other files with #file:
Speed matters more than precision You want to iterate with follow-up messages
Try This — 5 minutes

Pick any one of the 8 patterns above that's closest to something you're working on right now. Use the exact prompt structure shown. Don't simplify it. Notice whether the output is better than what you'd normally get from a shorter, vaguer prompt.

Slash Commands on the Same Test

See what /explain, /fix, /tests, and /doc each produce from identical input.

One of the best ways to understand what each slash command does differently is to run all four on the same piece of code. Here's what each produces when run on a simple login test:

Same login test — four different outputs 
// Select this entire test, then use each slash command
test('user can log in with valid credentials', async ({ page }) => {
  await page.goto('/login');
  await page.fill('#email', 'user@example.com');
  await page.fill('#password', 'ValidPass1!');
  await page.click('#submit');
  await expect(page).toHaveURL('/dashboard');
});
/explain
Returns a plain-English description: "This test navigates to /login, fills the email and password fields using CSS ID selectors, clicks the submit button, and asserts that the page redirects to /dashboard after a successful login. It does not verify whether the user's session is stored or whether the dashboard renders correctly."
/fix
Suggests improvements: "Consider using page.locator() instead of raw CSS selectors for better resilience. The test only verifies the URL — consider adding a check that a dashboard element is visible, since /dashboard could redirect again."
/tests
Generates additional test cases based on this one: invalid email format, wrong password, empty fields, account locked, session persistence. Picks up the pattern from your existing test.
/doc
Adds a JSDoc comment above the test: selector assumptions, preconditions (user account must exist with those credentials), expected state after the test.
Try This — 3 minutes

Open any existing test. Select the whole test function. Open Chat (Ctrl+Alt+I). Run /explain. Then run /fix on the same selection. Compare the two responses — /explain describes what it does; /fix describes what it should do differently.

When Copilot Gets It Wrong

Three predictable failure modes — knowing them saves time and prevents broken tests.

Copilot is confidently wrong in predictable ways. Knowing these patterns saves time and prevents you committing broken tests.

Failure 1 — Inventing API methods that don't exist

Copilot may suggest page.waitForStable(). It sounds completely plausible — there's a waitForSelector, a waitForURL, a waitForLoadState — so the model predicts waitForStable should exist too.

It doesn't. When you run the test, you get: TypeError: page.waitForStable is not a function.

Fix: For any Playwright method Copilot suggests that you're not sure about, check the Playwright API docs. The official docs list every method. If it's not there, it doesn't exist. The real equivalent for "wait until the page stops loading" is page.waitForLoadState('networkidle') or page.waitForLoadState('domcontentloaded').

Failure 2 — Using Playwright v1.20 patterns in a v1.44 project

Copilot's training data contains Playwright code from many different versions. It may suggest page.waitForSelector('#submit') when you're using Playwright v1.44+ where the recommended pattern is page.locator('#submit').waitFor() or simply using await page.locator('#submit').click() (which auto-waits).

The old patterns still work — Playwright maintains backwards compatibility — but they're deprecated and will generate warnings. More importantly, they bypass Playwright's smart auto-waiting and can produce flaky tests.

Fix: Add a clear version constraint to your prompt: "Use Playwright v1.44 patterns. Prefer page.locator() and page.getByRole() over raw selectors. Do not use waitForSelector."

Failure 3 — Writing always-passing assertions

Copilot generates this kind of test:

// This test passes whether login works or not
test('login page is visible', async ({ page }) => {
  await page.goto('/login');
  await expect(page.locator('#email')).toBeVisible();
});

The email field is always visible — it's part of the static page structure. This test passes even if the login form is completely broken. It's testing that the page loaded, not that login works.

Fix: After generation, ask yourself: "If the feature I'm testing broke, would this test fail?" If the answer is "not necessarily," the assertion needs strengthening. For login tests, assert on what happens after you submit: redirect URL, dashboard content visible, session cookie set.

Rule of thumb: toBeVisible() on a static element is almost never a meaningful assertion. Meaningful assertions check: content (toHaveText()), state (toBeEnabled() / toBeDisabled()), navigation (toHaveURL()), or element count (toHaveCount()).

Progressive Example — The Same Login Test, Four Ways

Watch how AI assistance transforms a vague test into a production-ready one.

Watch how the same test evolves as you apply AI assistance at each stage.

Stage 1 — Prompt → raw output

You asked: "Write a Playwright test for login." Copilot generates: 

// ⚠ Raw output — do not commit without review
test('login', async ({ page }) => {
  await page.goto('/login');
  await page.fill('input[type="email"]', 'user@test.com');
  await page.fill('input[type="password"]', 'password');
  await page.click('button[type="submit"]');
  await expect(page.locator('.dashboard')).toBeVisible();
});
Stage 2 — After review: better selectors, stronger assertion

You asked Copilot: "Improve the selectors to use getByRole or getByLabel, and strengthen the post-login assertion to check a specific dashboard element." Result: 

test('user can log in with valid credentials', async ({ page }) => {
  await page.goto('/login');
  await page.getByLabel('Email address').fill('user@example.com');
  await page.getByLabel('Password').fill('ValidPass1!');
  await page.getByRole('button', { name: 'Sign in' }).click();
  await expect(page).toHaveURL('/dashboard');
  await expect(page.getByRole('heading', { name: 'Welcome back' })).toBeVisible();
});
Stage 3 — Add to Page Object via Copilot

You asked: "Extract the login steps into a method on #file:tests/pages/loginPage.ts." Copilot adds a login(email, password) method to your Page Object. Then it updates the spec: 

import { LoginPage } from '../pages/loginPage';

test('user can log in with valid credentials', async ({ page }) => {
  const loginPage = new LoginPage(page);
  await loginPage.login('user@example.com', 'ValidPass1!');
  await expect(page).toHaveURL('/dashboard');
  await expect(page.getByRole('heading', { name: 'Welcome back' })).toBeVisible();
});

Three stages of improvement, each one a short focused Copilot interaction. The final test is readable, resilient, and ready to commit.
