What Is a Terminal Agent?

Same LLM core as Copilot — but running in your shell with direct access to your entire project.

A terminal agent is an LLM that has been given access to your shell. It can read and write files anywhere in your project, run any terminal command, install packages, run your test suite, and read the output — all in a loop, without you manually executing each step.

Copilot Agent Mode (VS Code)
  • Runs inside VS Code — scoped to open workspace
  • Shows proposed file changes before applying
  • Runs terminal commands with your approval each time
  • Good for focused, file-level tasks within one project
  • Familiar UI — same editor you work in
Terminal Agents (Claude Code / Opencode)
  • Runs in your shell — works from any directory
  • Reads and writes files autonomously (configurable)
  • Executes commands directly without per-step approval
  • Better for multi-step tasks across many files
  • Runs headless — useful for CI/CD and scripting

Higher capability = higher oversight need. A terminal agent can delete files, run database migrations, or install packages system-wide if you ask it to. This is powerful and requires you to review what it's about to do before confirming — especially for anything irreversible. Session 13 covers safe task decomposition in detail.

Claude Code vs Opencode

Two terminal agents with different model backends and pricing models. Both follow the same agentic loop.

Claude Code
Anthropic — Claude models
The official Anthropic CLI agent. Uses Claude Sonnet / Opus models. Billed per token via your Anthropic account (or Claude Max subscription for flat-rate). Excellent at multi-file reasoning, long context, and test suite tasks. Install: npm install -g @anthropic-ai/claude-code
Opencode
Open source — multiple model backends
Open-source terminal agent. Supports Anthropic, OpenAI, Google Gemini, and GitHub Copilot as model backends. If your organisation has a Copilot Enterprise licence, Opencode can use that — no additional billing. Install: npm install -g opencode-ai

Which one to use?

You have an Anthropic API key or Claude Max subscription
Claude Code — most capable, best at long-context reasoning
Your organisation has GitHub Copilot Enterprise
Opencode with Copilot backend — no extra billing
You want to try terminal agents without committing to a billing account
Opencode with a free-tier OpenAI or Gemini key
You want the best results for a complex QA automation task
Claude Code with Claude Sonnet — optimised for code tasks

Session 15 covers Opencode with the GitHub Copilot backend in detail — including how to configure it and what model options are available.

Installing Claude Code

Prerequisites, install command, and first run.

Prerequisites

1
Node.js 18+ installed
Check: node --version. If missing, install from nodejs.org. Claude Code is distributed as an npm package and requires a modern Node runtime.
2
Anthropic API key or Claude Max plan
Get an API key from console.anthropic.com, or use a Claude Max subscription which includes Claude Code at a flat monthly rate. The API key is set via ANTHROPIC_API_KEY environment variable.
3
Install Claude Code
Run: npm install -g @anthropic-ai/claude-code. This installs the claude CLI globally. Verify with claude --version.
4
Navigate to your project and start
Change to your project directory: cd my-playwright-project. Then run: claude. You'll be greeted with an interactive prompt. Type your task and press Enter.
# Install
npm install -g @anthropic-ai/claude-code

# Set API key (add to your shell profile for persistence)
export ANTHROPIC_API_KEY=your-key-here

# Navigate to your project
cd my-playwright-project

# Start Claude Code
claude

The Agent Loop — What Happens

A real task from prompt to completion — what the agent does at each step and why.

When you give Claude Code a task, it doesn't respond immediately with code. It first explores your project, then plans, then acts, then verifies. Here's what that looks like for a typical QA task:

Claude Code Session — "Add tests for the password reset flow"
You
Add Playwright tests for the password reset flow. The page is at /reset-password. Match the style of existing tests.
Agent
Reading playwright.config.ts… Reading tests/ directory structure… Reading tests/auth/login.spec.ts for style reference… I can see you use the Page Object Model with data-testid selectors, and each test is self-contained. I'll follow that pattern. Let me look at the reset page structure.
Agent
Reading src/pages/reset-password.tsx… I found the selectors: input[data-testid="email-input"], button[data-testid="send-reset-link"], and the success state shows div[data-testid="confirmation-message"]. I'll write three test cases: successful submission, invalid email, and already-used token.
Agent
Writing tests/auth/reset-password.spec.ts… Running: npx playwright test tests/auth/reset-password.spec.ts…
Agent
Test output: 2 passed, 1 failed — "invalid email" test timed out on selector. The error shows the validation message uses span.field-error not a data-testid. Updating the selector now. Editing tests/auth/reset-password.spec.ts… Running again… 3 passed. Done.

Notice: you gave one instruction. The agent read 4 files, wrote 1 file, ran tests twice, read the failure, and self-corrected — all without further input from you.

What to Watch For During a Run

How to monitor what the agent is doing and when to intervene.

Terminal agents are designed to run to completion. But you should watch the output, not walk away. These are the moments that warrant your attention:

→
When it asks for permission — Claude Code will pause and ask before running commands that could have side effects (installing packages, deleting files, running migrations). Read the proposed command before approving.
→
When it edits files you didn't expect — If the agent modifies a file you didn't mention, check why. It may have correctly identified a related file that needed updating — or it may have gone off-scope.
→
When it loops on errors — If the agent tries the same fix three times and keeps failing, interrupt with Ctrl+C and re-task it with more specific information about the problem.
→
At the end — review the git diff — Run git diff after a session to see everything that changed. This is your final review gate. The agent's output summary is accurate but abbreviated — the diff shows the full picture.
# After a Claude Code session, always review what changed
git diff

# Or for a summary of changed files
git diff --stat

# To discard changes you don't want
git checkout -- path/to/file

Your First Terminal Agent Task

A safe, scoped task to get your first run done.

Try This — 15 minutes

Run a scoped, low-risk task on your project. Good first tasks are ones where the scope is clearly bounded and reversible:

Option A — Test generation: "Add tests for [one specific feature]. Match the style of existing tests in tests/[folder]." Review the git diff after. Check selectors and assertions as you learned in Session 07.

Option B — Review task: "Review the tests in tests/[folder] for quality issues. List what you find, don't change any files yet." This gives you the agent's analysis without any file writes — safe for a first run.

Option C — Documentation: "Read the test files in tests/ and write a short markdown summary of what each test suite covers. Save it as TEST-COVERAGE.md." Low risk, useful output, and you'll see the full read → plan → write cycle.

After your run: check git diff, review what changed, and note which parts of the workflow felt unfamiliar.
