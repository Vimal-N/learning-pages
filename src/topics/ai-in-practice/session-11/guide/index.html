The Agent Evolution — Inline to Autonomous

The four stages of AI tooling, what changes at each step, and the three mental shifts that make you effective with agents rather than just frustrated by them.

Four Stages of AI Interaction

Each stage changes what you do and what the AI does. Understanding the difference prevents you from using a Stage 4 tool with Stage 1 habits.

STAGE 1
Inline Completions
Ghost text
You type, AI suggests the next tokens. You accept or dismiss with a keypress. Zero autonomy — the AI never initiates, never reads your project. Your oversight model: glance at the suggestion before pressing Tab.
STAGE 2
Conversational Chat
One turn at a time
You ask, AI answers. You copy code manually. One turn at a time — each response is complete and self-contained. Your oversight model: read the output before using it.
STAGE 3
Agent Mode (Editor)
Scoped autonomy
AI reads files, proposes changes, you approve. Multi-step task in a loop. Scoped to your VS Code workspace. Your oversight model: review proposed diffs and terminal commands before each is applied.
STAGE 4
Terminal Agent
Full autonomy
AI reads, writes, runs, and iterates autonomously. Full filesystem access. Loop runs to completion. Your oversight model: review git diff at the end — not each individual step.

Common failure mode: Using Stage 4 tools with Stage 2 habits. Treating the agent's output summary as a full code review, rather than running git diff to see what actually changed. The oversight model has to match the stage of the tool.

Three Mental Shifts for Agent Work

Moving from chat habits to agent habits isn't automatic. These three shifts are where most people get stuck.

SHIFT 1 From prompting to tasking
In chat, you write a detailed prompt that shapes every aspect of the output — format, tone, constraints, examples. In agent mode, you write a task: the goal, the constraints on scope, and how you'll verify it's done. The agent figures out the steps. Overly detailed step-by-step instructions interfere with the agent's ability to plan — you're doing its job for it.
Example
Chat habit (too procedural): "First read the login spec, then look at the POM, then write a test for forgot-password that uses page.fill() and has three test cases and uses the existing LoginPage class…"
Agent habit (goal + constraints): "Add tests for the forgot-password flow. Match the existing test style. Don't modify any existing files — create a new spec file only."
SHIFT 2 From reviewing text to reviewing actions
In chat, you review a text response. In agent mode, you review what the agent did to your project — file writes, file deletes, commands run, packages installed. The agent's summary is not your review. git diff is your review.
Example
Agent says: "Done — I added 3 tests and updated the Page Object." You run git diff and see it also modified playwright.config.ts to add a new project. That change is fine — but you wouldn't have known without the diff.
SHIFT 3 From one-shot to iterative tasking
Chat works best when you get everything into one long prompt. Agents work best when you break tasks into independently verifiable chunks and run them sequentially. Each task should have a clear completion condition you can check before moving on.
Example
Too big (one shot): "Refactor the entire test suite to use Page Objects, fix the flaky tests, add coverage for the checkout flow, and update the README."
Better (chunked):
Task 1: "Extract a CheckoutPage object from checkout.spec.ts." → review diff
Task 2: "Update checkout tests to use the new CheckoutPage." → run tests → review diff
Task 3: "Add tests for the payment failure scenario." → run tests → review diff

The Right Oversight Model

What to review, when to review it, and what to do when the agent goes wrong.

Oversight doesn't disappear with agents — it changes shape. You're reviewing less "is this code correct?" and more "is this what I asked for and is it safe to commit?"

The end-of-session review checklist

→
Run git diff — See every change. Check that nothing outside your intended scope was modified.
→
Run your tests — The agent ran them during its session, but run them again yourself. Tests pass in the agent's environment doesn't mean they pass in CI with your full suite.
→
Read the new test logic — Not every line, but understand what each new test is checking. A test that's structurally correct but asserts the wrong thing is worse than no test.
→
Check for scope creep — Did the agent modify shared utilities, config files, or package.json unexpectedly? These need extra scrutiny — changes here affect every test, not just the ones you asked about.

When to intervene mid-session

Agent asks to run a command you don't recognise
Read the command carefully before approving — ask the agent to explain it
Agent loops on the same error 3+ times
Interrupt with Ctrl+C, re-task with more specific context
Agent proposes editing a file you didn't mention
Read its reasoning — it may be correct, or it may have misunderstood scope
Agent seems confused about project structure
Add a CLAUDE.md to help it understand your conventions (Session 12)
