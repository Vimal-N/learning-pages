What Is a Subagent?

A subagent is an agent spawned by another agent to handle a bounded piece of work — then report back.

In a multi-agent workflow, there is typically one orchestrator (the coordinator agent) and one or more subagents (specialist agents). The orchestrator receives the high-level goal, breaks it into tasks, spawns subagents to handle each task, collects their results, and synthesises a final response.

Single agent
  • One LLM handles everything in sequence
  • Context window fills up on large tasks
  • Can't do two things simultaneously
  • Good for focused, single-domain tasks
  • Simpler to debug — one conversation to inspect
Orchestrated subagents
  • Each subagent has a fresh, focused context
  • Subagents can run in parallel (where supported)
  • Specialist subagents perform better on their domain
  • Good for large, multi-domain tasks
  • More complex — more moving parts to debug

Don't reach for multi-agent by default. A single well-prompted terminal agent handles most QA tasks better than a complex orchestrated workflow. Multi-agent is worth it when tasks are genuinely parallel, when context window limits are a real constraint, or when specialist domain separation produces measurably better output.

The Orchestrator Pattern

How the coordinator and subagent relationship works — what each sees and what each does.

1
You give the orchestrator a high-level goal
The orchestrator's job is decomposition and coordination — not doing the work itself. It receives your goal, reads the available context, and decides what tasks need doing and in what order.
2
Orchestrator spawns subagents with specific prompts
Each subagent receives a focused prompt: the task, the relevant context, the scope constraints, and the expected output format. The subagent doesn't know the broader goal — it only knows its task.
3
Subagents complete their tasks and return results
Each subagent works independently, using its own tools and context. When done, it returns a structured result to the orchestrator — not to you directly.
4
Orchestrator synthesises and reports
The orchestrator collects all subagent results, resolves any conflicts, and produces the final output. You review this — not each individual subagent session.

Workflow Storyboard — Multi-Agent QA Regression Run

Step-by-step: the exact prompts each agent receives in a real QA workflow. Not just the architecture — the actual text.

Goal: After a deploy, run a full regression check, diagnose any new failures, and produce a signed-off test report.

STEP 1 You → Orchestrator
You give the orchestrator a single high-level prompt. It decides the subagent tasks.
Your prompt to the orchestrator
"A new build was deployed 10 minutes ago. Run the full regression suite, diagnose any failures, and produce a test report summarising what passed, what failed, and whether the failures are new or pre-existing."
STEP 2 Orchestrator → Test Runner Subagent
Orchestrator spawns a subagent to run the tests and return raw results.
Prompt sent to the subagent
"Run the full Playwright test suite against BASE_URL=https://staging.app.com. Command: npx playwright test. Wait for completion. Return: total tests, passed count, failed count, and the full text of any failure messages. Do not interpret the results — just return the raw data."
STEP 3 Orchestrator → Diagnosis Subagent
Orchestrator passes the failure data to a second subagent specialised in diagnosis.
Prompt sent to the subagent
"Here are the test failures from the latest run: [failure list from Step 2]. Here is the list of failures from the previous run: [previous results from git/CI]. Classify each current failure as: NEW (not seen before), REGRESSION (was passing, now failing), or PRE-EXISTING (was failing before too). For NEW and REGRESSION failures, provide a likely root cause based on the error message and test name."
STEP 4 Orchestrator → Report Subagent
Final subagent formats the synthesised data into a shareable report.
Prompt sent to the subagent
"Using this data: [test results + diagnosis], write a test report in markdown. Structure: Executive Summary (3 bullet points), Test Results Table (pass/fail/skip counts), Failures by Category (new / regression / pre-existing), Recommended Actions. Tone: factual, no hedging. Save as TEST-REPORT-[date].md."

Designing Your Own Workflow

The questions to answer before building any multi-agent workflow.

1
Does this actually need multiple agents?
Ask: can a single agent with a good GCSV prompt and a CLAUDE.md do this? If yes, do that instead. Multi-agent is worth it for parallelism (run 5 things at once) or when the combined context would exceed a single agent's window.
2
What are the natural task boundaries?
Good subagent boundaries: tasks with different tools, tasks that can run in parallel, tasks that need a "clean slate" context. Bad boundaries: arbitrary splits that create unnecessary handoff complexity.
3
What does each subagent need in its prompt?
Every subagent starts with no context. Its prompt must include: its specific task, all the data it needs (passed from the orchestrator), the output format it should return, and any constraints on scope. It cannot ask follow-up questions.
4
How will you review the workflow's output?
You can't review every subagent's work in a multi-agent run. Design the workflow so the orchestrator's final output is itself reviewable — a report, a test result, a git diff you can check. The orchestrator is your review gate.
Task fits in one context window and can be done sequentially
Single agent — simpler, easier to debug
Multiple independent tasks that could run at the same time
