Writing Test Cases
with AI

Four strategies for using AI to go from zero to a complete test suite — with QA examples at every step.

Prerequisite

You already have the hard part

You
bring the testing strategy
You know what to test — the happy path, edge cases, error conditions, business rules, regression scenarios. This is the hard part that AI can't invent for you.
AI
brings the implementation speed
AI knows the API, the framework syntax, the patterns. It can turn a plain-English test scenario into working Playwright/Cypress/Jest code in seconds.
The result: you cover more test scenarios, faster, than either of you could alone.

4 strategies we'll cover

Strategy 1
New test from feature description
Start with user stories or acceptance criteria — get complete test code
Strategy 2
Tests from existing code
Point AI at a component or API and get tests that match its actual behaviour
Strategy 3
Edge case generation
Show AI your happy-path tests — ask it to find the gaps and generate the missing scenarios
Strategy 4
Test data generation
Get realistic, varied datasets — not just "test@test.com" repeated forever

Strategy 1 — Start with acceptance criteria

I need Playwright tests for our password reset feature. Here are the acceptance criteria: - User enters their email on /forgot-password - If email exists: success message shown, email sent - If email not found: generic error (don't reveal whether email exists) - Submit button disabled during API call - Rate limit: 3 attempts per 10 minutes, then locked out Use our existing Page Object pattern from tests/pages/. Test file goes in tests/auth/.
I'll create tests/pages/forgotPasswordPage.ts and tests/auth/passwordReset.spec.ts covering all 5 scenarios. Let me read your existing Page Object pattern first to match the style...
✓ Notice: acceptance criteria becomes the test specification. The AI uses it to know exactly what to assert — not just what to click.

Strategy 2 — Generate tests from the code itself

Here is our UserService.validateLogin() function. Write Jest unit tests for it. Cover: successful auth, wrong password, account locked, expired token, and null input guard. Output to tests/unit/userService.test.ts.

async validateLogin(email, password) {{ '{' }}...{{ '}' }}
I can see the function checks for account lock status before password comparison. I'll write tests that cover the exact conditional paths in the implementation, including the lock check ordering...
⚠️ Tests from existing code verify current behaviour — including existing bugs. Always compare generated assertions against your requirements too.

Strategy 3 — Find the gaps

Here are my existing tests for the product search feature. What important test cases am I missing? List them as test scenario descriptions first — don't write code yet.
Looking at your test coverage, I can see gaps in several areas:
  1. Search with special characters: &, %, ', "SQL injection attempt"
  2. Search term exceeding max length (what happens at 500+ chars?)
  3. Concurrent searches — does the UI handle rapid successive requests?
  4. Search while page is still loading initial data
  5. Empty results state — is the "no results" message displayed correctly?
  6. Results pagination edge cases — last page, single result page
Want me to generate test code for any of these?
✓ Ask for scenario descriptions first, then code. Lets you validate the test strategy before writing anything.

Strategy 4 — Generate realistic test data

Generate a JSON fixture file for user test data. I need 10 users covering: standard users, admin users, read-only users, accounts with MFA enabled, accounts with expired passwords, locked accounts, and accounts with no profile photo. Use realistic UK names, email addresses, and phone numbers. Output as tests/fixtures/users.json.
Here's tests/fixtures/users.json with 10 realistic UK users covering all the scenarios you specified, each with a clear type label so tests can select the right user by scenario...
Other great test data prompts
"Generate 20 product records with varying prices, stock levels, and category combinations"
"Create boundary-value test inputs for a date-of-birth field: valid, too young, too old, invalid format, leap year edge cases"
"Generate API response mocks for success, 400 validation error, 401 unauthorised, 404 not found, 500 server error"

The non-negotiable review rule

Before committing any AI-generated test, ask these questions
?
If this feature broke tomorrow, would this test catch it?
?
Are the assertions checking the right thing, not just that the element exists?
?
Are all async operations awaited properly?
?
Does the test data actually exercise the scenario, or is it always-valid data?
?
Do I understand every line of this test? If not, ask AI to explain it.
A test you understand and trust is worth more than five tests you just accepted.

Building a full suite — the workflow

S1
Generate from acceptance criteria
Write the core happy-path and documented error scenarios
S3
Ask AI to identify gaps
Show it what you have, get a list of missing edge cases
S4
Generate test data fixtures
Create the realistic data your tests will use across all scenarios
✓
Review, run, iterate
Read every assertion, run the suite, fix failures with AI assistance, commit

Mistakes to avoid

✗
"Write tests for my checkout feature" (no context)
Without acceptance criteria, selectors, framework, or constraints — the output is a generic template with no real value.
✗
Accepting tests without running them first
Always run generated tests before committing. Syntax issues, wrong selectors, missing imports — they happen.
✗
Only writing happy-path tests
The most valuable tests are the ones that fail when something goes wrong. Always include error scenarios.
✗
Using AI selectors without checking they exist
AI may invent plausible-looking selectors that don't exist in your actual DOM. Always verify selectors in the browser.

Key takeaways

1
You bring the strategy, AI brings the code
AI can't invent your acceptance criteria or know your business rules. You provide those — AI handles the implementation.
2
Four strategies for different starting points
Feature description → existing code → edge case finding → test data. Pick the right one for your situation.
3
Ask for scenarios before code
Validating the test plan first catches strategy errors cheaply — before any code is written.
4
Review every assertion — not just that it runs
A passing test that doesn't actually test your feature is a false confidence trap. Read every assert.
