Why AI debugging help fails

Not enough context (useless)
My test is failing. Why?
TimeoutError: locator not found
This could be caused by many things: element not in DOM, wrong selector, timing issue, page not loaded...
Full context (useful)
Playwright v1.44, VS Code, TypeScript. Test clicks a submit button but fails with TimeoutError. Full error + test code + selector attached. Only fails in CI, works locally. CI uses headless Chrome.
The difference between local and CI headless is significant. The selector .submit-btn is likely inside an iframe in CI builds. Here's how to verify and fix...

The debugging brief — what to include

1
Framework + version
Playwright 1.44, Jest 29, Cypress 13 — version matters because behaviour changes between releases
2
The full error output
All of it — not just the first line. Stack traces, logs, timing information. The important detail is usually buried.
3
The failing test code
The whole test, not just the line that failed. Context around the failure matters.
4
When it fails vs when it passes
Local only? CI only? Sometimes? After another test? This is the most diagnostic information you have.
5
What you've already tried
Prevents AI from suggesting the same things you've already ruled out.

Pattern 1 — Error message investigation

TimeoutError: locator.click: Timeout 30000ms exceeded.
Call log:
  - waiting for locator('[data-testid="submit-btn"]')
  - locator resolved to <button disabled>Submit</button>
Playwright 1.44, TypeScript. Getting this error on the checkout form submit test. The test fills in all required fields then clicks submit. Error only happens in our CI environment, not locally. Full test attached. What are the most likely causes, ranked?
The error log is key — the locator resolved to the button (it was found), but the button was disabled. Playwright won't click a disabled element. Most likely causes:
  1. Form validation hasn't completed before the click (CI is slower)
  2. An API call to validate a field is still in-flight
  3. A race condition: the submit button is briefly disabled during page load
Fix: await expect(page.locator('[data-testid="submit-btn"]')).toBeEnabled() before clicking.

Pattern 2 — Diagnosing flaky tests

This Playwright test fails about 20% of the time with "element not found" on the notification banner. It's non-deterministic — no pattern to when it fails. Here's the test. The notification appears after an API call that takes 200–800ms depending on server load.
Classic race condition. On line 14 you have await page.waitForTimeout(500) — hardcoded 500ms wait. When the API takes 600–800ms the notification hasn't appeared yet and the locator times out.

Replace the fixed sleep with an explicit wait: await expect(page.locator('.notification')).toBeVisible(). This waits up to 30s (configurable) and passes as soon as the element appears.
FLAKY
await page.waitForTimeout(500);
await page.locator('.notification').click();
RELIABLE
await expect(page.locator('.notification')).toBeVisible();
await page.locator('.notification').click();

Pattern 3 — Fixing brittle selectors

This selector worked last month but broke after a UI refactor: .main-content > div:nth-child(3) > button.btn-primary. The HTML structure changed. Can you suggest more resilient alternatives?
That selector is fragile in three ways: DOM position dependent, class-name dependent, and structure dependent. Better options in order of preference:
  1. Best: Add data-testid="submit-order-btn" to the HTML and use getByTestId('submit-order-btn')
  2. Good: getByRole('button', {{ '{' }} name: 'Submit Order' {{ '}' }}) — survives structure changes
  3. OK: getByText('Submit Order') — fragile if button text changes

Pattern 4 — CI vs local failures

Test passes 100% locally, fails 100% in CI with screenshot showing the login form is empty when it should be pre-filled. Local: macOS, Chrome headed. CI: Ubuntu, Chrome headless via Docker. What should I check?
The "pre-filled but empty" symptom points to a few specific culprits:
  1. Autofill: Chrome headed respects browser autofill. Headless Chrome in Docker does not — stored credentials won't pre-fill.
  2. localStorage/sessionStorage: If the pre-fill comes from stored state, Docker containers start fresh with no prior state.
  3. Environment variable missing: If pre-fill data comes from an env var, confirm it's set in your CI config.
  4. Cookie not persisted: If using browser state files (storageState), the CI run may not have a valid state file.
Most likely #1 or #4 based on the symptom. Start there.

When AI gets the diagnosis wrong

AI diagnoses are hypotheses, not verdicts
Test the suggested fix quickly before fully implementing it. A 2-minute test run saves an hour of rabbit-holing.
Ask for multiple possible causes
"What are the 3 most likely causes?" is better than "what's the cause?" — gives you options to eliminate.
Ask "how do I verify this diagnosis?"
A good AI suggestion includes how to confirm it's the right problem, not just the fix.
Check official docs for framework-specific issues
Playwright, Cypress, Jest changelogs and GitHub issues often have the real answer for edge cases.

Your AI-assisted debugging workflow

1
Gather the full context — error output, test code, when it fails, what you've tried
2
Ask AI for ranked hypotheses — "what are the 3 most likely causes given this context?"
3
Test hypothesis #1 quickly — add a log, check an assumption, run a narrow test
4
Feed results back to AI — "hypothesis 1 ruled out because X, now what?"
5
Apply fix, run full suite, commit — verify the fix doesn't break other tests

Key takeaways

1
Context is everything
Full error output + test code + failure conditions = useful diagnosis. Vague questions get vague answers.
2
Treat suggestions as hypotheses
Test the fix quickly before fully implementing it. AI diagnoses are probabilistic, not certain.
3
The four patterns cover most failures
Error investigation, flaky tests, selector issues, and CI/local differences — these cover the majority of test failures.
4
Replace hardcoded sleeps with proper waits
If AI suggests removing waitForTimeout, it's almost always right. Hardcoded sleeps are the #1 cause of flakiness.
