Copilot Agent Mode —
Your First Taste of Autonomous AI

What happens when Copilot can plan, edit multiple files, run commands, and iterate — all from a single goal you give it.

45 min
Duration
Intermediate
Level
Session 05
Prerequisite

From assistant to agent

Copilot Chat (normal mode)
  • → You ask a question
  • → Copilot responds with an answer or code snippet
  • → You copy, paste, and apply manually
  • → You ask the next question
  • → You're always in the driver's seat
Copilot Agent Mode
  • → You give a goal ("add login tests for these user stories")
  • → Copilot reads your codebase to understand patterns
  • → Edits multiple files, runs tests, checks for errors
  • → Loops and self-corrects without you prompting each step
  • → Shows you what it did and asks for confirmation

How to enable Agent Mode

Step 1
Open Copilot Chat
Click the chat icon in the VS Code sidebar, or press Ctrl+Alt+I / ⌃⌥I
Step 2
Switch to Agent Mode
Click the mode dropdown at the top of the chat panel and select Agent
Step 3
Give it a goal
Type your task as a complete goal, not a question. "Add Playwright tests for the login flow" not "how do I write a login test?"
Step 4
Review and approve
Copilot will show you each change before applying. You stay in control — it won't modify files without your approval.
⚠️ Requires paid Copilot plan. Not available on the free tier.

What Agent Mode can actually do

📁
Read your codebase
Explores your project structure, reads existing tests and code to understand patterns before generating anything
✏️
Edit multiple files
Can create new files and modify existing ones across your project in a single task run
⚡
Run terminal commands
Can execute npm commands, run tests, and check their output to verify changes work correctly
🔄
Self-correct on errors
If a test fails or a command errors, it reads the output and adjusts its approach without you prompting it
🧠
Plan multi-step tasks
Breaks a large goal into logical steps, shows you the plan, then executes them in order
✅
Show its work
Logs what it's doing at each step — you can follow along, pause, or redirect at any point

What it cannot do

Myth
"Agent Mode tests are always correct"
Reality
It generates plausible tests. They may have logic errors, incorrect assertions, or miss edge cases. You must review and run them.
Myth
"It knows your app's business rules"
Reality
It can infer patterns from your code, but it doesn't know your requirements documents, user stories, or acceptance criteria unless you provide them.
Myth
"It stays within scope automatically"
Reality
With a vague prompt it may touch files you didn't expect. Be explicit: "only modify files in /tests/login/" to constrain its scope.

You stay in control

You're always in control. Agent Mode never applies changes without your approval.
How the approval flow works
1
Agent shows its plan
Before touching anything, it describes what it intends to do. You can stop here.
2
Each file change is shown as a diff
Green = added, red = removed. You see exactly what changes before they're applied.
3
You approve, reject, or redirect
Accept the change, reject it, or type a correction ("don't use page.goto here, use the loginHelper instead").
4
Agent runs verification (if needed)
It can run npx playwright test itself to check if the generated tests pass.

Demo: from goal to test file

Write Playwright tests for the user login flow. The login page is at /login. Valid credentials are in tests/fixtures/users.json. Follow the same Page Object pattern as in tests/pages/homePage.ts. Cover: successful login, wrong password, empty fields, account locked scenario.
I'll start by reading your existing Page Object structure and test fixtures to match your patterns.

Plan:
1. Read tests/pages/homePage.ts to understand your Page Object pattern
2. Read tests/fixtures/users.json for available credentials
3. Create tests/pages/loginPage.ts (Page Object)
4. Create tests/login.spec.ts (4 test scenarios)
5. Run npx playwright test tests/login.spec.ts to verify

Shall I proceed?
✓ Notice: it reads your project first, creates a Page Object, generates tests, then runs them to verify. All from one prompt.

When to reach for Agent Mode

Good candidates for Agent Mode
  • ✓ Build a whole test file from scratch
  • ✓ Refactor a test suite to use Page Objects
  • ✓ Add tests across multiple spec files for a new feature
  • ✓ Set up a new test framework from zero
  • ✓ Fix multiple failing tests after an API change
Stick with Chat or Completions
  • ✗ Explaining what a piece of code does
  • ✗ Fixing a single assertion in a single test
  • ✗ Generating a quick snippet you'll paste manually
  • ✗ Answering a question about a framework or API
  • ✗ Adding one test case to an existing file

The anatomy of a good agent prompt

Too vague
"Add some tests for the checkout flow"
Structured and clear
Goal: Add Playwright E2E tests for the checkout flow.
Context: Checkout starts at /cart, uses Stripe test cards, ends at /order-confirmation.
Constraints: Only create new files in tests/checkout/. Use the existing helpers/stripe.ts for test card data. Don't modify any existing files.
Verification: Run npx playwright test tests/checkout/ after creating the files.
Rule of thumb: if a new developer joining your team couldn't complete the task from your prompt alone, the agent can't either.

Mistakes people make with Agent Mode

🎯
Giving it a vague goal and expecting precision
Vague prompt = unpredictable output. Be specific about scope, pattern, and constraints.
🔍
Approving file changes without reading them
The diffs are there for a reason. Always read them. Agents can be confidently wrong.
🛡️
Trusting passing tests as proof of correctness
A test that always passes regardless of app behavior is worse than no test. Check assertions, not just exit codes.
📦
Letting it run without scope constraints
Without explicit constraints, it might helpfully "fix" other files it decides need improving. Always set boundaries.

Key takeaways

1
Agent Mode is Copilot with tools
It can read files, write files, run commands, and self-correct — not just generate text.
2
You always approve changes
Nothing is applied without your review. The diff-review workflow keeps you in control.
3
Best for multi-file, multi-step tasks
Use it for building test suites, refactoring, scaffolding — not for quick one-liner fixes.
4
Prompt quality determines output quality
Structured prompts with goal, context, constraints, and verification steps produce far better results.
5
Review the tests, not just the exit code
A passing test that doesn't actually test anything is worse than no test. Always verify assertions.
