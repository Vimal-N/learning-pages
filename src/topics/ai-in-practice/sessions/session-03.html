What is a prompt?

A prompt is everything you send to an LLM — the question, the context you provide, examples, constraints, and the format you want back. The LLM reasons only from what's in the prompt. Nothing else.

THE LLM KNOWS
  • Everything you wrote in this prompt
  • Everything it learned during training
  • The conversation history (in the same session)
THE LLM DOES NOT KNOW
  • Your codebase or project
  • Your team's conventions or standards
  • Previous conversations (in a new session)

The four elements of a strong prompt

1
Context
What situation are you in? What tool, language, framework, or codebase is involved?
2
Task
What specifically do you want the AI to do? Be precise. "Write a test" is not a task. "Write a Playwright test that verifies X" is.
3
Format
How do you want the response? Code only? With explanation? A numbered list? Short or detailed?
4
Constraints
What should it NOT do? What are the limits? "Don't use third-party libraries." "Keep it under 20 lines." "No async/await."
Element 1 of 4

Context — give it your situation

Missing context
"My test is failing. How do I fix it?"
The AI has to guess the framework, language, and error. Its answer will be generic and probably not useful.
Rich context
"I'm using Playwright 1.40 with TypeScript. My test is clicking #submit-btn on the login page, but it times out in CI (GitHub Actions, Ubuntu runner) while passing locally on macOS. Here's the test and the error..."
Framework, language, environment, symptom, and code — the AI can give a targeted, actionable answer.
Element 2 of 4

Task — be specific about what you want

TOO VAGUE
"Write a test"
SPECIFIC
"Write a Playwright TypeScript test that navigates to /checkout, fills the address form, clicks Submit, and asserts the confirmation page URL contains /order-confirmed"
TOO VAGUE
"Explain this code"
SPECIFIC
"Explain what this Playwright beforeEach hook does and why the page.waitForLoadState call is needed"
TOO VAGUE
"Fix this"
SPECIFIC
"This test fails with Error: strict mode violation — getByRole finds 2 elements. Show me how to make the selector more specific."
Element 3 of 4

Format — tell it how to respond

USEFUL FORMAT INSTRUCTIONS
  • → "Give me only the code, no explanation"
  • → "Explain in simple terms, no jargon"
  • → "List the options as bullet points"
  • → "Give me a one-sentence answer"
  • → "Show before and after code side by side"
  • → "Explain each line with inline comments"
CHOOSE BASED ON YOUR GOAL
  • → Just need the code? Ask for code only
  • → Learning something new? Ask for explanation + code
  • → Presenting to team? Ask for a structured summary
  • → Need options? Ask for a comparison list
Tip: If you're copying code into a file, adding "give me only the code" saves you 30 seconds of cleanup every time.
Element 4 of 4

Constraints — tell it what NOT to do

WITHOUT A CONSTRAINT
"Write a test for the API response" → AI picks whatever assertion library it prefers, possibly one you don't use
WITH A CONSTRAINT
"Write a test for the API response using Jest and supertest only — no other libraries. Keep it under 25 lines."
USEFUL CONSTRAINT PHRASES
Don't add new imports Use only built-in methods No async/await Match the existing code style Keep under 30 lines No third-party libraries

A complete prompt — all four elements

I'm writing Playwright TypeScript tests for an e-commerce checkout flow. Write a test that adds a product to the cart, goes through checkout, and asserts the order confirmation number is visible on the success page. Return only the test code with inline comments explaining each step. Do not use any helpers or utilities not already in Playwright — no third-party packages.
CONTEXTPlaywright TypeScript, e-commerce checkout
TASKCart → checkout → assert confirmation
FORMATCode only, inline comments per step
CONSTRAINTSPlaywright built-ins only

The four most common mistakes

01
Too vague
"Fix this code" / "Write a test"
Fix
Name the tool, the behaviour, the expected outcome
02
No context
Assumes the AI knows your setup
Fix
Include framework, language, error message, and relevant code
03
Asking too many things at once
"Fix, refactor, add tests, and explain"
Fix
One clear task per message. Follow up for each additional thing.
04
Not iterating
Accepting a mediocre first response
Fix
Follow up: "Make it shorter." / "Now add error handling." / "That's not right — here's why..."
QA Pattern 1 of 3

Writing test cases with AI

// Prompting template for new test
I'm writing Playwright TypeScript tests.
Page: [URL or component name]
User action: [what the user does]
Expected result: [what should happen]
Write a test. Use getByRole and getByLabel for selectors
where possible. Return code only.
// Filled example
I'm writing Playwright TypeScript tests.
Page: /account/settings
User action: Updates display name and saves
Expected result: Success toast appears, name updates in header
Write a test. Use getByRole and getByLabel for selectors
where possible. Return code only.
QA Pattern 2 of 3

Debugging a failing test

// Debug prompt template
My Playwright test is failing with this error:
[paste full error message here]
Here is the failing test:
[paste test code here]
The test passes locally but fails in CI (GitHub Actions).
What are the most likely causes? List them ranked by probability.
Always paste the full error — not a description of it
Include the test code — not a description of what it does
Mention environment differences (local vs CI, OS, versions)
Ask for ranked hypotheses — gives you a debugging order
QA Pattern 3 of 3

Reviewing your own tests

// Review prompt template
Review this Playwright test and flag any issues.
Look for: brittle selectors, missing assertions,
missing edge cases, anti-patterns, and anything
that would make this test flaky in CI.
[paste your test code here]
Return a numbered list of issues with a severity rating.
Pro tip: At the end add "Also suggest any additional test cases I might have missed for this feature." — you often get 2–3 edge cases you hadn't thought of.

Prompting is a conversation, not a one-shot

TURN 1
Send your initial prompt → get a first response
TURN 2
Refine: "That's close, but the selector should use getByRole. Update it." → improved code
TURN 3
Push further: "Now add a test for the error state when the form is submitted empty." → extended
DONE
You have exactly what you need. Three turns, five minutes.
Useful follow-up phrases: "Make it shorter" · "Add error handling" · "That's wrong — here's why" · "Show me an alternative"
Key Takeaways

What to take from today

1
A prompt is everything the LLM has to work with. The more you give it, the better it can help.
2
The four elements: Context, Task, Format, Constraints. Not all are required — but all improve the result.
3
Specificity is the single biggest lever. Vague in, vague out. Specific in, specific out.
4
Prompting is a conversation. The first response is a starting point — follow up and refine.
5
Three QA patterns to use today: writing tests, debugging failures, reviewing your own code.
