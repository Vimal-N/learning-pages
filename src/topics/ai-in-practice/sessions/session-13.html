Working Effectively

The patterns and practices that separate good results from great results — from task framing to iteration to error handling.

Intermediate

The task description formula

G
Goal — what do you want to end up with?
"Add Playwright E2E tests for the password reset flow"
C
Context — what does the agent need to know?
Framework, existing patterns, which files to read first, relevant business rules
S
Scope — what can and can't it touch?
"Only create new files in tests/auth/. Don't modify existing page objects."
V
Verification — how do you know it worked?
"Run npx playwright test tests/auth/ and confirm all tests pass"
Goal + Context + Scope + Verification = predictable, reliable results

Building the prompt — step by step

Starting idea: "I need tests for the login page"
G
Goal
Add Playwright E2E tests covering the login page (/login) — valid login, invalid credentials, forgot password link.
C
Context
App uses React + TypeScript. Auth API is at /api/auth/login. Read tests/pages/loginPage.ts — follow its Page Object pattern. Selectors use getByTestId() convention.
S
Scope
Create tests/pages/loginPage.ts and tests/e2e/login.spec.ts only. Do not modify any existing files.
V
Verification
Run npx playwright test tests/e2e/login.spec.ts and confirm all 3 tests pass before finishing.
This prompt took ~2 minutes to write. It will save 30+ minutes of bad output and re-prompting.

Vague vs structured — a real comparison

Vague (60 seconds, poor results)
"Write tests for the user profile page"
Missing: which framework? Missing: existing patterns? Missing: scope? Missing: verification?
Structured (2 minutes, reliable results)
Goal: Write Playwright E2E tests for the user profile page (/profile).
Context: The profile page lets users update: display name, email, avatar, password. Changes require re-authentication. Read tests/pages/loginPage.ts to match our Page Object pattern.
Scope: Create tests/pages/profilePage.ts and tests/profile.spec.ts. Don't modify other files.
Scenarios: Successful name update, email change requiring verification, invalid password on re-auth.
Verify: Run npx playwright test tests/profile.spec.ts after creating the files.

What happens after you press Enter

1
Exploration — agent reads your project structure
Lists directories, reads CLAUDE.md if present, identifies relevant files from your task description
2
Context building — reads files you mentioned and related ones
Reads the Page Object you referenced, the test it's similar to, the config file — building understanding before writing
3
Planning — agent proposes what it will do
Often announces its plan before executing: "I'll create X, Y, Z files with the following structure..."
4
Permission prompts — asks before editing or running commands
Shows you the exact change or command. You approve each one: y (yes), n (no), a (yes to all of this type)
5
Verification — runs the command you specified
Runs your test command, reads the output, fixes failures, reports the final result — or asks you for help if it can't fix something
The exploration phase (steps 1-3) is where the quality of your prompt — and your CLAUDE.md — makes the biggest difference.

What to check when the agent finishes

Green flags — agent working well
Agent asked a clarifying question before starting
Described its plan before writing any files
Only created/modified files in the scope you specified
Ran the verification command you provided
Code matches your existing patterns and conventions
All tests pass without modifications to existing tests
Red flags — review carefully
Modified files you didn't mention or expect
Skipped the verification step entirely
Code has // TODO: implement placeholders
Changed assertions in existing tests to make them pass
Selector patterns don't match your conventions
Tests pass by skipping or marking as todo
Always read the diff before accepting changes. "Tests pass" is necessary but not sufficient — the agent is your collaborator, not your approver.

Managing context in long sessions

Common assumption
"The agent remembers everything I've told it"
Reality
Terminal agent sessions have context limits. In very long sessions, the agent may lose track of instructions from early in the conversation. Symptoms: it ignores constraints you set at the start, or generates code that contradicts earlier decisions.
Context management strategies
  • · One task per session — don't chain unrelated tasks in the same session
  • · Restate key constraints at the start of each new session
  • · Use CLAUDE.md files to give the agent persistent context about your project
  • · For multi-stage tasks: "Summarise what was done before I start the next stage"

Breaking large tasks into stages

One huge task (unpredictable)
"Set up our entire Playwright test framework with Page Objects, fixtures, CI config, and write tests for all 12 features"
Same work, staged (controllable)
1 Session 1: Set up folder structure, install dependencies, configure playwright.config.ts. Verify: npx playwright --version runs cleanly.
2 Session 2: Create base Page Object classes and auth fixtures. Verify: fixtures load without error.
3 Sessions 3–N: One feature's tests per session, each verified independently before moving on.

Iterating on agent output

Redirect — change approach
"Stop using class selectors. Use getByRole() and getByTestId() instead. Redo the selectors in the Page Object you just created."
Refine — adjust the output
"Good start. Now add a test for the 'forgot password' link and rename all test descriptions to use present tense."
Explain before changing
"Why did you use page.waitForLoadState('networkidle') here? Is there a better alternative for this case?"

When things go wrong

Agent is looping without progress
Stop the session. Start fresh with a more constrained task description. Break the task into a smaller first step.
Generated code has errors
Paste the error output back into the chat: "Running the code produced this error: [paste]. Fix it." Give it the information it needs.
Agent went in a completely wrong direction
Use git to restore the previous state (git restore .), then restart with a clearer description. This is why you commit first.
Agent ignores your constraints
Restate them explicitly at the top of the next message: "IMPORTANT: Only create new files. Do not modify any existing files." Repetition works.

Commit before every agent session

Your workflow every time
1.
Commit your current work
git add -A && git commit -m "pre-agent checkpoint"
2.
Run the agent session
Review all permission prompts before approving
3a.
If the result is good — commit it
git commit -am "agent: add login tests"
3b.
If the result is wrong — restore instantly
git restore .
Why this matters
· A bad session is recoverable in 3 seconds
· Clean diff = easy to review exactly what the agent did
· Commit history shows which changes were agent-generated
· You can partial-accept: keep some changes, discard others
Treat every agent session like a speculative experiment — your checkpoint makes it safe to experiment freely.

Project context files — permanent briefs

CLAUDE.md — what to include
# Project: Checkout E2E Test Suite
## Framework: Playwright 1.44, TypeScript
## Test runner: npm test
## Page Objects in: tests/pages/
## Fixtures in: tests/fixtures/
## Selector convention: getByTestId first, getByRole second
## NEVER use: waitForTimeout, class selectors
## CI: GitHub Actions, headless Chrome only
CLAUDE.md is committed to git — the whole team benefits from it. Every session starts with the agent already knowing your project's conventions.

What makes a CLAUDE.md useful

Unhelpful — too vague
# Project
This is our test project.
We use Playwright.
Follow best practices.
Write clean code.
The agent already knows to follow best practices. This adds nothing.
Useful — specific and actionable
# Project: Checkout E2E Suite
## Stack: Playwright 1.44, TypeScript 5  ← exact versions
## Run: npm test  ← agent can self-verify
## Pages: tests/pages/ (one class/page)  ← structure
## Selectors: getByTestId() first  ← convention
## NEVER: waitForTimeout()  ← forbidden
## NEVER: class-based selectors  ← forbidden
## CI: headless Chromium only  ← env constraint
Rule: if you've had to tell the agent the same thing twice, it belongs in CLAUDE.md.

Key takeaways

1
Use the GCSV formula for every task
Goal + Context + Scope + Verification. Two extra minutes of task description saves hours of bad output.
2
Commit before every agent session
A git checkpoint makes every session reversible in seconds. Enables confident experimentation.
3
Read the output — you're the reviewer
Check for red flags: unexpected file changes, skipped verification, placeholder TODO comments, modified assertions.
4
Invest in your CLAUDE.md
A good project context file means every session starts with the agent already knowing your conventions. If you've told it something twice, it belongs in the file.
