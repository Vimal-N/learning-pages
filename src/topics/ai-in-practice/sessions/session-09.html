Prerequisite

Test code is production code

💸
Duplicated selectors
A button ID changes → update 40 tests. Or: extract to a Page Object once, update one line.
🔍
Unreadable test names
"test_1234" failing in CI tells you nothing. "should show error when email already exists" tells you exactly where to look.
🔗
Tests sharing state
Test B depends on data Test A created. Test A fails → Test B fails → Test C fails. The actual bug gets buried.
📦
3000-line test files
No grouping, no hierarchy, no structure. Finding the failing test is a job in itself.

Pattern 1 — Extract Page Objects

Here's my login.spec.ts test file. Extract a LoginPage Page Object class that encapsulates all the selectors and interactions. Create it at tests/pages/loginPage.ts. Then update the test file to use it. TypeScript, Playwright style.
I can see 6 distinct selectors and 4 common action sequences across your tests. I'll create a LoginPage class with:
  • Properties: emailInput, passwordInput, submitButton, errorMessage
  • Methods: goto(), login(email, password), getErrorMessage()
Before
await page.fill('#email', 'user@test.com');
await page.fill('#password', 'pass');
await page.click('button.submit');
After
const login = new LoginPage(page);
await login.goto();
await login.login('user@test.com', 'pass');

Pattern 3 — Improve readability

Hard to read
// test1
test('login', async () => {{ '{' }}...{{ '}' }});

// test2
test('login_error', async () => {{ '{' }}...{{ '}' }});

// test3
test('locked', async () => {{ '{' }}...{{ '}' }});
Self-documenting
describe('Login page', () => {{ '{' }}
  test('logs in with valid credentials',
    async () => {{ '{' }}...{{ '}' }});
  test('shows error for wrong password',
    async () => {{ '{' }}...{{ '}' }});
  test('blocks locked accounts',
    async () => {{ '{' }}...{{ '}' }});
{{ '}' }});
Quick win prompt: "Rename all test() calls in this file to describe what the test verifies, not what it does. Use present tense: 'shows error for wrong password' not 'test wrong password'."

Refactoring safely

✅
Run the suite before starting
Establish the baseline. Know how many tests pass. If any are already failing, fix those before refactoring.
📦
Refactor one pattern at a time
Extract Page Objects → run suite. Extract helpers → run suite. Don't batch all refactoring into one step.
🔍
Read the diff before applying AI changes
AI may subtly change assertions while refactoring. Read the diff the same way you'd review a PR.
✅
Run the suite after each step
Same number of passes as before = safe refactor. Any new failures = investigate immediately before continuing.

Key takeaways

1
Test code quality pays dividends every day
A well-structured test suite is faster to maintain, easier to debug, and builds more team confidence in CI results.
2
Start with an AI code review
Point AI at your test files and ask for a prioritised list of issues. It spots patterns you've gone blind to.
3
Four high-impact refactoring patterns
Page Object extraction, helper extraction, readability improvements, and parameterised tests eliminate most test maintenance debt.
4
Run the suite before and after each change
Refactoring test code should not change what the tests verify — only how they're written. The test results should be identical.
