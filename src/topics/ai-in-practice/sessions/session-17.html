Two ways to register an MCP server

Option 1 — CLI (easiest)
claude mcp add <name> <command> [args...]
Claude Code updates the config file automatically. Best for quick setups and experimenting.
# Real example — add the filesystem server
claude mcp add filesystem npx -y @modelcontextprotocol/server-filesystem /Users/you/projects
Option 2 — JSON config file
Edit the config directly. Stored at ~/.claude.json. Good for teams — commit it to your repo.
Both options do the same thing. The CLI is just a shortcut for editing the JSON file.

The JSON config — every line explained

{{ '{' }} // ~/.claude.json
  "mcpServers": {{ '{' }}

    "filesystem": {{ '{' }} // ← A
      "command": "npx", // ← B
      "args": [ // ← C
        "-y", // ← D
        "@modelcontextprotocol/server-filesystem", // ← E
        "/Users/you/projects" // ← F
      ]
    {{ '}' }}
  {{ '}' }}
{{ '}' }}
A The name you give this server. Anything descriptive. You'll see this in Claude's tool list.
B The command Claude uses to start the server process. "npx" = run an npm package without installing it first.
C Arguments passed to that command, as a JSON array. Each word/value is a separate item.
D The "-y" flag tells npx to automatically say "yes" to install prompts — no confirmation needed.
E The npm package name of the MCP server to run. This is the community filesystem server from Anthropic.
F The path on your computer the server is allowed to access. The AI can only read/write inside this folder.

What happens when Claude Code starts up

Step 1 — Read config
Claude reads ~/.claude.json and finds the list of registered MCP servers
Step 2 — Launch servers
Claude starts each server as a background process using the command and args from the config
Step 3 — Discovery
Claude sends a tools/list request to each server. Each server responds with its capabilities.
Step 4 — Awareness
Claude now knows: "I have a read_file tool, a write_file tool, a list_directory tool..." — and can use them in any conversation
In practice
This all happens in milliseconds at startup. You never see it. When you type a question, Claude already knows what tools it has available.

Demo — Setting up the Filesystem MCP server

Step 1 — Register the server
# Add the filesystem MCP server, scoped to your tests folder
claude mcp add filesystem npx -y @modelcontextprotocol/server-filesystem ./tests
Replace ./tests with any folder path. The AI will only be able to access files inside that path.
Step 2 — Restart Claude Code
Type /exit and reopen Claude Code. The server starts fresh on every session.
Step 3 — Verify it's connected
claude mcp list
You should see filesystem listed. You can also ask Claude: "What tools do you have?" and it will list read_file, write_file, list_directory.

Demo — Setting up the GitHub MCP server

Step 1 — Get a GitHub token
GitHub → Settings → Developer Settings → Personal Access Tokens → Tokens (classic) → Generate new. Select repo scope.
Step 2 — Register the server with your token
# Register and pass the token as an environment variable
GITHUB_TOKEN=ghp_yourtoken claude mcp add github npx -y @modelcontextprotocol/server-github
Alternative — in the JSON config (team-friendly)
"github": {{ '{' }}
  "command": "npx",
  "args": ["-y", "@modelcontextprotocol/server-github"],
  "env": {{ '{' }} "GITHUB_TOKEN": "ghp_yourtoken" {{ '}' }}
{{ '}' }}

QA workflows MCP transforms

🔄
Post-CI triage
After a CI run: AI reads results (filesystem MCP) → checks what changed (GitHub MCP) → creates Jira tickets for new failures (Jira MCP) → posts summary to Slack (Slack MCP). One prompt, four systems.
📋
Release readiness check
AI reads all merged PRs for this release (GitHub), checks open bugs assigned to this sprint (Jira), reads test coverage summary (filesystem) — returns a "ready to release?" report.
🔍
Exploratory testing sessions
Point Playwright MCP at a new feature, ask AI to explore it with a test charter. AI navigates, clicks, screenshots, and reports issues — faster first-pass than manual testing alone.
📊
Realistic test data generation
Database MCP gives AI access to real production-like records — generate tests using actual product IDs, real user edge cases, and valid data combinations from your system.

When MCP connections fail — what to check

🔄
Server not appearing — check restart first
Always restart Claude Code after adding a server. Run claude mcp list to confirm it's registered. This fixes 80% of connection problems.
🔑
Authentication errors
Check environment variables are set correctly. Run echo $GITHUB_TOKEN to verify. The token may have expired — generate a new one.
📦
npx can't find the package
Always use the -y flag with npx. Or pre-install: npm install -g @modelcontextprotocol/server-filesystem
📝
JSON syntax error in config
Open ~/.claude.json in VS Code — it highlights JSON errors. A missing comma or bracket will silently break all servers.
🛡️
Permission denied errors
Filesystem server needs access to the paths you specified. Check the path exists and you have read permissions: ls -la ./tests

Key takeaways

1
Configuration is straightforward — one command or a JSON block
One CLI command or a JSON config entry. Most community servers are npx packages — no separate install needed. The hardest part is usually the restart.
2
Start with filesystem, then add more as you need them
The filesystem server is the lowest friction starting point and immediately useful. Add GitHub, Playwright, and others when you identify a specific workflow that needs them.
3
Playwright MCP = AI-powered exploratory testing
An AI with browser control can explore new features, find UI bugs, and report issues — it's genuinely useful for first-pass coverage of features your automated tests haven't reached yet.
4
Multiple servers unlock end-to-end QA workflows
Test results → AI analysis → ticket creation → Slack notification. MCP connects the steps without manual copy-pasting between systems. Session 15 shows you how to build custom ones.
