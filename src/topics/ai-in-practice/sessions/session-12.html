Why do you have to explain the same things every session?

Session A — Monday
"Before we start: we use Playwright with TypeScript, page object pattern, tests in /tests/e2e/, utilities in /tests/utils/, we never use waitForTimeout, we use waitForLoadState('networkidle') instead..."
Session B — Tuesday
"Before we start: we use Playwright with TypeScript, page object pattern, tests in /tests/e2e/, utilities in /tests/utils/..."
You wrote this exact same thing yesterday. And the day before.
Context files solve this. Write your project knowledge ONCE. Every agent reads it automatically.

One purpose, three files — one for each tool

CLAUDE.md — the pre-brief for Claude Code

What it IS
→ Markdown file named exactly CLAUDE.md
→ Lives in your project root (or subdirectories for sub-project scope)
→ Read automatically by Claude Code at the start of every session
→ Can also be placed in ~/.claude/CLAUDE.md for user-level preferences
What goes IN it
• Tech stack and versions ("We use Playwright 1.48, TypeScript 5.4")
• Project structure ("Tests live in /tests/e2e, utilities in /tests/utils")
• Coding conventions ("Never use waitForTimeout. Always await locators before asserting")
• What NOT to do ("Don't install new packages without asking")
• How to run things ("npm run test:e2e runs all tests")

copilot-instructions.md — persistent instructions for GitHub Copilot

Path: .github/copilot-instructions.md  —  this exact path
Effect on completions
Completions now follow your patterns
Copilot learns your variable naming conventions
Suggestions use your existing helper functions
Won't suggest deprecated patterns you've explicitly banned
Effect on Copilot Chat & Agent Mode
@workspace answers filtered by your instructions
Chat references your actual tech stack
Agent Mode follows your architectural decisions
Consistent suggestions across all team members
Team effect: This file applies to ALL Copilot features — completions, chat, and agent mode. It affects your whole team when committed to the repo.

AGENTS.md — the multi-agent standard

Works with Opencode, Claude Code, and any agent that follows the emerging AGENTS.md convention.
What it is
A growing convention (not a single vendor standard)
Placed in project root as AGENTS.md
Supported by Opencode, and growing in Claude Code support
Designed to be vendor-neutral — write once, works anywhere
When to use it
Your team uses multiple agent tools
You want one source of truth for agent context
Building CI/CD workflows that invoke agents
You want future-proofing as new tools emerge
Relationship to other files
Can coexist with CLAUDE.md and copilot-instructions.md
Prefer AGENTS.md content for cross-tool information
Keep tool-specific guidance in tool-specific files

Where you put the file determines what it affects

What a well-structured context file looks like

# CLAUDE.md — QA Test Suite

## Tech Stack                   A: Versions matter, not just names
- Playwright 1.48 (TypeScript)
- Tests: /tests/e2e/**/*.spec.ts

## Run Commands               B: Agent can self-verify
- All tests: npm run test:e2e
- Debug mode: npx playwright test --debug

## Conventions                 C: The rules that matter
- NEVER use waitForTimeout() — use waitForLoadState('networkidle')
- Selectors: prefer data-testid attributes over CSS selectors

## Do NOT                     D: Explicit forbidden actions
- Don't install npm packages without confirming
- Don't edit files in /src — that's the application code
A — Stack: Include exact versions. "Playwright 1.48" beats "Playwright."
B — Commands: Agent can run and verify tests without guessing the command.
C — Conventions: Specific, actionable rules. Write the comment you'd leave in a PR review.
D — Do Not: Explicit prohibitions prevent the most costly mistakes.

Every section you can include — and what each one does

## Tech Stack High
Framework names, exact versions, language. Agent picks correct APIs and avoids deprecated syntax.
## Directory Structure High
Where tests, page objects, utilities, fixtures, and config files live. Agent creates files in the right place.
## Run Commands High
Exact commands to run tests, build, lint. Without this the agent guesses — usually wrong.
## Coding Conventions High
Selectors, naming patterns, wait strategies, assertion style. Shapes every line of code the agent writes.
## Do Not High
Explicit prohibitions. What the agent must never do. Most impactful section for preventing costly mistakes.
## CI / Environment Medium
CI-specific flags, env vars, headless mode, retry settings. Prevents "works locally, breaks in CI."
## Security Notes Medium
Which files are sensitive, which env vars hold secrets, what must never be committed. Keeps credentials safe.
## Project Context Situational
Non-obvious decisions, known constraints, why certain patterns exist. Helps agent respect intentional design choices.
## Common Tasks Situational
Shortcuts for things you ask the agent to do repeatedly — test generation patterns, migration steps, PR checklists.

A complete CLAUDE.md — every section explained

# CLAUDE.md — My QA Project

## Tech Stack ← A
- Playwright 1.48, TypeScript 5.4, Node 20
- Test runner: npm run test:e2e
- Lint: npm run lint

## Directory Structure ← B
- E2E specs: /tests/e2e/**/*.spec.ts
- Page objects: /tests/pages/[Name]Page.ts
- Utilities: /tests/utils/

## Coding Conventions ← C
- NEVER use waitForTimeout()
- Always: waitForLoadState('networkidle')
- Selectors: data-testid first, getByRole() second
- Tests must be independent — no shared state

## Do Not ← D
- Never edit /src or /app directories
- Never install packages without confirming
- Never hardcode URLs; use BASE_URL env var

## CI Notes ← E
- CI sets HEADLESS=true automatically
- Max retries: 2 (set in playwright.config.ts)
- Never commit .env or credential files
A — Tech Stack
Factual context. Exact versions prevent the agent from using deprecated APIs. The run commands tell it how to verify its own work.
B — Directory Structure
Ground truth for navigation. Without this the agent might create login.test.ts in the root instead of /tests/e2e/.
C — Coding Conventions
Behavioural rules. Applied as constraints on every line of code. The agent steers toward waitForLoadState automatically even when it wasn't asked explicitly.
D — Do Not
Hard prohibitions. The model treats these as inviolable limits — it won't touch /src even if the task description seems to imply it should.
E — CI Notes
Environment context. Prevents "works locally but fails CI" surprises. The agent knows CI is headless and respects your retry config.

What happens when the agent has NO context file

Same prompt: "Add a test for the checkout flow. It should verify that the order summary page appears after clicking Pay Now."
Without CLAUDE.md
Uses page.waitForTimeout(2000) ← forbidden pattern
Creates test in root folder ← wrong location
Uses CSS selectors like .pay-button ← wrong style
No page object pattern
With CLAUDE.md
Uses waitForLoadState('networkidle') ← correct
Creates test in /tests/e2e/checkout.spec.ts
Uses data-testid="pay-button" ← correct
Follows page object pattern correctly
Same prompt. Same model. Completely different quality of output.

What happens with multi-file agent tasks

Prompt: "Find all tests that use waitForTimeout and update them to use the correct Playwright wait pattern. Run the tests after updating to make sure nothing broke."
Without CLAUDE.md — 3 problems
1. Doesn't know what "correct wait pattern" means for your project — defaults to page.waitForSelector()
2. Doesn't know where tests are — searches entire repo including node_modules
3. Doesn't know how to run tests — guesses npm test which fails
With CLAUDE.md — 3 successes
1. Replaces with waitForLoadState('networkidle') — your exact convention
2. Searches only /tests/e2e/**/*.spec.ts — the right location
3. Runs npm run test:e2e — the correct command
15 wrong changes and a failed CI build
27 correct replacements, all tests passing

A real, usable CLAUDE.md template for your test suite

# CLAUDE.md — [Project Name] QA Test Suite

## Stack
- Framework: Playwright [version] with TypeScript
- Test runner: npm run test:e2e (all), npm run test:unit (unit only)

## Directory Structure
- E2E tests: /tests/e2e/**/*.spec.ts
- Page objects: /tests/pages/[PageName]Page.ts
- Utilities: /tests/utils/   Fixtures: /tests/fixtures/

## Coding Rules
- Never use waitForTimeout() — always waitForLoadState('networkidle') or explicit locator waits
- Use data-testid attributes for selectors, not CSS class names
- Each test must be independent — no shared state, no test order dependencies

## Do Not
- Do not edit files in /src or /app — that's application code, not test code
- Do not install npm packages without asking first
- Do not hardcode URLs — use BASE_URL from .env
- Do not commit .env files or files containing real credentials

## CI Notes
- CI uses HEADLESS=true (already set in playwright.config.ts)
- Retries set to 2 — flaky tests should be fixed, not retried away

A real copilot-instructions.md template

# Copilot Instructions — [Project] Test Automation

## Language and Framework
This project uses Playwright with TypeScript for end-to-end testing.
Always use async/await. Never use callbacks or .then() chains.
TypeScript strict mode is enabled — avoid `any` types.

## Test Structure
- Tests are in /tests/e2e/ as *.spec.ts files
- Use the Page Object Model — page objects live in /tests/pages/
- Use Playwright fixtures for setup/teardown where possible

## Selectors
Always prefer data-testid attributes: page.getByTestId('submit-button')
Use getByRole() as second choice: page.getByRole('button', {{ '{' }} name: 'Submit' {{ '}' }})

## Waits
Never suggest waitForTimeout(). Instead:
- For navigation: await page.waitForLoadState('networkidle')
- For element: await expect(locator).toBeVisible()
- For API: await page.waitForResponse('**/api/endpoint')

## Code Style
- Descriptive test names: test('should display error when login fails', ...)
- Group related tests with test.describe() blocks

What NOT to do — and why it doesn't work

Mistake 1
Writing vague guidance
"Write clean, readable code with good test coverage"
Why it fails: Agent can't act on "clean." It needs specific, measurable instructions.
Fix
"Tests must have descriptive names starting with 'should'. Coverage threshold is 80%."
Mistake 2
Making it a novel
"Here is the complete history of this project..."
Why it fails: Context files compete with your task for the model's attention window. Shorter = better signal.
Fix
Keep to 50-100 lines. Every line should be something the agent will act on.
Mistake 3
Putting secrets in context files
DATABASE_URL=postgres://user:pass@host/db
Why it fails: CLAUDE.md is committed to git. Secrets in git = security incident.
Fix
Reference env vars by name only: "Database URL is in DATABASE_URL env var."
Mistake 4
Never updating it
Writing CLAUDE.md once and leaving it for 6 months
Why it fails: Project conventions change. A stale CLAUDE.md teaches the agent the wrong things.
Fix
Review and update every 2-4 weeks, or whenever you adopt a new convention.

Key takeaways

1
Context files eliminate the Groundhog Day problem
CLAUDE.md, copilot-instructions.md, and AGENTS.md teach your agent about your project once — so you never have to explain it again. Every session starts pre-briefed.
2
Placement determines scope
User-level (~/.claude/CLAUDE.md) for personal preferences. Project-level for team conventions. Subdirectory for scope-specific rules in large monorepos.
3
The before/after difference is dramatic
Same prompt + same model + context file = output that fits your codebase immediately, in the right location, following your patterns, with your naming conventions.
4
CLAUDE.md is living documentation — keep it evolving
Commit it to git, keep it short, keep it specific, and update it as your conventions evolve. If you write the same PR review comment three times, it belongs in CLAUDE.md.
