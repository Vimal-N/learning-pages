Concept 1

What AI actually means

What people imagine
  • 🤖 Robots with human-like consciousness
  • 🧠 A digital brain that "thinks" like us
  • 🎬 Skynet from the movies
  • ✨ Magic that just "knows" things
What it actually is
  • 💻 Software trained to recognise patterns
  • 📊 Very good at tasks humans find tedious
  • 🔧 A tool — powerful but with real limits
  • 📚 Learned from human-generated data

Simple definition: A computer system designed to do tasks that would normally require human intelligence — like reading, writing, recognising images, or understanding language.

Concept 2

AI vs ML vs LLM

These three terms get used interchangeably. They shouldn't be.

AIArtificial Intelligence

The broad umbrella. Any system doing tasks that normally need human intelligence. Includes AI from the 1950s onwards. Your spam filter is AI.

MLMachine Learning

A type of AI where the system learns from data rather than following hand-written rules. Netflix recommendations are ML.

LLMLarge Language Model

A specific type of ML model trained on massive amounts of text. ChatGPT, Claude, Gemini — these are all LLMs. This is what we focus on.

Concept 3

How we got here

1950s
Rule-based AI
Hand-written rules. If X, then Y. Chess programs. Expert systems.
1990s
Machine Learning takes off
Systems learn from data instead of rules. Spam filters, recommendation engines.
2017
The Transformer breakthrough
A new ML architecture published in a research paper. Quietly, everything changed.
Nov 2022
ChatGPT launches 🚀
100 million users in 2 months. AI became accessible to everyone overnight.
Concept 4

November 2022

Why ChatGPT changed everything — and why it felt different

What existed before
  • • AI tools required technical expertise
  • • APIs, code, and configuration needed
  • • Output was narrow and specialised
What ChatGPT did
  • • You just... type in plain English
  • • No setup, no code, no expertise needed
  • • 100 million users in 2 months

The technology wasn't brand new. The accessibility was. That's what changed everything.

Concept 5

What is an LLM?

📚 Step 1: Read almost everything humans have ever written
Books, websites, code, articles, forums, documentation — billions of documents. This is called training.
🔗 Step 2: Learn patterns in language
Not memorising text — learning which words, ideas, and structures tend to appear together. Like learning grammar by reading, not by memorising rules.
✨ Step 3: Predict what should come next — very, very well
When you send a message, the LLM uses those patterns to predict the most appropriate next word. Then the next. Then the next. That's a response.
Concept 6

How an LLM generates a response

No magic. Just very sophisticated pattern prediction — one word at a time.

You type:
"Explain why this Playwright test is flaky"
LLM predicts, word by word:
Flaky tests typically occur due to timing issues...

The LLM never "looks up" an answer. It generates one — word by word — using patterns from its training.

Concept 8

What "prompting" means

A prompt is simply what you type when you talk to an LLM. How you write it determines what you get back.

Vague prompt
"Fix my test."
LLM has to guess what the test does, what the error is, what stack you're on, and what "fix" means.
Specific prompt
"This Playwright test for the login flow fails intermittently with a timeout on the submit button. Here's the test code: [paste]. What could cause this?"
LLM has context, technology, error type, and code — and can give a precise, useful answer.
Concept 9

Prompting: what it looks like

You
My Playwright test keeps timing out on the login button click. The test works fine locally but fails in CI. Here's the selector I'm using: await page.click('#submit-btn'). What could be causing this in CI specifically?
AI
CI environments often have slower CPU and network. A few things to check:

1. Add an explicit wait — try waitForSelector before the click
2. Race condition — the button may not be interactive yet even if visible
3. Increase timeout in playwright.config for CI specifically

Specific context → specific, actionable answer. This is the core skill.

Why This Matters for QA

What this means for your work

Writing tests — Describe the feature, get a test scaffold. Paste an API response, get assertions. Goes from blank page to starting point in seconds.
Debugging failures — Paste the error, paste the test, ask what's wrong. Get a reasoned hypothesis in seconds instead of minutes of guessing.
Understanding unfamiliar code — Paste a function and ask "what does this do?". Useful when testing code you didn't write.
Reviewing test coverage — Ask "what edge cases am I missing?" on a test suite. Gets a second perspective fast.
Clearing the Air · 1 of 4

Myth: AI is sentient or conscious

The myth

"AI has feelings. It can get bored, offended, or creative. It's basically digital consciousness."

The reality
  • • LLMs predict text — they do not think, feel, or experience anything
  • • When an LLM says "I think..." it's mimicking how humans write — not reporting an opinion
  • • There is no "it" in the room with you — you're running a very sophisticated pattern-match
  • • This matters for setting expectations — don't anthropomorphise the tool
Clearing the Air · 2 of 4

Myth: AI is always right

The myth

"If an AI says it with confidence, it must be correct. It has access to more information than me."

The reality
  • • LLMs hallucinate — they generate confident-sounding wrong answers
  • • They can invent library methods, cite non-existent docs, get logic wrong
  • • They don't know they're wrong — there's no internal "confidence score"
  • • Rule: always verify AI output before using it in real work
Clearing the Air · 3 of 4

Myth: AI knows my context

The myth

"The AI knows what project I'm working on, what I meant last time, and what my team's conventions are."

The reality
  • • The LLM only sees what you type in the current conversation
  • • It has no access to your codebase, Jira, or previous chats unless you paste them
  • • The more context you give, the better the output — always
  • • Tools like Claude Code change this by reading your files — we cover that in Session 10
Clearing the Air · 4 of 4

Myth: AI will replace engineers

The myth

"AI is going to automate QA engineering out of existence. Why learn this if it's taking our jobs?"

The honest reality
  • • AI is automating tasks, not roles — specifically the tedious, repetitive ones
  • • What remains: judgment, critical thinking, understanding requirements, strategy
  • • QA engineers who use AI tools effectively become significantly more productive
  • • This series is about being the engineer who knows how to use the tools

Real-World Scenario

The situation

A Playwright test that passes locally, but intermittently fails in CI with a timeout. You've been staring at it for 20 minutes.

The failing test
test('user can log in', async ({ page }) => {
  await page.goto('/login');
  await page.fill('#email', 'user@test.com');
  await page.fill('#password', 'pass123');
  await page.click('#submit-btn');  // ← Times out here in CI
  await expect(page).toHaveURL('/dashboard');
});
TimeoutError: page.click: Timeout 30000ms exceeded
Real-World Scenario · Continued

Asking AI — and what you get

You
My Playwright test clicks #submit-btn and times out in CI but passes locally. The button exists and is visible. Using Playwright 1.40, Node CI runner. Could it be a race condition?
AI
Very likely a race condition. In CI the CPU is slower, so the button might be visible but not yet interactive (event listeners not attached). Try:
  1. Replace click() with locator click — includes auto-waiting for the element to be actionable
  2. Add waitForLoadState('networkidle') before the click to ensure the page is fully settled
  3. Explicitly wait for the button to be enabled:
    await page.locator('#submit-btn').waitFor({state:'enabled'})

20 minutes of staring → 30 seconds of asking. That's the value.

Key Takeaways

Five things to leave with today

1
AI is a broad category. LLMs (like ChatGPT and Claude) are the specific type we care about — trained on text, generating text.
2
LLMs don't think or understand — they predict what text should come next based on patterns. Very powerful, very different from human intelligence.
3
The way you prompt determines the quality of the output. Vague in, vague out. Specific and contextual in, useful out.
4
AI can be confidently wrong. Always verify output before using it in real work — especially code and API references.
5
For QA engineers: AI is most useful for writing test scaffolds, debugging failures, and understanding unfamiliar code.
