The right mindset

Wrong mental model
"Copilot is smart autocomplete. I'll use it to save a few keystrokes."
  • ✗ Only use completions, ignore Chat
  • ✗ Accept first suggestion without reviewing
  • ✗ Give it vague context and expect magic
  • ✗ Use it only for typing speed, not thinking
Right mental model
"Copilot is a pair programmer. I set direction, it executes. I review, it refines."
  • ✓ Use both Completions and Chat for different tasks
  • ✓ Always review and test generated code
  • ✓ Give it rich context — framework, pattern, constraints
  • ✓ Use it to think through problems, not just type faster

Pattern 1

Writing a test from scratch

STEP 1
Start with a descriptive comment or test name that states the behaviour
// Test: logged-in user can update their email in account settings
// - navigates to /account/settings
// - changes email to a new valid address
// - saves and verifies success message
test('user can update email in account settings', async ({ page }) => {
  // Copilot generates the rest — Tab to accept
  await page.goto('/account/settings');
  await page.getByLabel('Email').fill('new@example.com');
  await page.getByRole('button', { name: 'Save' }).click();
  await expect(page.getByText('Settings saved')).toBeVisible();
});
Always verify: the label text, button name, and success message text against the actual UI. Copilot guesses — you confirm.
Pattern 2

Adding to an existing test suite

WHY IT WORKS Copilot reads the existing tests in the file and automatically matches your selector strategy, assertion style, test structure, and imports — no setup needed. 
// Existing tests in the file use this pattern:
test('user can add item to cart', async ({ page }) => { /* ... */ });
test('user can remove item from cart', async ({ page }) => { /* ... */ });

// You type just the test name — Copilot sees the pattern and continues it:
test('user can update item quantity in cart', // ← Copilot generates the rest
  async ({ page }) => {
    await page.goto('/cart');
    await page.getByLabel('Quantity').fill('3');
    await page.getByRole('button', { name: 'Update' }).click();
    await expect(page.getByText('Cart updated')).toBeVisible();
});

Generating test data

GENERATED RESULT

Pattern 4

Understanding unfamiliar code

YOU SELECT THIS CODE
async function waitForNetworkIdle(
  page: Page,
  timeout = 5000
) {
  await Promise.race([
    page.waitForLoadState('networkidle'),
    new Promise((_, reject) =>
      setTimeout(() => reject(
        new Error('Network idle timeout')
      ), timeout)
    )
  ]);
}
COPILOT EXPLAINS
This function waits for the network to go idle (no active requests) with a custom timeout. It races two promises: Playwright's built-in waitForLoadState('networkidle') against a timer that rejects after timeout ms. If the network doesn't settle in time, it throws a descriptive error instead of hanging. Used to ensure the page is fully loaded before making assertions.
HOW TO TRIGGER
Select code → right-click → Copilot: Explain Chat: /explain Chat: "What does this function do?"
Pattern 5

Debugging a failing test

THE WORKFLOW
  1. Copy the full error message from the terminal
  2. Copy the failing test code
  3. Open Copilot Chat
  4. Paste both and ask: "What are the most likely causes of this error? List them ranked by probability."
  5. Work through the suggestions from most to least likely
You
My Playwright test is failing with:
Error: strict mode violation: getByRole('button', {name: 'Save'}) resolved to 2 elements

What are the most likely causes?
Copilot
Most likely causes, ranked:
  1. Two buttons with label 'Save' are visible at the same time (e.g., in a form and a modal). Make the selector more specific with .locator('[data-testid="save-settings"]')
  2. An invisible Save button from a hidden form is still in the DOM. Filter with .filter({ visible: true })
Pattern 6

Refactoring repetitive test patterns

Before — copy-paste in every test
test('test A', async ({ page }) => {
  await page.goto('/login');
  await page.fill('#email', 'user@test.com');
  await page.fill('#pass', 'pass123');
  await page.click('#submit');
  // ... actual test ...
});

test('test B', async ({ page }) => {
  await page.goto('/login');
  await page.fill('#email', 'user@test.com');
  await page.fill('#pass', 'pass123');
  await page.click('#submit');
  // ... actual test ...
});
After — Copilot extracts a helper
async function loginAs(page: Page, email: string) {
  await page.goto('/login');
  await page.fill('#email', email);
  await page.fill('#pass', 'pass123');
  await page.click('#submit');
}

test('test A', async ({ page }) => {
  await loginAs(page, 'user@test.com');
  // ... actual test ...
});
Chat: "Select repetitive code → Extract this login setup into a reusable helper function"
Pattern 7

Inline completions — best practices

✓
Write a comment before the function you want generated
A one-line comment like // navigate to checkout and assert order total matches cart is a direct brief to Copilot.
✓
Use Alt+] / Option+] to cycle through alternatives
The first suggestion isn't always the best one. Check 2–3 alternatives before accepting. One of them often fits better.
✓
Accept word-by-word when the full suggestion is mostly right
Use Ctrl+→ / Cmd+→ to accept one word at a time. Great when the first half is perfect but the ending needs adjustment.
✓
Keep files focused — Copilot reads the whole open file
The more relevant context in the file, the better. A file full of unrelated code dilutes the signal Copilot gets.
Pattern 8

Chat vs Completions — when to use which

USE COMPLETIONS WHEN
  • →You're in flow and want to keep typing
  • →The task is repetitive (adding a new test to a file that has many)
  • →You've started the code and want it completed
  • →You need a simple helper or utility
USE CHAT WHEN
  • →You have a specific question about code
  • →You're debugging a failure and need hypotheses
  • →You want an explanation of something
  • →You need complex generation from a description
  • →You want to iterate: "now change it to also handle..."
Quick rule: If you can describe what you want in one sentence, use Completions. If you need a conversation, use Chat.

Always review before keeping

✓
Run the test
Does it pass? Does it fail when you expect it to fail (make it fail intentionally to verify the assertion is real)?
✓
Check the selectors against your real app
Does getByRole('button', {name: 'Save'}) actually match your button? Open the app and check.
✓
Verify the assertions are meaningful
Is the assertion actually testing what the feature should do, or is it just checking that a page loaded?
✓
Check for library/API currency
If Copilot uses an API method you don't recognise, check the Playwright docs. Training data has a cutoff date.

Building good daily habits

AM
When starting a new test file, open Chat and describe the feature you're testing. Ask for a skeleton of test cases before writing any code.
→
When you hit a test failure you can't immediately diagnose, paste the error in Chat before spending more than 5 minutes staring at it.
→
When you see code you don't understand — in a PR review, a legacy file, or a new codebase — select and Explain before spending time decoding manually.
PM
Before marking a test PR as ready for review, paste your test code in Chat and ask: "Review this for brittle selectors, missing assertions, and edge cases."
Key Takeaways

What to take from today

1
Copilot is a pair programmer, not autocomplete. Steer it with context, constraints, and clear descriptions — it will execute.
2
Use Completions for flow and repetition. Use Chat for questions, debugging, explanations, and complex generation.
3
Descriptive comments are prompts. A good comment before a function tells Copilot exactly what to generate.
4
Always review: run the test, verify selectors, confirm assertions are meaningful. Generated code needs the same QA as human-written code.
5
Build habits: Copilot for debugging after 5 minutes, Explain for unfamiliar code, review your tests with Chat before PRs.
