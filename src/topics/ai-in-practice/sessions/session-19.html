What makes something an 'agent'?

A model
The AI that reasons and makes decisions — Claude, GPT-4o, Gemini
️
Tools
The capabilities to act — read files, run tests, call APIs, control browsers, create tickets
Memory
Context about the task, what's been done, what worked, what failed — the conversation history and state
A goal
A task to accomplish that requires multiple steps and decisions — not just a single question to answer
Claude Code with MCP tools enabled is an agent. Copilot Chat by itself is not — it lacks tools that take real actions.

How an agent executes a task

Accumulates context
Each cycle adds to what the agent knows — files read, errors seen, results returned by tools
Typically 5–20 cycles
Most tasks complete in under 20 tool calls. Complex multi-file work can reach 40–50
Model decides when done
The model, not your code, judges when to stop — based on whether the goal is met

Three ingredients — that's all it takes

1 — Model
The brain
Choose a model capable of multi-step reasoning. Claude Opus 4, GPT-4o, Gemini 1.5 Pro. Bigger models handle longer task chains more reliably.
2 — Tools
The hands
Functions the model can call: read_file, run_tests, create_issue. Each needs a name, description, and parameter schema the model can read.
3 — System Prompt
The boundaries
Defines role, goal, and constraints. "Only read from /reports/. Never modify source code." This is where safety and scope are enforced.
import anthropic

client  = anthropic.Anthropic()
tools   = [{"name": "read_file", ...}, {"name": "create_issue", ...}]
system  = "You are a QA triage agent. Analyse CI failures, create bug tickets.\n"
          "Read from /reports/ only. Never modify source code."

response = client.messages.create(
    model="claude-opus-4-7", tools=tools, system=system,
    messages=[{"role": "user", "content": "Triage today's CI failures."}]
)
# Claude runs the loop — perceive → think → act → observe → decide
You don't write the loop logic — the model runs it. Your job: choose tools wisely, write a clear system prompt, and test on real tasks before going live.

Agents add overhead — use them when they earn it

Single question or generation — no real-world action needed
Direct prompt
One API call — fast, cheap, no loop overhead
Multi-step task requiring real-world actions in sequence
Single agent
The loop handles the steps — define the goal once
Same task repeated across many independent items in parallel
Parallel agents
Run simultaneously — much faster than waiting for each
Complex workflow spanning distinct specialised domains
Multi-agent
Each agent has focused tools and a tight system prompt
Rule of thumb: If you're writing a long complex prompt to simulate what a tool could just do — you need an agent, not a better prompt.
Don't over-engineer: Most day-to-day QA tasks still work fine with a direct prompt or copilot. Agents are for complexity, not novelty.

Agents are only as capable as their tools

Filesystem
read_file · write_file · list_directory · search_file
Read test reports, write test files, explore the codebase
Process
run_command · execute_tests · run_script · check_output
Run the test suite, execute build scripts, invoke CLI tools
API Connectors
github_create_issue · jira_ticket · slack_post · http_call
Create tickets, post updates, trigger deploys, call any REST API
Browser
navigate · click · screenshot · fill_form · wait_for
Drive a real browser, take screenshots, perform UI actions
Search
web_search · codebase_search · grep · docs_lookup
Find relevant code, look up docs, check for existing tickets
MCP servers (Sessions 17–18) deliver most of these tools. You configure which tools an agent can see — it cannot use capabilities you haven't granted.

See the agent loop live — paste this into Claude Code

Open your terminal, run claude, and paste the prompt below. No MCP setup needed — built-in file tools are enough. 
Act as a QA triage agent. Work through these four steps one at a time:

1. Create sample-failures.txt with 3 realistic Playwright test failures
   (each failure: test name, error message, file path)
2. Read the file back and analyse each failure — most likely root cause?
3. Draft a GitHub issue title and body for the most critical failure
4. Tell me what MCP tools would let you automate this completely

Think out loud as you work through each step.
What you'll watch happen
Act Model creates the file with realistic failures
Observe Model reads it back, confirms content
Think Model reasons about each failure, forms hypotheses
Act Model drafts the issue with structured content
Decide Model reflects on what else it would need
What the loop looks like
Between each action, the model shows its reasoning — what it found, what it thinks it means, what it's going to do next. That visible chain of thought is the perceive-think-act-observe cycle in real time.

The orchestrator and worker pattern

A multi-agent test pipeline
Orchestrator agent — "CI tests just ran. Coordinate the failure response."
Analyserreads report, categorises failures
Ticket creatoropens Jira tickets with context
Notifierposts summary to Slack channel
Each worker specialises. The orchestrator decides the sequence. You define the goal once.

Three ways to coordinate multiple agents

Sequential Chain
Each output feeds the next input
Use when steps must happen in order and each agent depends on the previous result. A reader feeds an analyser feeds a writer. Reliable, predictable.
Parallel Fan-out
Multiple agents run simultaneously
Use when tasks are independent. The orchestrator delegates to several agents in parallel and merges the results. Much faster than sequential for divisible work.
Hierarchical
Orchestrators manage sub-orchestrators
Use for large complex tasks that break into distinct domains. Each sub-orchestrator manages its own team of specialist workers. Scales to very large workflows.

A real agentic QA workflow

1
CI completes — triggers an agent session via webhook
2
Agent reads test report via filesystem MCP — identifies 3 failures
3
Agent checks GitHub via GitHub MCP — finds related PR changes that caused failures
4
Agent creates Jira tickets via Jira MCP — one per failure, with root cause hypothesis
5
Agent posts summary to Slack — "3 failures, tickets created, root cause identified"
Total time: ~2 minutes. You wake up in the morning with failures already investigated and triaged.

Five ways agents fail — and how to catch them

Looping
Same tool call repeated with no forward progress — agent is stuck
→ set max_iterations limit
Hallucinated
outcomes
Claims success before checking — assumes tool calls worked without verifying
→ require re-read to verify
Scope creep
Accesses files, APIs, or takes actions outside what you intended
→ explicit constraints in system prompt
Context
exhaustion
Long runs fill the context window — agent contradicts or repeats earlier work
→ break into phases, add checkpoints
Overconfident
completion
Declares done before the goal is actually met — output doesn't match objective
→ audit trail, human review gate
QA engineers are trained to spot failure modes. Apply that same skill to reviewing agent output — trust but verify every time, especially early on.

Before you deploy: four safeguards

Scope Constraints
Limit what it can touch
Write explicit rules in the system prompt: allowed directories, allowed APIs, forbidden actions. The model respects these — they are your primary guardrail.
Read from /reports/ only.
Never modify source code.
Never delete any file.
Approval Gates
Pause before irreversible actions
Before posting to Slack, creating tickets, or sending any external message — have the agent surface a summary for human review. Claude Code does this by design. Match the gate to the blast radius of the action.
Stopping Conditions
Define when to halt the loop
Set explicit limits: maximum tool calls, time budget, cost ceiling. Stop if the same error appears three consecutive times. Prevents runaway agents from doing more damage trying to recover from a bad state.
Audit Trail
Log everything the agent does
Record every tool call: name, arguments, result, timestamp. Not just what the model decided — what it actually executed. Essential for debugging failures and for building the trust that lets you widen the scope over time.
Start narrow. Give the agent minimal permissions, watch it run, then expand scope gradually as you build confidence in its behaviour.

What stays human

Agents handle well
  • Reading and summarising test reports
  • Creating tickets with standard information
  • Generating test code from specifications
  • Updating tests after selector changes
  • Posting routine status updates
Humans still essential
  • Deciding what to test and why
  • Evaluating product quality and risk
  • Investigating truly novel failures
  • Working with teams on priorities
  • Catching what tests can't catch
AI makes the mechanical faster. QA engineers focus on the judgement. Together: better coverage, faster feedback, less toil.

The direction of travel

Models are getting significantly more capable
What requires careful prompting today will be more reliable and intuitive next year. The threshold for what agents can handle autonomously is rising.
MCP ecosystem is growing rapidly
More tools, more services, more integrations are becoming MCP-compatible. The network effects of a standard protocol are kicking in.
Multi-agent coordination is becoming mainstream
What requires custom orchestration code today will become configurable workflows. Building pipelines will get progressively easier.
The skills you have now will compound
Understanding how to direct agents, write good prompts, connect tools — these skills grow in value as the technology grows in capability.
