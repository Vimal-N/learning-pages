What makes terminal agents different

️
Full codebase awareness
A terminal agent can explore your entire project — read any file, understand the directory structure, trace how modules connect — before it writes a single line.
Direct environment access
It can run your test suite, execute shell commands, interact with git, read build output, and act on the results — in a single conversation.
Conversational and iterative
You direct it in plain English. It acts, shows what it did, and you continue the conversation to refine or redirect.

Claude Code — what it is

What it is
Anthropic's official CLI for Claude. A terminal agent that can read, write, and operate on your codebase through natural language conversation. It uses the most capable version of Claude under the hood.
Who makes it
Anthropic — the AI safety company behind the Claude model family. Claude Code is their first-party tool, meaning it's built and supported by the same team that trains Claude. It gets updates alongside the model.
What access do you need
A Claude.ai Pro or Max subscription (includes Claude Code), or an Anthropic API key for pay-per-token usage. Check with your team lead about which your organisation uses.

Installing Claude Code — step by step

1
Check Node.js is installed (18+)
node --version
# Should print v18.x.x or higher
If not installed: download from nodejs.org → choose the LTS version.
2
Install Claude Code globally
npm install -g @anthropic-ai/claude-code
The -g flag installs it globally — available from any folder, not just one project.
3
Log in to your account
claude login
# Opens browser → sign in → returns to terminal authenticated
This links Claude Code to your Anthropic/Claude.ai account. Required once. Your credentials are stored securely.
4
Start your first session
cd your-project-folder
claude
Always navigate to your project folder first. Claude Code will read your files from wherever you start it.

How Claude Code understands your project

1
Reads the directory structure
When you start claude, it runs the equivalent of ls -la on your project to understand the structure — folders, files, what kind of project this is.
2
Reads CLAUDE.md (if it exists)
If you have a CLAUDE.md in your project root, Claude reads it immediately — conventions, framework info, what not to do. Your permanent brief to the agent. (More on this in Session 11.)
3
Reads relevant files for each task
When you give it a task, it reads the files it needs to understand before acting. For a test fix: reads the failing test file, the relevant page object, and maybe the component being tested.
4
Everything fits in its context window
The context window is the limit. A focused task on 5–10 files works great. "Refactor my entire 500-file codebase" is too big — break it into scoped stages first.

How Claude Code asks for permission

Before editing a file — shows you the diff
# Claude wants to modify tests/checkout.spec.ts
-   getByText('Place Order')
+   getByText('Confirm Order')

Allow this edit? [y/n/always]
y = approve this once | always = approve all future edits to this file | n = reject
Before running a command — shows you the exact command
# Claude wants to run:
npx playwright test tests/checkout.spec.ts

Run this command? [y/n]
You see the exact command before it runs. Never a surprise.
You are always in control. Claude asks, you decide. This is fundamentally different from running an untrusted script.

Inside a terminal agent session

1
You give a task in plain English
"Add Playwright tests for the checkout flow, covering happy path and card declined scenario"
2
Agent reads your project
Explores directory structure, reads relevant test files, understands existing patterns and conventions in your codebase
3
Asks permission, then acts
Shows you diffs before editing files. Shows exact commands before running them. Waits for your approval at each step.
4
Runs commands and verifies
After making changes, runs tests, reads output, fixes errors iteratively until the task is verified complete
5
Reports what it did
Summarises changes, shows file paths, explains key decisions. You review, ask follow-up questions, or redirect.

Your first Claude Code session

We have a Playwright test suite in tests/. Three tests are failing because we renamed a button from "Place Order" to "Confirm Order" in the UI. Find and fix all the affected tests.
Let me search the test files for references to "Place Order"...

[Reads directory structure → searches for "Place Order" text across all test files]

Found 3 occurrences across 2 files:
  • tests/checkout.spec.ts — lines 34, 67
  • tests/pages/checkoutPage.ts — line 12

I'll update all three. Showing you each change for approval...

[Shows diff for each file, waits for approval, makes changes]

Running npx playwright test tests/checkout.spec.ts...

3/3 tests passing. All fixes applied.

Opencode — the open-source alternative

Installing Opencode
Install
npm install -g opencode-ai
Start
cd your-project
opencode
Requires
API keys for whichever model you configure. Works with Anthropic, OpenAI, Google, or local Ollama models.
Claude Code vs Opencode
Interface: Claude Code is more chat-like. Opencode has TUI panels (chat, files, shell output) — more visual.
Models: Claude Code = Claude only. Opencode = any model you configure.
Open source: Opencode is fully open source. Claude Code is Anthropic-owned.
Best choice if: Claude Code if you want best-in-class Claude experience. Opencode if your org restricts which AI services you can use.

Where terminal agents excel

Codebase exploration
"Where is the cart total calculated? Show me all the places it's referenced."
Multi-file updates
"Update all API endpoint URLs from v1 to v2 across the test suite"
Test + fix loops
"Run the suite, identify failing tests, and fix the failures one by one"
Scaffolding new structure
"Set up the folder structure and base files for testing our new Payments module"
Documentation generation
"Read all our Playwright tests and generate a test coverage summary document"
Git-aware operations
"Show what changed in this branch and check if any test selectors need updating"

Claude Code permission modes

Default (interactive)
Recommended to start
claude
Asks permission before every file edit and shell command. You see and approve each action. The safest mode — start here.
Auto-approve edits in trusted project
# Inside a session, type:
/trust
# Claude will apply edits without prompting (still asks for shell commands)
Use after you've established trust in a project and want faster iteration. Still asks before running shell commands.
No-confirm mode (use with care)
Advanced only
claude --dangerously-skip-permissions
Skips all confirmations. For sandboxed automated workflows only. Never use on your main project directory with important files.

Using terminal agents safely

Commit before every session — always
If something goes wrong, git restore . returns you to a clean state in seconds. This is your undo button. Without a commit, a bad session means lost work.
Review all diffs before approving
The agent shows changes before applying them. Read them the same way you'd review a teammate's PR. Don't auto-approve everything — understand what's changing.
Give precise scope in your task description
"Only modify files in tests/checkout/" is more reliable than "update the checkout tests." Scope constraints prevent accidents in unrelated areas.
Never paste API keys or credentials into chat
Use environment variables instead. Treat the chat log as potentially readable by others. Credentials belong in .env files, not prompts.

Key takeaways

1
Terminal agents operate on your whole codebase, not just one file
They can explore, read, modify, and run commands across your entire project — understanding architecture before they act.
2
Claude Code asks permission before every consequential action
You see file diffs before they're applied. You see exact commands before they run. You are always in control — never surprised.
3
Always commit before a session — git is your safety net
A committed working state means you can recover from any agent mistake in seconds. Make this a habit before you ever type "claude".
4
Best for multi-file, exploration-heavy tasks
When a task requires understanding your project structure before acting — or making changes in many places simultaneously — terminal agents are the right tool.
