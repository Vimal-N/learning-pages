What Is MCP?

Understanding the Model Context Protocol — the standard that lets AI tools connect to external systems, data, and actions.

45 min
Duration

The problem before MCP

Before MCP — fragmented
  • ✗ Every AI tool has its own custom integration format
  • ✗ Integrations are one-off — can't be reused across tools
  • ✗ AI can only access data it was trained on or you paste in
  • ✗ No standard way to let AI take actions in external systems
  • ✗ Building integrations requires deep AI platform knowledge
With MCP — standardised
  • ✓ One standard protocol — any MCP server works with any MCP client
  • ✓ Build once, reuse with Claude, Copilot, Cursor, any MCP-compatible tool
  • ✓ AI can read live data from your systems, not just its training data
  • ✓ AI can take structured actions — create tickets, query databases, run scripts
  • ✓ Community ecosystem of ready-built servers to install and use

A second analogy — the restaurant

🧑‍💼
AI (Claude)
The diner. Knows what they want. Speaks in plain requests.
🧾
MCP Protocol
The menu & waiter. Standard format for ordering. Translates requests to the kitchen.
👨‍🍳
MCP Server
The kitchen. Does the real work — reads files, queries databases, calls APIs.
🍽️
Result
Structured data back to the AI. The AI uses it to form its answer.
Key insight: You can go to different restaurants (swap servers) and still order the same way. The protocol is standard — only the kitchen changes.

How MCP works — three pieces

💻
Host
Claude Code, Claude Desktop, Cursor, Zed
The AI app the user talks to
⟷
🔄
MCP Protocol
Standardised JSON messages
The shared language both sides speak
⟷
🗄️
MCP Server
GitHub, database, filesystem, custom
The bridge to real-world systems
What servers expose
Resources (data to read), Tools (actions to take), Prompts (templates to use)
Transport options
Local (stdio) or remote (HTTP). Most community servers run locally.
Who controls access
The server decides what data and actions the AI can access. You own the server.

What actually happens when you ask Claude something

1
You ask: "What tests failed in today's test run?"
2
Claude decides: "I have a filesystem tool available. I'll use it to read the test results."
3
Protocol layer sends: A structured JSON message to the filesystem MCP server: "read file: tests/results/latest.json"
4
MCP server acts: Opens the file from your filesystem, reads the content
5
Server responds: Returns the file content as structured data back through the protocol
6
Claude answers you: "3 tests failed today: checkout.spec.ts, auth.spec.ts, and profile.spec.ts. Here's what happened..."

Three things MCP servers can provide

📄
Resources — data the AI can read
READ ONLY
Files, database records, GitHub issues, Confluence pages, Jira tickets. Think of these like books on a shelf — the AI can open them and read them.
Examples: read a file · read a database row · fetch a ticket · list directory contents
⚙️
Tools — actions the AI can take
READ + WRITE
Create a Jira ticket, query a database, run a Playwright test, send a Slack message, commit a file. The AI can act, not just advise.
Examples: create ticket · run SQL query · click a browser element · send notification
💬
Prompts — reusable templates
TEMPLATES
Pre-built prompt templates for common tasks. A "write test report" template that knows your format and automatically pulls the latest test run data.
Examples: daily standup summary · test failure report · release notes template

Resources vs Tools — the key difference

📖
Resources (read-only)
· AI can observe data but not change it
· Like opening a file to read it
· Safe for sensitive data — read access only
· Examples: read test report, read ticket details, read code file
⚡
Tools (active)
· AI can take actions that change things
· Like clicking a button or submitting a form
· More powerful — requires careful scope control
· Examples: create ticket, write file, send message, run query
Practical rule: Start with read-only resources. Add tools only when the AI needs to take action. The scope you grant is the scope the AI can use — keep it minimal until you need more.

What MCP messages actually look like

📤 Request (AI → Server)
{{ '{' }}
  "jsonrpc": "2.0",
  "method": "tools/call",
  "params": {{ '{' }}
    "name": "read_file",
    "arguments": {{ '{' }}
      "path": "tests/results/latest.json"
    {{ '}' }}
  {{ '}' }}
{{ '}' }}
📥 Response (Server → AI)
{{ '{' }}
  "jsonrpc": "2.0",
  "result": {{ '{' }}
    "content": [{{ '{' }}
      "type": "text",
      "text": "{{ '{' }} \"passed\": 18, \"failed\": 3, ... {{ '}' }}"
    {{ '}' }}]
  {{ '}' }}
{{ '}' }}
You don't write these by hand — the MCP SDK generates them automatically. But they explain why MCP is interoperable: every server speaks the same JSON format.

MCP vs just calling an API directly

Direct API call (old way)
  • · Requires custom code for every AI + API combination
  • · Doesn't work across different AI tools
  • · Auth, error handling all bespoke per integration
  • · No standard for what data to expose to AI
  • · Each integration is a one-off engineering project
MCP server (new way)
  • · Build once, works with Claude, Copilot, Cursor, any MCP client
  • · AI automatically discovers what tools are available
  • · Standardised authentication and error handling patterns
  • · Community ecosystem of ready-to-install servers
  • · Reusable across your whole team — one server, everyone benefits

Key terms to know

MCP
Model Context Protocol. The open standard that defines how AI tools and external systems communicate. Published by Anthropic, adopted by the industry.
MCP Host
The AI application the user works in. Examples: Claude Code, Claude Desktop, Cursor, Zed. The host contains the MCP client.
MCP Client
The protocol layer built into the host. Handles sending requests to servers and receiving responses. Invisible to the user.
MCP Server
The program that connects your AI to a real-world system. Can be community-built (filesystem, GitHub) or custom-built by your team.
Transport
How the client and server connect. stdio = local process (most common). HTTP/SSE = remote server over the network.
Tool Call
When the AI decides to use an MCP tool. The AI constructs a request, the server executes it, the result comes back to the AI.

Key takeaways

1
MCP = the USB standard for AI integrations
One standard protocol so any MCP server works with any MCP-compatible AI tool. Build once, use everywhere — the same server works with Claude, Copilot, and Cursor.
2
Three capabilities: Resources, Tools, Prompts
Read data passively (resources), take active actions (tools), and reuse prompt templates — the three things an MCP server can provide.
3
A growing ecosystem of ready-to-use servers
GitHub, filesystem, Playwright, databases, Jira — most common integrations already exist. Session 14 covers using them. Session 15 covers building your own.
4
QA engineers have strong concrete use cases
CI test results, issue tracking, live test data, browser control — MCP makes these accessible to AI without any copy-pasting between systems.
