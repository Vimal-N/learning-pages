Why build your own MCP server?

Your internal systems
Custom CI pipelines, internal test management, company-specific deployment tools — these don't have community servers. You build the bridge.
Full data control
You decide exactly what data the AI can see and what actions it can take. No community code touching your systems.
Team-specific tools
A "summarise today's test failures in our format" tool that knows your team's naming conventions, report structure, and ticket numbering.
Eliminate copy-paste friction
Any time you copy data from one system into an AI chat, that's a candidate for an MCP tool. Identify the friction, build the bridge.

Setting up your MCP server project

1 — Create the project folder
mkdir my-qa-mcp-server && cd my-qa-mcp-server
npm init -y
npm init -y creates a package.json with defaults. The -y skips all the questions.
2 — Install dependencies
npm install @modelcontextprotocol/sdk
npm install -D typescript tsx @types/node
The next slide explains what each of these packages does.
3 — Update package.json and create the server file
# Add to package.json manually:
"type": "module",
"scripts": {{ '{' }} "start": "tsx src/server.ts" {{ '}' }}

# Create the server file:
mkdir src && touch src/server.ts

What each dependency does

@modelcontextprotocol/sdk
npm install (production)
The official MCP library from Anthropic. Handles the entire protocol — request parsing, response formatting, error handling, transport management. Without this, you'd have to implement MCP from scratch. This is the one that matters.
typescript
npm install -D (dev only)
The TypeScript compiler. Lets us write type-safe code — it'll warn us if we pass the wrong type to a function, before anything runs.
tsx
npm install -D (dev only)
Runs TypeScript files directly without a build step. Instead of: compile → run the JS, we just: run the TS. Makes development much faster. Used in the npm start command.
@types/node
npm install -D (dev only)
Type definitions for Node.js built-ins like fs (filesystem) and path. Without this, TypeScript doesn't know what fs.readFileSync is.

The complete server file — all at once

// src/server.ts — our complete MCP server
import {{ '{' }} Server {{ '}' }} from '@modelcontextprotocol/sdk/server/index.js';
import {{ '{' }} StdioServerTransport {{ '}' }} from '@modelcontextprotocol/sdk/server/stdio.js';
import {{ '{' }} ListToolsRequestSchema, CallToolRequestSchema {{ '}' }} from '@modelcontextprotocol/sdk/types.js';
import * as fs from 'fs';

const server = new Server({{ '{' }}
  name: 'qa-tools',
  version: '1.0.0'
{{ '}' }});

server.setRequestHandler(ListToolsRequestSchema, async () => ({{ '{' }}
  tools: [{{ '{' }}
    name: 'get-test-summary',
    description: 'Summarise the latest Playwright test run results',
    inputSchema: {{ '{' }} type: 'object', properties: {{ '{' }} reportPath: {{ '{' }} type: 'string', description: 'Path to the JSON report file' {{ '}' }} {{ '}' }} {{ '}' }}
  {{ '}' }}]
{{ '}' }}));

server.setRequestHandler(CallToolRequestSchema, async (request) => {{ '{' }}
  if (request.params.name === 'get-test-summary') {{ '{' }}
    const {{ '{' }} reportPath {{ '}' }} = request.params.arguments as {{ '{' }} reportPath: string {{ '}' }};
    const data = JSON.parse(fs.readFileSync(reportPath, 'utf8'));
    const failed = data.suites.flatMap((s: any) => s.specs).filter((s: any) => s.ok === false);
    return {{ '{' }} content: [{{ '{' }} type: 'text', text: JSON.stringify({{ '{' }} failed, total: data.stats {{ '}' }}, null, 2) {{ '}' }}] {{ '}' }};
  {{ '}' }}
  throw new Error(`Unknown tool: ${{ '{' }}request.params.name{{ '}' }}`);
{{ '}' }});

const transport = new StdioServerTransport();
await server.connect(transport);

Part 1 — The imports, line by line

import {{ '{' }} Server {{ '}' }} from '@modelcontextprotocol/sdk/server/index.js';
import {{ '{' }} StdioServerTransport {{ '}' }} from '@modelcontextprotocol/sdk/server/stdio.js';
import {{ '{' }} ListToolsRequestSchema, CallToolRequestSchema {{ '}' }} from '@modelcontextprotocol/sdk/types.js';
import * as fs from 'fs';
Server
The main MCP server class. We create one instance of this. It's the core object that receives requests and holds our handlers.
StdioServerTransport
The connection method. "Stdio" means the server communicates via standard input/output — used when running as a local process on your machine. This is the right one for local MCP servers.
ListToolsRequestSchema
The schema for "list my tools" requests. When Claude asks "what tools do you have?", this schema identifies that message type so our handler can respond.
CallToolRequestSchema
The schema for "run this tool" requests. When Claude calls one of our tools, this schema identifies that message so our handler knows to execute it.
fs
Node.js built-in filesystem module. Lets us read files from disk. We'll use fs.readFileSync() to read the test report file.

Part 2 — Creating the server instance

const server = new Server({{ '{' }}
  name: 'qa-tools',
  version: '1.0.0'
{{ '}' }});
new Server()
Creates a new MCP server object. This is the container for our tools and the object that manages all communication. We create exactly one per server.
name
A human-readable name for this server. Appears in Claude's "what tools do you have?" output. Name it after what it does — 'qa-tools', 'test-results', 'ci-helper'.
version
Semantic version string. Used by MCP clients to detect when the server has been updated. Start at '1.0.0', increment when you add or change tools.
After this line: a server object exists but has no tools and no connection. The next two sections wire those up.

Part 3 — Registering your tools (ListTools handler)

server.setRequestHandler(ListToolsRequestSchema, async () => ({{ '{' }}
  tools: [{{ '{' }}
    name: 'get-test-summary', // ← A
    description: 'Summarise the latest Playwright test run results', // ← B
    inputSchema: {{ '{' }} // ← C
      type: 'object',
      properties: {{ '{' }}
        reportPath: {{ '{' }} type: 'string', description: 'Path to the JSON report file' {{ '}' }} // ← D
      {{ '}' }}
    {{ '}' }}
  {{ '}' }}]
{{ '}' }}));
A — name: The identifier Claude uses to call this tool. Must be unique within this server. Use kebab-case: 'get-test-summary', 'create-ticket', 'check-coverage'.
B — description: What Claude reads to decide WHEN to call this tool. Write it clearly: 'Use this when you need to see which tests passed or failed in the most recent test run.'
C — inputSchema: Describes what parameters the tool accepts, in JSON Schema format. Claude uses this to know what to send when calling the tool.
D — reportPath: One parameter: a string containing the path to the test report file. Claude will be prompted to provide this value when calling the tool.

Part 4 — The tool handler (CallTool), line by line

server.setRequestHandler(CallToolRequestSchema, async (request) => {{ '{' }}

  if (request.params.name === 'get-test-summary') {{ '{' }}

    const {{ '{' }} reportPath {{ '}' }} =
      request.params.arguments as {{ '{' }} reportPath: string {{ '}' }};

    const data = JSON.parse(
      fs.readFileSync(reportPath, 'utf8')
    );

    const failed = data.suites
      .flatMap((s: any) => s.specs)
      .filter((s: any) => s.ok === false);

    return {{ '{' }}
      content: [{{ '{' }}
        type: 'text',
        text: JSON.stringify({{ '{' }} failed, total: data.stats {{ '}' }}, null, 2)
      {{ '}' }}]
    {{ '}' }};
  {{ '}' }}

  throw new Error(`Unknown tool: ${{ '{' }}request.params.name{{ '}' }}`);
{{ '}' }});
setRequestHandler(CallToolRequestSchema, ...) — Registers the function that runs whenever Claude calls one of our tools. "async" means it can do file I/O without blocking.
The if-check routes the request to the right tool. When you add a second tool, you add another if-block.
request.params.arguments — The parameters Claude sent. We destructure reportPath from it.
fs.readFileSync — Reads the JSON file from disk synchronously. Returns the raw string, which we parse with JSON.parse.
flatMap + filter — Digs into the Playwright JSON structure: each suite has specs, each spec has an ok field. We collect all the failed ones.
return {{ '{' }} content: [...] {{ '}' }} — Required format for all tool responses. The text value is what Claude reads and uses in its answer.

Part 5 — Connecting the transport (last 2 lines)

const transport = new StdioServerTransport();
await server.connect(transport);
new StdioServerTransport()
Creates a transport object that uses standard input/output (stdin/stdout). Claude Code starts your server as a background process and sends MCP messages through these channels. Stdio is the right choice for local servers.
await server.connect(transport)
Starts the server — it begins listening for incoming MCP messages. The await is critical: this call never resolves. It keeps the process alive, reading and responding to messages in an ongoing loop.
That's the whole file. Two imports, create a server, register handlers for "list tools" and "call tool", connect transport. Under 50 lines for a fully working MCP server.

Connecting your custom server to Claude Code

Register via CLI
claude mcp add qa-tools node /absolute/path/to/my-qa-mcp-server/src/server.ts
Use the full absolute path — relative paths can fail depending on where Claude Code starts. You can run pwd in your project folder to get it.
Or directly in ~/.claude.json
"qa-tools": {{ '{' }}
  "command": "node",
  "args": ["/absolute/path/to/server/src/server.ts"]
{{ '}' }}
Restart and verify
What tools do you have available?
I have access to tools from your qa-tools server:
get-test-summary — Summarise the latest Playwright test run results.

Shall I use it? Provide the path to your test report file.

Testing your server end to end

Use the get-test-summary tool to summarise the results in tests/results/test-results.json
I'll use the get-test-summary tool from your qa-tools server...

[Claude calls your tool with reportPath = "tests/results/test-results.json"]

Test Run Summary:
Total: 21 tests — 18 passed, 3 failed

Failed tests:
  • tests/checkout.spec.ts — "should complete payment" (timeout)
  • tests/auth.spec.ts — "should redirect to dashboard" (assertion)
  • tests/profile.spec.ts — "should upload avatar" (element not found)
If something goes wrong:
1. Check Claude Code restarted after changes
2. Verify the file path is absolute and the report file exists
3. Look for errors in the Claude Code output — server errors appear there
4. Run node src/server.ts directly to check for TypeScript errors

More custom QA tools worth building

Flakiness tracker
Reads test history files, calculates a flakiness score per test (pass rate over last 20 runs), returns the worst offenders. Input: none. Output: ranked list of flaky tests.
Ticket creator
Takes test failure details (name, error, file, line) and creates a formatted Jira ticket via the Jira REST API. Input: failure object. Output: created ticket URL.
Coverage reporter
Reads your Istanbul/V8 coverage JSON and answers "which modules have less than 80% coverage?" Input: coverage threshold. Output: list of files below threshold.
Environment health checker
Pings your staging/UAT/dev environment URLs and checks HTTP status. Input: none (list hardcoded). Output: which environments are up, which are down. Run before starting a test session.

Key takeaways

1
An MCP server is five things: imports, server, list-tools, call-tool, transport
Every server follows this exact structure. The SDK handles the protocol. You only write the tool logic. A complete working server fits in about 50 lines.
2
The tool description is what Claude reads to decide when to use your tool
Write descriptions clearly: "Use this when you need to see which tests failed in the most recent Playwright run." Vague descriptions lead to the AI calling the wrong tool or missing it entirely.
3
Build for your team's specific friction points
Look for the moments where you're copying data from one system into an AI chat. Those are your best candidates for custom tools. The pattern: identify the friction → build the bridge.
4
Commit your server to the team repo — it's shared infrastructure
A useful MCP server benefits everyone on the team, not just the person who built it. The config entry in ~/.claude.json can be shared too — add it to your project's README.
