What is GitHub Copilot?

Inline completions and Chat — the two things Copilot does for you in VS Code.

GitHub Copilot is an AI coding assistant built into VS Code (and JetBrains, GitHub.com). It watches what you type and offers two things: inline completions that appear as grey "ghost text" while you write, and a Chat interface where you hold a proper conversation about your code.

It was trained on billions of lines of public code from GitHub, which is why it understands Playwright, TypeScript, and testing frameworks so well — far better than a general-purpose AI like ChatGPT, because it's seen how real test suites are structured.

Mode 1 — Inline Completions

Passive. Copilot watches what you type and suggests what comes next — shown in grey, right in your file. Tab to accept, Esc to dismiss.

Best for: boilerplate, repetitive patterns, completing a test you've started.

Mode 2 — Copilot Chat

Active. Open a chat panel (Ctrl+Alt+I) and have a conversation. Ask questions, request test generation, get explanations, debug failures.

Best for: complex generation, debugging, understanding unfamiliar code.

How Inline Completions Work

What Copilot reads, how it predicts, and why descriptive names matter.

Copilot reads a window of context around your cursor — the current file, your imports, recent variable names, and any open files VS Code has used recently. It predicts what tokens (code) are most likely to come next based on patterns in its training data, then shows that prediction as grey ghost text.

The practical implication: descriptive names produce better suggestions. A test named loginWithInvalidPassword tells Copilot far more than test1. A comment like // Test: verify rate limiting blocks after 5 failed attempts above a blank test function will generate a much more targeted suggestion than no comment at all.

The Context Reference System — @workspace & @vscode

Search your entire project and query VS Code's editor settings.

Copilot Chat has a set of @-mentions and #-references that let you explicitly add context to your messages. This is the difference between "write me a test" and "write me a test that matches my actual project".

@workspace Search your entire project
Tells Copilot to search across all files in your VS Code workspace. It can find where things are defined, what patterns exist, and how your project is structured — without you having to reference specific files.
Best for Finding patterns, understanding project structure, questions that span multiple files QA Example "@workspace Where do we define our Page Objects and what pattern do they follow?" Watch out Slower than asking about a specific file. Use @workspace for discovery, #file: when you know which file matters.
@vscode VS Code settings & config
Gives Copilot access to your VS Code workspace settings, installed extensions, and editor configuration. Use it when your question is about the editor itself, not your code.
Best for VS Code settings, workspace config, extension behaviour, editor keybindings QA Example "@vscode How do I configure the Playwright test timeout in this workspace's settings?" Watch out Don't confuse with @workspace. @vscode is for editor config; @workspace is for your project's code.

Context References — @terminal & #file:

Debug from terminal output and ground Copilot in your actual project files.

@terminal Active terminal output
Reads the output currently visible in your VS Code terminal. Indispensable when you have a test failure on screen and want Copilot to explain or diagnose it — you don't have to copy and paste the error.
Best for Test failures, build errors, command output you want explained QA Example Run your tests. Watch them fail. Open Chat → "@terminal Explain this test failure and suggest a fix." Watch out Copilot sees the terminal snapshot, not a live stream. If the terminal has scrolled past the error, it may miss it — scroll up first.
#file:path/to/file.ts Include a specific file as context
Attaches a specific file's contents to your message. This is the most important reference for QA work — referencing your existing Page Object or spec file ensures Copilot generates code that matches your actual patterns, not invented ones.
Best for Pattern matching, generating code consistent with an existing file, adding to a test suite QA Example "Write tests for the user profile page using the same style as #file:tests/auth/login.spec.ts" Watch out Large files slow responses. Reference the smallest file that contains the pattern you want to match.

Context References — #selection & #codebase

Ask about highlighted code and run semantic search across your entire test suite.

#selection Your currently selected code
Includes whatever code you've highlighted in the editor. Useful when you want to ask about a specific function or block without naming the file — just select it and ask.
Best for Questions about a specific highlighted function, explaining or reviewing a specific block QA Example Select a beforeEach hook → open Chat → "What does #selection do and under what conditions could it fail?" Watch out If nothing is selected, #selection sends an empty context. You'll get a generic response. Always select first.
#codebase Semantic search across all files
A more powerful version of @workspace. Performs semantic search across your indexed codebase. Good for finding usage patterns, deprecated APIs, or anti-patterns across your entire test suite.
Best for Finding patterns or anti-patterns, deprecated API usage, coverage gaps across the suite QA Example "#codebase Find all tests that use page.waitForSelector() — we're migrating to page.locator()" Watch out Slower than targeted searches. Use when you genuinely don't know which file to look in.

The most valuable combination for QA: #file:tests/pages/loginPage.ts + a clear task description. This gives Copilot your real selectors and your real pattern, so what it generates is immediately usable rather than a generic template.

Inline Chat (Ctrl+I / Cmd+I)

A mini prompt that opens right in your editor — four scenarios you'll use every week.

While the Chat panel is a separate window, Inline Chat opens a mini prompt directly in your editor at the cursor position. You type your request, Copilot shows the diff inline, and you accept or discard without leaving your code. It's faster for targeted in-file work.

Four Inline Chat scenarios you'll use every week

Scenario 1 — Explain a specific assertion

Select an assertion you don't recognise → press Ctrl+I → type: "Explain exactly what this assertion is checking and when it would fail"

Why this beats Chat: you don't have to copy-paste the code. The context is exact — just the lines you selected.

Scenario 2 — Generate a test body at the cursor

Write a test('user cannot log in with expired password', async ({ page }) => { then position your cursor inside the empty function body → Ctrl+I → type: "Implement this test. Login page is at /login. Use page.locator() not page.waitForSelector()."

Copilot generates the implementation inline. Review it before accepting — check the selectors and assertions.

Scenario 3 — Fix a failing assertion

Select a failing line → Ctrl+I → type: "This assertion is failing with: [paste error]. Fix it."

The fix appears as a diff. Accept if it looks right, reject and redirect if it doesn't.

Scenario 4 — Add JSDoc to a Page Object method

Position cursor on a Page Object method → Ctrl+I → type: "Add a brief JSDoc comment to this method describing what it does and what it returns."

The doc block appears above the function. Faster than writing it manually.

Try This — 2 minutes

Open any test file. Select a complete test function. Press Ctrl+I (Windows/Linux) or Cmd+I (Mac). Type: "Add a comment above this test explaining what user behaviour it verifies." See the diff, accept it. That's Inline Chat.

Settings Worth Configuring

Four settings to review once — then you can stop thinking about them.

Open VS Code Settings (Ctrl+, / Cmd+,) and search "GitHub Copilot" for all options. These four are worth reviewing now:

INLINE SUGGESTIONS: ENABLE
Global on/off toggle. Leave on. If you want to disable for a specific project, use the per-workspace setting instead.
INLINE SUGGESTIONS: LANGUAGES
Enable or disable Copilot per language. Useful if you want completions in TypeScript/JavaScript but not in Markdown or JSON. Defaults to enabled for most languages.
CHAT: PREFERRED MODEL
You can choose which model backs Copilot Chat (GPT-4o, Claude Sonnet, etc. — depending on your plan). For complex test generation, a more capable model produces better results. Check what's available on your plan.
ADVANCED: TELEMETRY
On Individual plan, usage data may be sent to GitHub for model improvement. You can opt out here. On Business/Enterprise plans, this is disabled by default — your code is not used for training.

What Copilot Can & Cannot Do

Set the right expectations — knowing the limits is as important as knowing the capabilities.

Copilot can
  • Write test boilerplate from a comment or description
  • Generate test data (valid, invalid, boundary values)
  • Explain what a piece of code does in plain English
  • Suggest fixes for failing tests given the error and code
  • Refactor repetitive test patterns into Page Objects
  • Answer questions about Playwright, Jest, TypeScript APIs
  • Complete code patterns you've already established in a file
Copilot cannot
  • Know your actual application (selectors, URLs, data are guesses)
  • Run code or verify its own output — you must run the tests
  • Remember previous conversations (each session starts fresh)
  • Know your team's conventions unless you tell it or it infers from files
  • Be reliable about very recent library versions (has a training cutoff)
  • Access the internet or your live application

The most dangerous assumption: that Copilot knows your app. It doesn't. Selectors like #submit-btn, URLs like /checkout/payment, and API responses are all educated guesses based on patterns in training data. Always run generated tests and verify selectors against your actual DOM before committing anything.

Privacy & Code Sharing

What happens to your code and what you should never share with Copilot.

When you use Copilot, code snippets are sent to GitHub's servers to generate suggestions. The key facts for your team's setup:

Individual
Code may be used for model improvement. You can opt out of telemetry in settings (search "Copilot telemetry").
Business / Enterprise
Code is not used for training by default. Prompts and suggestions are not retained beyond the session. Enterprise adds additional security controls.

Always applies: Never paste API keys, passwords, or credentials into Copilot Chat. Never paste production customer data or PII. When in doubt, check with your manager or security team before using Copilot on sensitive codebases.
