The GCSV Task Prompt Formula

Goal · Context · Scope · Verification — four elements that make any agent task prompt clear and safe.

A good agent task prompt answers four questions before the agent starts. Missing any one leads to a common failure: the agent completes something — just not quite what you wanted.

G
Goal — what does "done" look like?
State the outcome, not the steps. "Add three Playwright tests for the checkout flow" is a goal. "First read checkout.spec.ts, then look at the Page Object, then write…" is you doing the agent's planning — let it plan.
C
Context — what does the agent need to know?
Anything not in your CLAUDE.md that's specific to this task: a particular selector, an edge case to handle, a constraint that applies only to this work. Don't repeat what's already in the context file — add only what's new for this task.
S
Scope — what's in and what's out?
Explicitly state what the agent should and should not touch. "Only create new files — don't modify any existing tests." "Only changes in tests/checkout/ — don't touch auth tests." Scope limits prevent the most disruptive mistakes.
V
Verification — how will you know it worked?
Tell the agent how to verify its own work: "Run the tests and confirm all pass." "Run the linter after." "Don't ask me to review — run them yourself and report the outcome." This closes the loop inside the session rather than leaving it open.

GCSV in Practice

Three example task prompts — before (vague) and after (GCSV-structured).

Scenario: Adding tests for a new feature

Vague

"Add tests for the discount code feature."

GCSV

G: Add Playwright tests for applying a discount code at checkout.
C: Valid codes show a green badge and reduce the total. Invalid codes show span[data-testid="promo-error"] with "Invalid code".
S: Create tests/checkout/discount.spec.ts only — don't modify existing checkout tests.
V: Run the new spec and confirm all tests pass before finishing.

Scenario: Refactoring to remove duplication

Vague

"Refactor the login tests to use the Page Object."

GCSV

G: Update tests/auth/login.spec.ts to use the existing LoginPage PO instead of inline selectors.
C: LoginPage is in tests/pages/login.page.ts. All needed methods already exist — don't add new ones.
S: Only modify login.spec.ts. Don't change the PO or any other spec.
V: Run npx playwright test tests/auth/login.spec.ts — all tests must pass.

Scenario: Investigating a problem (read-only)

Vague

"Why are the checkout tests failing?"

GCSV

G: Diagnose why tests/checkout/order.spec.ts is failing in CI but passing locally.
C: The error is: [paste error]. Tests pass when run alone but fail in the full suite.
S: Investigation only — don't modify any files. Just report your findings.
V: Provide a written diagnosis with the most likely root cause and the recommended fix.

Task Decomposition — Staying Safe on Large Work

How to break a large piece of work into agent tasks that are each reviewable and recoverable.

The most dangerous agent sessions are the ones where the agent makes many changes across many files before you review anything. The solution is decomposing large work into tasks where each one ends with a git diff review and a test run before the next begins.

The decomposition rule

A task is the right size when you can answer yes to all three:

→
Can I verify it's done in under 5 minutes? — If reviewing the output takes longer than 5 minutes, the task is too large. Split it.
→
If I discard everything, have I lost less than 30 minutes of work? — If a task goes wrong and you run git checkout ., the cost should be small. Large tasks that go wrong are expensive.
→
Does this task have a clear pass/fail condition? — "The tests run and pass" is a clear condition. "The code looks better" is not. Tasks without clear verification conditions are harder to review.

Example: Refactoring a full suite

TASK 1
Extract CheckoutPage PO
Goal: extract a Page Object from checkout.spec.ts. Scope: create the PO only — don't update the spec yet. Verify: PO file created, TypeScript compiles. → review diff → proceed.
TASK 2
Update checkout spec
Goal: update checkout.spec.ts to use the new PO. Scope: only this spec file. Verify: all checkout tests pass. → review diff → run tests → proceed.
TASK 3
Add new tests
Goal: add 3 tests for the discount code feature using the new PO. Scope: new spec file only. Verify: new tests pass. → review diff → commit.

Managing the Decision Loop

When to let the agent proceed, when to redirect, and when to stop and restart.

Agent is reading files and planning — output looks reasonable
Let it proceed — this is the normal explore phase
Agent proposes a file write you didn't expect
Ask "why are you modifying [file]?" before approving
Agent proposes running a destructive command (rm, drop table, git reset)
Deny it, explain the constraint, re-task more carefully
Agent fails the same test 3 times in a row
Interrupt, provide the missing context it's clearly lacking
Agent completes the task but tests still fail
Check the error output yourself — the agent may have missed something obvious
Agent seems to be going in circles, switching approaches
Stop, run git checkout . to reset, start with a smaller more specific task

The "just one more try" trap: Agents are persuasive. When one approach fails, they confidently propose another. After the third failed approach, the agent is guessing — and you're accumulating file changes that will be harder to untangle. Stop earlier than feels necessary.

Try This — GCSV on a Real Task

Apply the GCSV formula to your next agent task.

Try This — 20 minutes

Pick a real piece of work — adding a test, debugging a failure, or a small refactor — and write the task prompt using the GCSV formula before opening your terminal agent.

Step 1 (5 min): Write your GCSV prompt on paper or in a text file before you open the terminal. Don't start the agent until you've answered all four letters. If you can't answer V ("how will I know it worked?"), the task isn't well-defined enough yet.

Step 2 (10 min): Run the agent. Watch the output. When it finishes, run git diff and your test suite.

Step 3 (5 min): Compare what you got against your G (goal). Was it done? Was scope respected? Did V (verification) pass? Note where the GCSV formula helped and where you needed more detail.
