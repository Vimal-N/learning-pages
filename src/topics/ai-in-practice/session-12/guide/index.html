Context Files — CLAUDE.md, copilot-instructions & AGENTS.md

How to write a context file that makes your AI agent understand your project without being told the same things every session.

Session 12  ·  Phase 4: Terminal Agents & Agentic Tools  ·  Intermediate
⚡
What you'll get from this session
By the end of Session 12 you will have a working CLAUDE.md for your Playwright project, understand the equivalent files for Copilot and OpenAI agents, and know what to put in a context file versus what to leave out.

From Sessions 10–11: You know agents read your project before acting. Context files are how you direct that reading — telling the agent what matters about your project, your conventions, and what it should never do.

Why Context Files Exist

Every time you start a new agent session, it has no memory of your previous sessions. Context files are the solution.

A terminal agent starts each session with no knowledge of your project conventions, your team rules, or your preferences. Without a context file, you either explain these things in every prompt (tedious), or the agent makes decisions that don't match your project (frustrating).

Without a context file
  • Agent uses page.locator() when your project uses page.fill()
  • Agent creates a beforeEach when your tests are self-contained
  • Agent invents selector naming conventions instead of using data-testid
  • Agent modifies files in src/ when it should only touch tests/
  • You re-explain the same rules every session
With a good context file
  • Agent knows your test structure before reading a single test file
  • Agent follows your conventions automatically
  • Agent knows which directories are off-limits
  • Agent knows your framework, runner, and assertion style
  • Every session starts with the same baseline understanding

The Three Context Files

Each AI tool reads a different file. The content is similar — the filename and location differ.

CLAUDE.md
Claude Code
Placed at the root of your project (or in any subdirectory for scoped rules). Claude Code reads it automatically at the start of every session. Also read by Opencode when using Anthropic as the backend. Plain markdown — no special syntax required.
copilot-instructions.md
GitHub Copilot
Placed at .github/copilot-instructions.md. Read by Copilot Chat and Copilot Agent Mode in VS Code. Also picked up by Opencode when using the Copilot backend. The same conventions you'd put in CLAUDE.md go here.
AGENTS.md
OpenAI Codex / multi-agent
Used by OpenAI's Codex agent and some multi-agent frameworks. Same concept — project context, conventions, constraints. If you're using multiple agent tools on one project, you may have all three files. They can share the same content, just with different filenames.

One project, one set of rules: If your project might be used with multiple AI tools, write your context once and maintain a simple copy under each filename. The content is identical — the agent tool just looks for its specific file.

What to Put In a Context File

Six sections that cover what every agent needs to know about a QA project.

1
Project overview — one paragraph
What the project is, what technology stack it uses, and what the AI's role is in this project. "This is a Playwright TypeScript E2E test suite for an e-commerce application. AI assistance is used for test writing, debugging, and refactoring."
2
Run commands
How to run the tests, what the most common commands are. Without this, the agent guesses — and sometimes runs the wrong command. Include: test command, lint, build if applicable.
3
Code conventions
Your selector strategy (data-testid, IDs, etc.), whether tests use Page Objects, assertion style, how test files are named, whether beforeEach is used. These are the rules the agent would otherwise have to infer.
4
Directory structure — what's where
Which directory contains tests, which contains Page Objects, which contains fixtures and helpers. The agent will explore your project regardless — this just makes it faster and less likely to look in the wrong place.
5
Hard constraints — what NOT to do
Files or directories the agent must never modify. Common examples: never edit src/ (application code), never modify package.json without asking, never delete test files. These prevent the most damaging mistakes.
6
Known gotchas
Things that caught the agent out before: a known flaky test that should be ignored, a selector pattern that looks fragile but is intentional, an environment variable needed for tests to run. Write these down when you discover them.

Annotated CLAUDE.md Example

A complete CLAUDE.md for a Playwright project — copy and adapt this for your own project.

# Project Overview
Playwright TypeScript E2E test suite for an e-commerce web application.
Tests cover: auth, checkout, product catalogue, and user account flows.
AI assistance is used for test writing, debugging, and refactoring.

## Run Commands
- Run all tests: `npx playwright test`
- Run one spec: `npx playwright test tests/auth/login.spec.ts`
- Run with UI: `npx playwright test --ui`
- Lint: `npm run lint`
- Type check: `npx tsc --noEmit`

## Project Structure
- `tests/` — all spec files, organised by feature
- `tests/pages/` — Page Object classes (one per page)
- `tests/fixtures/` — shared test fixtures and helpers
- `tests/data/` — test data factories
- `playwright.config.ts` — framework config
- `src/` — application code — DO NOT MODIFY

## Code Conventions
- Selectors: always use `data-testid` attributes. Never use class names or XPath.
- Page Objects: one class per page. Selectors are private readonly. Methods return void.
- Test isolation: each test must be self-contained. No shared beforeEach that sets up state.
- Assertions: use Playwright's `expect()` API. Never use `page.waitForTimeout()`.
- Imports: use relative paths. PO files import from `./pages/`.
- Test naming: `test.describe('Feature name')` > `test('should [do thing]')`

## Hard Constraints
- NEVER modify files in `src/` — this is application code
- NEVER modify `package.json` or `package-lock.json`
- NEVER delete spec files — ask before removing anything
- NEVER commit or push — only the human commits

## Known Gotchas
- The `products/catalogue.spec.ts` test is intentionally skipped (`test.skip`) — the
  feature is in development. Do not unskip it.
- Tests require `BASE_URL` env var to be set. For local: `BASE_URL=http://localhost:3000`
- The checkout flow requires a logged-in user. Use the `loggedInUser` fixture from
  `tests/fixtures/auth.fixture.ts` — don't write a manual login in checkout tests.

Before and After — Context File Impact

The same task prompt, with and without a CLAUDE.md. What changes.

Task given to the agent: "Add tests for the product search feature."

Without CLAUDE.md
  • Uses page.locator('.search-input') — class selector, fragile
  • Creates a beforeEach that logs in — duplicating fixture logic
  • Adds page.waitForTimeout(500) for the search results
  • Creates the spec in the project root, not in tests/
  • Uses toContain assertion on a list that could be empty
With CLAUDE.md
  • Uses data-testid="search-input" — correct selector strategy
  • Uses the loggedInUser fixture — no duplicate login logic
  • Uses expect(results).toHaveCount(greaterThan(0)) — no arbitrary wait
  • Creates spec at tests/search/search.spec.ts — correct location
  • Follows naming convention: test.describe('Product Search')
Try This — 10 minutes

Create a CLAUDE.md (or copilot-instructions.md) for your current project. Start with sections 1–5 from the "What to Put In" slide. Don't try to make it perfect — a rough first version that covers your selector strategy and hard constraints is already 80% of the value. Add gotchas as you discover them over the next few agent sessions.
