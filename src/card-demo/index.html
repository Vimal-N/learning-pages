Informational CalloutBlue · supplementary context, prerequisites, useful-to-know details mid-page

Note: The context window limit varies by Copilot plan. If you hit the limit mid-session, start a new chat rather than extending the current one — the model loses earlier context anyway.

Critical CalloutAmber · important warnings, mindset shifts, things that frequently trip people up

Critical point: An agent can modify files, run commands, and make decisions — all without asking you each time. This requires a different kind of oversight than Chat. You review what it did, not just what it said.

End of Chapter CalloutGreen · achievement summary at end of a page — what was accomplished

What just happened: The agent read 3 files, created 1 new file, ran tests, diagnosed a failure, proposed a fix, and verified the fix — all in one session. You reviewed 2 diffs and approved a plan. That's the Agent Mode value.

Danger CalloutRed · "Common Mistake" inside Pattern Cards — something that has actively broken things

Not referencing an existing file with #file:. Without a style reference, Copilot generates valid code using patterns that don't match your project's conventions. Adding #file: takes 10 seconds and saves 20 minutes of cleanup.

Comparison CardTwo-column contrast — before/after · good/bad · option A vs B · neutral/neutral
Before Transformers (RNNs)
  • Read text word by word, left to right
  • Earlier context "faded" in long sequences
  • Slow to train — couldn't parallelise
With Transformers
  • Processes all tokens simultaneously
  • Attention mechanism learns what matters
  • Scales with more data and more compute

Pattern CardNumbered reusable workflow — pattern name · labelled sections · common mistake
Pattern 1
Generate from comment + context
The pattern

Write a comment describing the test intent, add @workspace for project context, then let inline completions generate the test body.

When to use

New test files, boilerplate-heavy tests, when you want Copilot to match existing test style.

Common mistake

Accepting completions without checking selectors. Generated selectors are often plausible but wrong — always verify against your Page Object.

Command Result CardColoured left border · monospace command label · plain-English result — one card per command or output
/explain
Returns a plain-English description: what the code does, what selectors it uses, what it asserts, and what it does not verify.
/fix
Suggests improvements: prefer page.locator() over raw CSS selectors; strengthen assertions to check specific dashboard elements.
/tests
Generates additional test cases based on this one: invalid email format, wrong password, empty fields, account locked, session persistence.
/doc
Adds a JSDoc comment above the test: selector assumptions, preconditions (user account must exist), expected state after the test completes.

Scenario 1 — Explain a specific assertion

Select an assertion you don't recognise → press Ctrl+I → type: "Explain exactly what this assertion is checking and when it would fail."

Why this beats Chat: the context is exact — just the lines you selected, nothing to copy-paste.

Scenario 2 — Generate a test body at the cursor

Position your cursor inside an empty test function body → Ctrl+I → type the implementation instruction. Copilot generates the body inline as a diff.
